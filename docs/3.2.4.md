# **Documento técnico — Feature v3.2.4**

## **Referencias visuales opcionales por inventario**

## **1\. Resumen ejecutivo**

La feature **v3.2.4** incorpora la posibilidad de asociar **imágenes de referencia visual** a un inventario durante su creación, de forma **opcional**, con el objetivo de enriquecer el contexto visual utilizado por Gemini en el análisis de pasillos.

Estas referencias representan ejemplos válidos del estándar esperado dentro de un inventario, como por ejemplo:

* etiquetas de pallets,  
* disposición visual de códigos,  
* formato típico de impresión,  
* aspecto general esperado de una identificación válida.

Cuando el inventario tenga referencias cargadas, dichas imágenes se utilizarán como **contexto multimodal adicional** al momento de ejecutar un análisis sobre un pasillo. El prompt enviado a Gemini deberá adaptarse para indicar explícitamente que estas imágenes no son la evidencia primaria, sino material comparativo que ayuda a mejorar la interpretación visual.

La feature está pensada como una mejora incremental y práctica del flujo actual, sin introducir todavía un sistema complejo de catálogo, clasificación automática o gestión avanzada de datasets.

---

## **2\. Objetivo de negocio**

Mejorar la eficacia del análisis visual permitiendo que el sistema conozca previamente cómo deberían verse las etiquetas o pallets válidos dentro de un inventario determinado.

Esto busca reducir errores de interpretación en escenarios donde:

* existen etiquetas visualmente similares,  
* hay ruido o ambigüedad en la captura del pasillo,  
* el formato visual esperado no queda suficientemente claro solo con instrucciones textuales,  
* el proveedor multimodal necesita contexto de comparación para reconocer mejor elementos relevantes.

---

## **3\. Objetivo técnico**

Extender el flujo actual de creación de inventarios y de análisis de pasillos para soportar:

* carga opcional de imágenes de referencia por inventario,  
* persistencia de esas imágenes y su metadata,  
* resolución automática de referencias al analizar un pasillo,  
* envío de dichas referencias junto con el prompt multimodal a Gemini,  
* adaptación del prompt para que Gemini entienda cómo debe usar ese contexto.

---

## **4\. Alcance**

### **4.1 Incluye**

* un segundo paso opcional en el frontend al crear un inventario;  
* carga de entre 1 y 3 imágenes de referencia;  
* almacenamiento persistente de referencias visuales por inventario;  
* asociación uno-a-muchos entre inventario y referencias;  
* exposición de referencias para su visualización básica en frontend;  
* resolución automática de referencias al ejecutar análisis sobre un pasillo;  
* envío de referencias visuales al proveedor Gemini como parte del contexto del análisis;  
* adaptación del prompt para explicar el rol de esas referencias;  
* trazabilidad mínima de qué referencias fueron utilizadas en el job.

### **4.2 No incluye**

* entrenamiento formal de modelos;  
* catálogo global enterprise;  
* referencias compartidas entre múltiples inventarios;  
* clasificación automática por tipo de referencia;  
* scoring inteligente o selección compleja de referencias;  
* versionado avanzado;  
* librería global de estándares;  
* curación automática de calidad de imágenes.

---

## **5\. Problema actual**

Actualmente el pipeline analiza evidencia visual del pasillo sin una fuente explícita de contexto visual que represente el estándar esperado del inventario.

Esto puede provocar:

* menor precisión en la interpretación de etiquetas;  
* ambigüedad al comparar códigos o formatos impresos;  
* dificultad para inferir qué elementos son importantes dentro de la escena;  
* dependencia excesiva del prompt textual;  
* menor control contextual sobre lo que Gemini debería priorizar.

La feature v3.2.4 busca reducir esta brecha con una solución simple: permitir que el usuario aporte referencias visuales concretas del inventario.

---

## **6\. Casos de uso**

### **6.1 Crear inventario sin referencias**

El usuario crea un inventario y omite el paso opcional de referencias visuales.  
El sistema guarda el inventario normalmente.  
Los análisis posteriores funcionan con el comportamiento actual.

### **6.2 Crear inventario con referencias**

El usuario crea un inventario, avanza al segundo paso opcional y carga imágenes de referencia.  
El sistema almacena esas imágenes asociadas al inventario.  
Los análisis de pasillos futuros usan esas referencias como contexto.

### **6.3 Analizar pasillo con referencias disponibles**

El sistema recibe una solicitud de análisis para un pasillo perteneciente a un inventario con referencias visuales.  
Antes de invocar a Gemini, resuelve las referencias activas del inventario, las adjunta al análisis y adapta el prompt.

### **6.4 Analizar pasillo sin referencias disponibles**

El sistema ejecuta el análisis con el flujo actual, sin cambios.

---

## **7\. Requisitos funcionales**

### **RF-1**

El flujo de creación de inventario debe incluir un segundo paso opcional para cargar imágenes de referencia.

### **RF-2**

El usuario debe poder omitir dicho paso sin afectar la creación del inventario.

### **RF-3**

El sistema debe permitir subir múltiples imágenes de referencia, dentro de un límite configurado.

### **RF-4**

Cada imagen de referencia debe quedar asociada a un inventario específico.

### **RF-5**

El frontend debe poder listar y visualizar las referencias asociadas a un inventario.

### **RF-6**

Cuando se analice un pasillo, el backend debe resolver si el inventario asociado tiene referencias.

### **RF-7**

Si existen referencias, estas deben incluirse en el request multimodal enviado a Gemini.

### **RF-8**

El prompt debe adaptarse para aclarar el rol de las referencias visuales.

### **RF-9**

Si no existen referencias, el sistema debe comportarse exactamente igual que hoy.

### **RF-10**

El job debe persistir metadata mínima indicando si se utilizaron referencias y cuáles fueron.

---

## **8\. Requisitos no funcionales**

### **RNF-1 — Compatibilidad**

La feature no debe romper el flujo actual cuando un inventario no tenga referencias.

### **RNF-2 — Bajo acoplamiento**

La lógica de referencias no debe quedar fuertemente acoplada a Gemini ni a un prompt hardcodeado dentro del route.

### **RNF-3 — Performance**

Debe existir límite de cantidad y tamaño de imágenes para evitar degradación significativa del tiempo de análisis.

### **RNF-4 — Trazabilidad**

El sistema debe permitir auditar qué referencias fueron usadas por cada job.

### **RNF-5 — Escalabilidad futura**

La estructura debe dejar abierta la posibilidad de evolucionar a referencias categorizadas o librerías más complejas.

---

## **9\. Diseño funcional**

## **9.1 Flujo de usuario**

### **Paso 1 — Creación base del inventario**

El usuario completa los datos básicos del inventario.

### **Paso 2 — Referencias visuales (opcional)**

El usuario puede:

* cargar imágenes de referencia,  
* ver previsualizaciones,  
* continuar sin subir imágenes.

### **Resultado**

El inventario queda creado:

* con referencias, o  
* sin referencias.

---

## **10\. Diseño de frontend**

## **10.1 Cambio principal**

El flujo actual de creación de inventario debe transformarse en un proceso de dos pasos o wizard liviano:

### **Paso 1**

Datos básicos del inventario.

### **Paso 2**

Sección “Imágenes de referencia (opcional)”.

## **10.2 Comportamiento esperado**

* el segundo paso debe explicar para qué sirven las referencias;  
* debe quedar claro que son opcionales;  
* el usuario debe poder subir varias imágenes;  
* se debe permitir continuar sin imágenes;  
* se deben mostrar previews;  
* se deben validar formato y tamaño del lado del cliente antes del upload.

## **10.3 Copy sugerido en UI**

**Título:**  
Imágenes de referencia (opcional)

**Descripción:**  
Estas imágenes ayudan al sistema a entender mejor cómo se ven las etiquetas o pallets esperados en este inventario.

**Acciones:**

* Omitir este paso  
* Guardar y continuar

---

## **11\. Diseño de backend**

## **11.1 Nuevo concepto de dominio**

Se introduce una entidad técnica para representar referencias visuales del inventario.

### **Entidad: `InventoryVisualReference`**

Responsabilidad:

* representar una imagen de referencia asociada a un inventario;  
* almacenar metadata básica;  
* permitir su resolución durante el análisis.

## **11.2 Campos sugeridos**

* `id`  
* `inventory_id`  
* `filename`  
* `storage_path`  
* `mime_type`  
* `file_size`  
* `created_at`

Campos opcionales a considerar:

* `width`  
* `height`  
* `created_by`  
* `is_deleted` o `is_active`

Para este release, se recomienda mantener el modelo simple.

---

## **12\. Modelo de datos propuesto**

### **Tabla: `inventory_visual_references`**

Campos mínimos sugeridos:

* `id` UUID / string  
* `inventory_id` UUID / string  
* `filename` text  
* `storage_path` text  
* `mime_type` text  
* `file_size` bigint o integer  
* `created_at` timestamp

Relación:

* muchos `inventory_visual_references` pertenecen a un `inventory`.

Índice recomendado:

* índice por `inventory_id`

---

## **13\. Storage**

## **13.1 Estrategia recomendada**

Las imágenes deben guardarse en una estructura predecible y desacoplada del nombre original.

Ejemplo de convención:

`/inventories/{inventory_id}/visual_references/{reference_id}.{ext}`

## **13.2 Consideraciones**

* no depender del nombre original como clave principal;  
* validar extensiones permitidas;  
* sanitizar nombres si se conservan para display;  
* almacenar en base solo el path lógico o relativo;  
* permitir futura migración a otro backend de storage.

---

## **14\. API propuesta**

## **14.1 Crear inventario**

Se mantiene el endpoint de creación de inventario existente.

La carga de referencias puede resolverse de dos maneras:

### **Opción A — más simple y desacoplada**

1. crear inventario,  
2. subir referencias usando endpoint dedicado.

### **Opción B — creación compuesta**

un solo flujo hace todo.

Para esta versión, se recomienda **Opción A**, porque simplifica contratos y errores.

---

## **14.2 Upload de referencias**

### **`POST /api/v3/inventories/{inventory_id}/visual-references`**

Responsabilidad:

* recibir una o varias imágenes;  
* validarlas;  
* almacenarlas;  
* crear registros asociados al inventario.

**Response sugerida:**  
lista de referencias creadas con metadata.

---

## **14.3 Listado de referencias**

### **`GET /api/v3/inventories/{inventory_id}/visual-references`**

Responsabilidad:

* devolver las referencias asociadas al inventario.

---

## **14.4 Eliminación**

No es obligatoria para este release.  
Puede diferirse si el objetivo es salir rápido y mantener el scope enfocado.

---

## **15\. Integración con el pipeline**

## **15.1 Punto de integración**

La resolución de referencias debe ocurrir antes de construir la solicitud final a Gemini.

El flujo conceptual debe ser:

1. se recibe solicitud de análisis para un pasillo;  
2. se resuelve el inventario asociado;  
3. se buscan referencias visuales del inventario;  
4. se construye el contexto del análisis;  
5. se invoca a Gemini con:  
   * evidencia principal,  
   * referencias visuales,  
   * prompt enriquecido.

## **15.2 Principio clave**

Las referencias no reemplazan la evidencia principal.  
Se usan como **contexto comparativo y orientativo**.

---

## **16\. Adaptación del prompt**

## **16.1 Objetivo del prompt enriquecido**

El prompt debe dejar claro:

* que las referencias muestran ejemplos válidos del estándar esperado;  
* que deben ayudar a reconocer mejor etiquetas, formatos o pallets;  
* que no deben tratarse como evidencia principal;  
* que la observación real del pasillo sigue teniendo prioridad.

## **16.2 Instrucción conceptual sugerida**

El prompt debería incorporar una sección equivalente a esta:

“Además de la evidencia principal del pasillo, se incluyen imágenes de referencia del inventario. Estas referencias muestran ejemplos válidos del estándar visual esperado, como etiquetas o pallets típicos. Úsalas como contexto comparativo para reconocer mejor formatos, disposición visual y elementos relevantes. No asumas coincidencia exacta obligatoria y priorizá siempre la evidencia visible en la imagen real analizada.”

## **16.3 Riesgo a evitar**

No mezclar las referencias y la evidencia principal de forma ambigua.  
Gemini debe poder entender qué imágenes son:

* imágenes de referencia,  
* imágenes reales del pasillo a analizar.

---

## **17\. Arquitectura sugerida**

Para evitar acoplamiento excesivo, conviene introducir una pequeña separación de responsabilidades.

## **17.1 Componentes sugeridos**

### **`InventoryVisualReferenceRepository`**

Responsable de obtener referencias por inventario.

### **`InventoryReferenceService`**

Responsable de resolver referencias activas y validar límites.

### **`AnalysisContextBuilder`**

Responsable de armar el contexto total del análisis combinando:

* evidencia principal,  
* referencias visuales,  
* instrucciones textuales.

### **`GeminiPromptBuilder`**

Responsable de construir el prompt final con la sección de referencias cuando corresponda.

## **17.2 Beneficio**

Esta separación evita que la lógica quede incrustada dentro del route handler o del cliente de Gemini.

---

## **18\. Persistencia de metadata del job**

## **18.1 Necesidad**

Cada job debe dejar trazabilidad mínima sobre el uso de referencias visuales.

## **18.2 Información mínima recomendada**

* si se usaron referencias;  
* cuántas referencias se usaron;  
* qué IDs de referencia participaron.

## **18.3 Formato posible**

Dentro de metadata o JSON del job:

```json
{
  "visual_reference_context": {
    "used": true,
    "reference_ids": ["ref_1", "ref_2"],
    "count": 2
  }
}
```

## **18.4 Motivo**

Esto permite:

* auditar resultados;  
* depurar análisis;  
* verificar que el flujo realmente haya utilizado referencias;  
* sostener evolución futura de la feature.

---

## **19\. Validaciones recomendadas**

### **Validación de formato**

Permitir solo:

* jpg  
* jpeg  
* png  
* webp

### **Validación de cantidad**

Límite sugerido:

* máximo 3 imágenes por inventario en esta versión.

### **Validación de tamaño**

Definir un tamaño máximo razonable por archivo para no encarecer el request a Gemini.

### **Validación de existencia**

No permitir asociar referencias a inventarios inexistentes.

---

## **20\. Manejo de errores esperado**

### **En frontend**

* archivo inválido;  
* tamaño excedido;  
* error de upload;  
* fallo parcial si se suben múltiples archivos.

### **En backend**

* inventario inexistente;  
* formato no permitido;  
* límite de cantidad excedido;  
* error de storage;  
* error de persistencia.

### **En pipeline**

* si no se pueden cargar referencias, el sistema debe decidir entre:  
  * fallar explícitamente, o  
  * continuar sin referencias y dejar warning.

Para este release, si el inventario ya tiene referencias persistidas pero no se pueden resolver al momento del análisis, conviene registrar el problema de forma auditable.

---

## **21\. Riesgos técnicos**

## **21.1 Sobrecarga del request multimodal**

Más imágenes implican mayor costo, más latencia y más ruido contextual.

**Mitigación:** límite estricto de cantidad y tamaño.

## **21.2 Acoplamiento a Gemini**

Si toda la lógica queda incrustada en el proveedor actual, migrar a futuro será más caro.

**Mitigación:** builder de contexto y prompt.

## **21.3 Paths frágiles o archivos huérfanos**

Una mala estrategia de storage puede romper referencias.

**Mitigación:** convención de paths estable y capa dedicada de storage.

## **21.4 UX confusa**

El usuario podría interpretar que el paso es obligatorio.

**Mitigación:** copy claro y acción explícita de omitir.

## **21.5 Ambigüedad semántica en el prompt**

Si Gemini no distingue entre referencia y evidencia real, puede degradar el análisis.

**Mitigación:** instrucción clara y separación conceptual de imágenes.

---

## **22\. Dependencias**

* autenticación ya implementada o suficientemente estable;  
* contratos v3 razonablemente alineados;  
* storage de archivos operativo;  
* flujo actual de análisis con Gemini ya funcional;  
* acceso desde el análisis del pasillo al inventario padre.

## **Fase 2 — Frontend**

* agregar paso opcional de referencias;

## **Fase 3 — Integración pipeline**

* resolver referencias por inventario;  
* construir contexto enriquecido;

* enviar imágenes de referencia a Gemini.

---

## **24\. Criterios de aceptación**

* el usuario puede crear un inventario sin cargar referencias;  
* el usuario puede cargar referencias en un segundo paso opcional;  
* las referencias quedan asociadas correctamente al inventario;  
* el frontend muestra las referencias cargadas;  
* el análisis de un pasillo sigue funcionando aunque el inventario no tenga referencias;  
* si el inventario tiene referencias, estas se incluyen en el análisis enviado a Gemini;  
* el prompt se adapta para indicar que las referencias son contexto visual comparativo;  
* el job deja trazabilidad mínima de qué referencias fueron utilizadas;  
* la feature no rompe el flujo actual ni introduce dependencia rígida en la capa equivocada.

---

## **25\. Decisiones recomendadas para cerrar antes de implementar**

1. **Máximo exacto de imágenes**  
   Recomiendo máximo 3\.  
2. **Tamaño máximo por imagen**  
   Debe definirse antes de tocar frontend y backend.  
3. **Delete sí o no en este release**  
   Recomiendo dejarlo fuera si querés mantener el scope liviano.  
4. **Formato de metadata del job**  
   Recomiendo guardar `used`, `reference_ids`, `count`.  
5. **Si el análisis falla al resolver referencias**  
   Conviene decidir si continúa sin referencias o aborta.

---

## **26\. Conclusión**

La feature v3.2.4 debe implementarse como una mejora concreta del flujo de creación y del pipeline multimodal actual, no como un subsistema complejo de gestión de referencias.

Su valor principal está en permitir que Gemini reciba, junto con la evidencia del pasillo, ejemplos visuales reales del estándar esperado para ese inventario. Esto debería mejorar la capacidad del sistema para entender mejor qué buscar, cómo interpretar etiquetas y cómo reducir ambigüedad visual, manteniendo al mismo tiempo un diseño simple, opcional y compatible con la arquitectura actual.
