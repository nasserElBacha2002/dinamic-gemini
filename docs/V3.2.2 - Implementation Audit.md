# v3.2.2 Implementation Audit

**Release:** v3.2.2 — Minimum Count Rule and qty Hardening  
**Audit type:** Technical implementation audit (no new features)  
**Scope:** Raw qty entry → normalization → resolution → persistence → API serialization; source of truth; resolver semantics; mapper; persistence; API contract; frontend contract; test coverage.

---

## 1. Executive Verdict

**Continue with corrections**

The branch has a clear architectural direction: ProductRecord is the intended authoritative source for resolved qty and provenance; the resolver is well-defined and handles unresolved states with an explicit `is_resolved` flag; and the API prefers ProductRecord when present. Several issues remain: a type/contract bug in SQL row mapping (ProductRecord.description can receive `None` from DB), ambiguous semantics when `primary_product` exists but `qty_source` is empty (legacy path is used and could diverge from what was persisted), and no explicit handling of `corrected_quantity` in the qty contract (by design, but should be documented). Test coverage is strong for resolution and summary mapping; integration tests for “persistence round-trip then API” and for “list endpoint with ProductRecordRepository” would close remaining gaps. With the description fix and optional tightening of the “authoritative” condition, the branch is safe to continue implementing on.

## 2. Current Qty Flow

### 2.1 Entry

- **Parser** (`src/parsing/global_analysis_parser.py`): Builds `Entity` from LLM/global analysis JSON. Reads `product_label_quantity` via `_safe_int(e.get("product_label_quantity"))`; sets `final_quantity=None` (lines 189, 199, 206). So raw qty first enters as `Entity.product_label_quantity` (optional int).
- **Count status** (`src/decision/count_status.py`): `assign_count_status(entity)` sets `entity.final_quantity` from `entity.product_label_quantity` when status is COUNTED/EMPTY (e.g. PALLET with position + qty → `final_quantity = entity.product_label_quantity`; EMPTY_PALLET → `final_quantity = 0`). So after this stage, entities have both `product_label_quantity` and `final_quantity` where applicable.
- **Report** (`src/reporting/hybrid_report.py`): Builds `hybrid_report.json` with `product_label_quantity` and `final_quantity` from each `Entity` (lines 87, 92). Pipeline writes this report; `PersistAisleResultUseCase` later reads it and calls the mapper.

So **raw qty enters** in the parser as `product_label_quantity`; `final_quantity` is set by the decision layer and both are written into the report.

### 2.2 Field presence

- **Mapper** (`src/infrastructure/pipeline/v3_report_mapper.py`, `_qty_from_entity`): Uses key presence, not value alone. If `"final_quantity" in entity` → `present = True`, `raw = entity.get("final_quantity")`; else if `"product_label_quantity" in entity` → `present = True`; else `present = False`, `raw = None` (lines 101–112). So “field present” means the key exists in the report entity.
- **Legacy API path** (`src/api/routes/v3/shared.py`, `_resolve_qty_contract_from_position_legacy`): Same idea: `"final_quantity" in j` or `"product_label_quantity" in j` → present (lines 394–401).

So **field presence** is “key in dict” for `final_quantity` / `product_label_quantity` in both mapper and legacy API.

### 2.3 Normalization

- **Single place:** `src/domain/quantity/resolution.py`, `normalize_raw_qty(raw_qty, field_was_present=...)`. Returns `NormalizedQty(raw_qty, parse_status, explicit_qty)` with status one of: MISSING (field absent), NULL (explicit None), INVALID (non-int, negative, etc.), ZERO (0), VALID_POSITIVE (≥1). No other code path normalizes qty; the mapper and legacy path both call this.

### 2.4 Resolution

- **Single place:** `src/domain/quantity/resolution.py`, `resolve_final_qty(..., has_valid_evidence, is_product_present, normalized_qty, explicit_consolidated_qty=None, allow_zero_as_valid=False)`. Order: (1) valid explicit detected qty → DETECTED; (2) explicit consolidated qty ≥ 1 → CONSOLIDATED; (3) zero with `allow_zero_as_valid` → DETECTED, 0; (4) valid evidence + product-present → INFERRED 1; (5) else → qty_final=0, **is_resolved=False**.
- **Mapper** calls it in `_qty_from_entity` with `has_valid_evidence = not evidence_path_missing` (evidence_path present), `is_product_present` from count_status in {COUNTED, COUNTED_MANUAL}, `allow_zero_as_valid = (entity_type == "EMPTY_PALLET")`. Does not pass `explicit_consolidated_qty`.
- **Legacy API** path calls the same `resolve_final_qty` when summary JSON does not already contain usable `qty_final`/`qty_source`.

So **final qty is resolved** in one domain module; mapper and API legacy path both use it with consistent inputs (modulo evidence proxy).

### 2.5 Persistence

- **Mapper** builds one `ProductRecord` per entity with: `detected_quantity=quantity` (from resolver), `qty_source`, `qty_inference_reason`, `raw_qty`, `qty_parse_status` from the resolution meta; when `is_resolved` is False it persists `qty_source="unresolved"` (mapper lines 143–144, 249–254).
- **SQL** (`src/infrastructure/repositories/sql_product_record_repository.py`): `save()` INSERT/UPDATE includes `qty_source`, `qty_inference_reason`, `raw_qty_json`, `qty_parse_status`. `list_by_position` / `get_by_id` SELECT and map via `_row_to_product()` to the same entity fields.
- **Schema** (`src/database/schema.sql`): `product_records` has nullable columns `qty_source`, `qty_inference_reason`, `raw_qty_json`, `qty_parse_status` (v3.2.2 ALTERs).

So **final qty and provenance** are persisted on ProductRecord and in `detected_summary_json` (secondary); the mapper keeps them aligned.

### 2.6 API serialization

- **List** (`src/api/routes/v3/positions.py`): For each position, loads `product_record_repo.list_by_position(p.id)`, takes primary (first by created_at, id), calls `position_to_summary(p, primary_product=primary)`.
- **Detail** (`src/api/routes/v3/positions.py`): Gets primary product from result, calls `position_to_summary(result.position, corrected_quantity=..., primary_product=primary_product)`.
- **shared** (`src/api/routes/v3/shared.py`, `position_to_summary`): If `primary_product is not None and (primary_product.qty_source or "").strip()` → use `_qty_contract_from_product(primary_product)` for `(qty, qtySource, qtyInferenceReason)`. Else → `_resolve_qty_contract_from_position_legacy(p, has_evidence=...)` from `detected_summary_json`. Response DTO requires `qty` and `qtySource` (no defaults); `qtyInferenceReason` optional.

So **API** uses ProductRecord as the source of qty/provenance when a primary product exists and has a non-empty `qty_source`; otherwise it falls back to recomputing from `detected_summary_json`.

## 3. Authoritative Source of Truth Analysis

- **Intended design:** ProductRecord is the authoritative persisted record for resolved qty and provenance; `detected_summary_json` is secondary / compatibility / debug (mapper comment lines 145, 239).
- **In practice:**  
  - **Write path:** Only the mapper writes both ProductRecord and Position; it uses a single resolution and writes the same decision to both. So at write time there is one logical source.  
  - **Read path:** List and detail routes load ProductRecord and pass it into `position_to_summary`. When `primary_product` is not None and `qty_source` is non-empty, the API uses only ProductRecord for qty/qtySource/qtyInferenceReason. So for rows created by the current mapper, **ProductRecord is the effective authoritative source** for the API.  
  - **Edge case:** If a row has a ProductRecord but `qty_source` is NULL or "" (e.g. pre–v3.2.2 data or a bug), the API falls back to legacy and recomputes from `detected_summary_json`. That recomputation can differ from whatever was originally persisted in ProductRecord (e.g. if summary was later amended or is missing v3.2.2 meta). So **authority is conditional** on “primary_product and qty_source set.”  
- **Conclusion:** There is **one intended source of truth (ProductRecord)** and the API is wired to use it when available and populated. The legacy path is a fallback; duplication exists in `detected_summary_json` but is documented as secondary. Persistence and API can only diverge when the legacy fallback is used (no product or empty qty_source) or when summary JSON is edited independently.

## 4. Findings

### 4.1 Critical

**None.** No finding was classified as release-blocking critical.

### 4.2 High

**H1. ProductRecord.description can be None from DB (type/contract bug)**  
- **Files:** `src/domain/products/entities.py` (ProductRecord.description: str), `src/infrastructure/repositories/sql_product_record_repository.py` (_row_to_product: `description=getattr(row, "description", None)`).  
- **Why it matters:** Entity declares `description: str`; schema allows NULL. Loading a row with NULL description passes None into ProductRecord, violating the type and any code that assumes description is a string.  
- **Risk:** Type errors or AttributeError when description is used without null checks.  
- **Fix:** In `_row_to_product`, use `description=(getattr(row, "description", None) or "")` so the entity always receives a str (or change the entity to `Optional[str]` and handle None everywhere).

**H2. Authority condition: empty qty_source falls back to legacy**  
- **Files:** `src/api/routes/v3/shared.py` (`position_to_summary`: `if primary_product is not None and (primary_product.qty_source or "").strip()`).  
- **Why it matters:** When ProductRecord exists but qty_source is NULL or "", the API recomputes from `detected_summary_json`. For pre–v3.2.2 or migrated rows without qty_source set, the response can differ from the stored ProductRecord.detected_quantity.  
- **Risk:** Slight inconsistency for old data; acceptable if intentional “legacy compatibility” but should be explicit.  
- **Fix:** Document that empty qty_source means “treat as legacy.” Optionally, when primary_product exists but qty_source is empty, still use primary_product.detected_quantity for `qty` and default qtySource to "detected" so at least the number is from the DB.

### 4.3 Medium

**M1. has_valid_evidence is a proxy (evidence_path presence)**  
- **Files:** `src/infrastructure/pipeline/v3_report_mapper.py` (has_valid_evidence = not evidence_path_missing; comment lines 194–196).  
- **Why it matters:** Inference (qty=1) depends on has_valid_evidence. Today “valid evidence” means “evidence_path key present and non-empty,” not “evidence file exists and was validated.”  
- **Risk:** If the pipeline sometimes sets evidence_path without a real artifact, we could infer 1 when we shouldn’t.  
- **Fix:** Already documented in code. For a future release, consider a stronger signal (e.g. artifact existence or validation flag).

**M2. corrected_quantity not part of qty contract**  
- **Files:** `src/api/routes/v3/shared.py` (position_to_summary returns qty from ProductRecord/legacy and separately corrected_quantity).  
- **Why it matters:** The API exposes both `qty` (detected/inferred) and `corrected_quantity`. UI may need to show “final” = corrected_quantity if set, else qty.  
- **Risk:** Misunderstanding whether “qty” is pre- or post-correction.  
- **Fix:** Document in API or DTO that `qty` is the resolved detected/inferred value and that corrected_quantity, when present, is the reviewer override.

### 4.4 Low

**L1. Legacy path does not set qty_is_resolved**  
- **Files:** `src/api/routes/v3/shared.py` (_resolve_qty_contract_from_position_legacy returns only (qty, qtySource, qtyInferenceReason)).  
- **Why it matters:** When using legacy path, the response does not expose whether the resolution was “unresolved” (is_resolved=False).  
- **Risk:** Frontend cannot distinguish “valid 0” (e.g. EMPTY_PALLET) from “unresolved” when relying only on legacy data.  
- **Fix:** Optional: add an optional `qtyResolved?: boolean` to the summary response and set it from resolution when using legacy path.

**L2. explicit_consolidated_qty never passed from mapper**  
- **Files:** `src/infrastructure/pipeline/v3_report_mapper.py` (_qty_from_entity calls resolve_final_qty without explicit_consolidated_qty).  
- **Why it matters:** Consolidation rules (e.g. from multiple frames) could in theory feed a single explicit_consolidated_qty; the resolver supports it but the mapper does not use it.  
- **Risk:** None for current pipeline; only matters if consolidation is added later.  
- **Fix:** Leave as-is; document if consolidation is planned.

## 5. Invariant Check

| Invariant | Status | Notes |
|----------|--------|--------|
| Accepted valid entity + no explicit quantity ⇒ qty=1 | **Pass** | Resolver: has_valid_evidence and is_product_present ⇒ inferred 1. Mapper and legacy path both use this. |
| Accepted valid entity never exposes unjustified null qty | **Pass** | Response qty is always int (from _qty_contract_from_product or legacy); DTO qty is required int. No null in contract. |
| Accepted valid entity never exposes unjustified 0 qty | **Pass** | When resolved, ProductRecord has detected_quantity from resolver (1 or explicit). Unresolved is persisted as qty_source="unresolved" and API returns (0, "detected", None); docstring states 0 must not be treated as valid visible for unresolved. |
| Explicit valid qty always wins | **Pass** | resolve_final_qty priority (1) is VALID_POSITIVE → DETECTED with that qty. |
| Inference not triggered for insufficient evidence | **Pass** | has_valid_evidence=False (and/or not product-present) leads to branch (4) with is_resolved=False and qty_final=0; mapper persists "unresolved." |
| Persistence and API use the same resolved decision | **Pass** | For positions created by current mapper, API reads ProductRecord written by same resolution; legacy path recomputes with same resolver and same rules, so only divergence is when summary JSON is out of sync (e.g. empty qty_source). |

## 6. Test Coverage Audit

### 6.1 Existing relevant tests

- **Domain resolution** (`tests/domain/test_qty_resolution_v3_2_2.py`): Valid evidence + missing/null/zero (inferred 1); explicit qty wins; invalid evidence → no inference, is_resolved=False; consolidated qty; zero only when allowed; unresolved path does not leak as valid qty=0.
- **API summary mapping** (`tests/api/test_position_summary_mapping.py`): SKU/quantity from summary; position_to_summary with/without primary_product; authoritative primary product; unresolved primary → (0, "detected", None); legacy fallback; regression tests for no unjustified null qty and no unjustified zero when ProductRecord has inferred 1.
- **Mapper + persist** (`tests/infrastructure/pipeline/test_v3_report_mapper_and_persist.py`): Empty entities; one entity with explicit qty; needs_review and no artifact; COUNTED + evidence + missing qty → inferred 1 and qty_source in ProductRecord and summary; insufficient evidence → qty_source=unresolved and qty_is_resolved in summary; position_barcode/review_display_label.

### 6.2 Missing tests

- **Persistence round-trip:** Save a ProductRecord with qty_source in {detected, inferred, unresolved}, qty_inference_reason, raw_qty, qty_parse_status; reload via repository; assert equality (and that raw_qty_json round-trips for a few shapes).
- **List endpoint with ProductRecordRepository:** Integration test that GET list returns summaries whose qty/qtySource match the ProductRecords stored for those positions (and that legacy path is used when product has no qty_source).
- **EMPTY_PALLET explicit zero:** Mapper test with entity_type=EMPTY_PALLET, final_quantity=0 or product_label_quantity=0, and assert qty_final=0, is_resolved=True, qty_source detected (or equivalent).

### 6.3 Release-blocking gaps

- **None.** The description/None bug (H1) is worth fixing before release but is not a qty-semantics gap. The missing tests above are recommended for confidence, not strictly release-blocking given current coverage.

## 7. Recommended Next Steps

1. **Fix H1 (description):** In `sql_product_record_repository._row_to_product`, coerce NULL description to `""` (or adopt `Optional[str]` and handle None everywhere).  
2. **Document H2:** In `position_to_summary` or in a short design note, state that when `qty_source` is missing/empty we intentionally use the legacy path for backward compatibility.  
3. **Optional:** Add `qtyResolved?: boolean` to the API summary when using legacy path and set it from `res.is_resolved` so the frontend can distinguish unresolved.  
4. **Tests:** Add persistence round-trip test for ProductRecord qty fields; add one integration test for list positions with ProductRecord-driven qty; add EMPTY_PALLET zero test in mapper.  
5. **Docs:** Document in API/spec that `qty` is resolved detected/inferred and `corrected_quantity` is the review override when present.
