# **Dinamic Inventory v3.2.2 — Regla de Conteo Mínimo y Hardening de `qty`**

## **1\. Resumen ejecutivo**

La versión **3.2.2** tiene como objetivo corregir y endurecer la lógica de cantidad (`qty`) dentro del pipeline de inventario para evitar resultados inconsistentes cuando existe **evidencia válida de una etiqueta, SKU o producto**, pero el sistema no logra extraer una cantidad explícita.

Actualmente, en ciertos casos el sistema termina propagando valores `0` o `null` en situaciones donde, desde el punto de vista del negocio, ya existe evidencia suficiente para afirmar que **al menos una unidad** del producto fue observada. Esto genera dos problemas principales:

1. **Inconsistencia semántica** entre “producto detectado” y “cantidad reportada”.  
2. **Mala experiencia de consumo** para frontend, reportes y revisiones operativas, que reciben `qty` nulo o cero aun cuando hay evidencia material del producto.

Para resolverlo, esta versión introduce una **regla formal de inferencia mínima de cantidad**, junto con el hardening de todo el flujo de datos asociado:

* prompt al modelo,  
* parseo de respuesta,  
* consolidación intermedia,  
* DTOs finales,  
* persistencia,  
* y exposición al frontend.

Además, la solución incorpora **metadata explícita** para distinguir cuándo la cantidad fue:

* **detectada** directamente por el sistema, o  
* **inferida** por regla de negocio.

El objetivo no es mejorar conteo múltiple avanzado ni reentrenar modelos, sino asegurar que el sistema nunca reporte `null` o `0` de forma injustificada cuando ya existe evidencia válida de presencia de producto.

---

## **2\. Problema que resuelve**

En el pipeline actual existen escenarios donde una posición, evidencia o agrupación consolidada logra:

* detectar una etiqueta,  
* identificar un producto,  
* o vincular evidencia visual válida a un SKU,

pero el valor de cantidad no queda explícitamente informado por el modelo o se pierde en alguna etapa intermedia. En esos casos, el sistema puede terminar devolviendo:

* `qty = null`, o  
* `qty = 0`,

lo cual contradice el hecho observable de que existe al menos un producto asociado a dicha evidencia.

Desde una perspectiva funcional, si el sistema pudo validar la existencia de una evidencia confiable, entonces el piso lógico del conteo no debería caer por debajo de **1 unidad**, salvo que exista evidencia explícita que justifique otra interpretación.

Esta versión estandariza esa decisión y la convierte en una regla de negocio formal, trazable y testeable.

---

## **3\. Objetivo**

Definir e implementar una política consistente para que, cuando exista **evidencia válida de un producto o etiqueta** pero no haya cantidad explícita disponible, el sistema asigne:

```
qty_final = 1
```

como **cantidad mínima inferida**, en lugar de `0` o `null`.

Adicionalmente, la implementación debe:

* preservar la distinción entre cantidad detectada e inferida,  
* evitar ambigüedad en persistencia y DTOs,  
* impedir que el frontend tenga que reconstruir esta lógica,  
* y dejar la regla documentada y cubierta por tests.

---

## **4\. Alcance**

Esta versión incluye la revisión y endurecimiento de los siguientes componentes del sistema:

### **4.1 Prompt actual**

Se revisará el prompt utilizado por el proveedor/modelo para:

* desalentar respuestas con `qty: null` cuando haya evidencia clara de presencia,  
* indicar explícitamente cómo debe comportarse el modelo si no puede leer una cantidad pero sí detecta un único producto o etiqueta válida,  
* y mejorar la consistencia semántica entre presencia, identificación y cantidad.

### **4.2 Parseo de respuesta del modelo**

Se revisará la capa que transforma la salida del modelo a estructuras internas para:

* normalizar `qty` ausentes,  
* distinguir ausencia real de parseo defectuoso,  
* capturar metadatos sobre origen del valor,  
* y evitar que `null` inválidos pasen sin validación.

### **4.3 Consolidación intermedia**

Se revisará la etapa donde múltiples evidencias, detecciones o resultados se unifican para:

* aplicar la regla mínima en un punto único y controlado,  
* evitar duplicidad de decisiones entre pasos,  
* y asegurar que el valor consolidado final sea coherente.

### **4.4 DTOs finales y persistencia**

Se ajustarán modelos de datos y DTOs para:

* garantizar que el frontend reciba un `qty` consistente,  
* persistir si la cantidad fue detectada o inferida,  
* y registrar el motivo de inferencia.

### **4.5 Regla formal de inferencia mínima**

Se documentará la regla exacta, incluyendo:

* precondiciones,  
* casos permitidos,  
* exclusiones,  
* precedencia frente a otros valores,  
* y trazabilidad.

### **4.6 Metadata explicativa**

Se incorporarán campos que permitan distinguir claramente entre:

* cantidad detectada,  
* cantidad inferida,  
* origen de la inferencia,  
* nivel de confianza disponible.

---

## **5\. Fuera de alcance**

Esta versión **no incluye**:

* reentrenamiento de modelos,  
* cambios grandes en OCR,  
* optimización avanzada de conteo múltiple,  
* reglas complejas específicas por cliente,  
* heurísticas de multiplicidad por facing, profundidad o patrón espacial,  
* ni rediseños amplios del pipeline de detección.

El foco está puesto en la **consistencia del valor `qty`** cuando ya existe evidencia suficiente para afirmar presencia mínima.

---

## **6\. Definiciones operativas**

Para evitar ambigüedad, en esta versión se utilizarán las siguientes definiciones:

### **6.1 Evidencia válida**

Se considera **evidencia válida** a cualquier evidencia visual o resultado procesado que cumpla las condiciones mínimas de confianza y asociación requeridas por el pipeline para sostener que un producto o etiqueta fue observado.

Ejemplos:

* detección visual asociada a una posición,  
* crop o imagen vinculada a un SKU identificado,  
* etiqueta legible asociada a una posición válida,  
* evidencia principal de una posición marcada como detectada.

### **6.2 Cantidad detectada**

Valor numérico de cantidad obtenido directamente desde:

* la salida del modelo,  
* OCR,  
* lógica de conteo explícita,  
* o una etapa estructurada del pipeline que produzca una cantidad concreta.

### **6.3 Cantidad inferida**

Valor de cantidad asignado por regla de negocio cuando:

* existe evidencia válida,  
* no hay cantidad explícita confiable,  
* y el sistema necesita devolver un resultado consistente.

### **6.4 Cantidad mínima inferida**

Caso particular de cantidad inferida donde el valor asignado es exactamente:

```
1
```

porque la evidencia permite afirmar presencia, pero no cardinalidad mayor.

---

## **7\. Regla de negocio formal**

### **7.1 Regla principal**

Si existe **al menos una evidencia válida** asociada a un producto, etiqueta o posición consolidada, y **no existe una cantidad explícita válida**, entonces el sistema debe asignar:

```
qty_final = 1
```

como **cantidad mínima inferida**.

### **7.2 Precondiciones**

La inferencia mínima solo puede aplicarse si se cumplen todas las siguientes condiciones:

1. Existe evidencia válida asociada al resultado.  
2. La evidencia permite afirmar presencia de producto o etiqueta con confianza suficiente según las reglas actuales del pipeline.  
3. No existe una cantidad explícita válida mayor o igual a 1\.  
4. El valor `null`, vacío, ausente o inválido no proviene de una regla que indique explícitamente “cantidad desconocida pero no inferible”.

### **7.3 Precedencia**

La siguiente jerarquía define qué valor debe prevalecer:

1. **Cantidad detectada explícita válida**.  
2. **Cantidad consolidada válida derivada de múltiples fuentes**.  
3. **Cantidad mínima inferida \= 1**.  
4. Rechazo o exclusión del resultado, si no hay evidencia suficiente.

### **7.4 Casos donde no debe aplicarse automáticamente**

La inferencia mínima no debe aplicarse de forma automática cuando:

* la evidencia fue descartada por baja confianza,  
* la identificación de producto es demasiado ambigua,  
* el resultado representa una hipótesis no confirmada,  
* la entidad consolidada no alcanzó el umbral mínimo de validez,  
* o el pipeline no puede afirmar presencia real del producto.

### **7.5 Resultado esperado**

Cuando la regla aplique, el sistema debe devolver un resultado equivalente a:

* `qty = 1`  
* `qty_source = inferred`  
* `qty_inference_reason = valid_evidence_without_explicit_quantity`

---

## **8\. Casuística funcional**

### **8.1 Caso A — Evidencia única válida sin cantidad explícita**

**Entrada esperada:**

* hay una única evidencia válida,  
* el producto o etiqueta fue identificado correctamente,  
* `qty` no viene informado.

**Salida esperada:**

* `qty_final = 1`  
* `qty_source = inferred`

### **8.2 Caso B — Cantidad explícita detectada**

**Entrada esperada:**

* el modelo informa `qty = 3` de forma válida.

**Salida esperada:**

* `qty_final = 3`  
* `qty_source = detected`

### **8.3 Caso C — `qty = 0` pero existe evidencia válida de presencia**

**Interpretación esperada:**

Debe revisarse si ese `0` es semánticamente válido o si representa una degradación de parseo/modelo. En principio, si el sistema afirma presencia válida del producto, `0` entra en conflicto con esa afirmación.

**Salida esperada:**

* si `0` no tiene justificación de negocio explícita, debe normalizarse mediante la regla mínima,  
* resultando en `qty_final = 1`.

### **8.4 Caso D — Sin evidencia suficiente**

**Entrada esperada:**

* el resultado no tiene evidencia confiable,  
* o la identificación no superó el umbral mínimo.

**Salida esperada:**

* no debe inferirse `1`,  
* el resultado debe permanecer descartado o no consolidado.

### **8.5 Caso E — Múltiples evidencias pero cardinalidad incierta**

**Entrada esperada:**

* existen varias evidencias asociadas,  
* pero el sistema no puede estimar cardinalidad real sin riesgo de sobreconteo.

**Salida esperada:**

* la regla mínima puede seguir garantizando `qty >= 1`,  
* pero no debe escalar a valores mayores sin soporte explícito.

---

## **9\. Diseño técnico propuesto**

## **9.1 Principio general**

La decisión de inferir cantidad mínima debe implementarse en un **punto central del backend**, evitando que la lógica quede repartida entre:

* proveedor/modelo,  
* parseador,  
* consolidación,  
* persistencia,  
* y frontend.

El modelo puede ser orientado por prompt, pero la **fuente de verdad** debe residir en la lógica backend del dominio o de consolidación final.

### **9.2 Revisión del prompt**

El prompt debe ajustarse para reducir ambigüedad en la respuesta del modelo. Debe dejar claro que:

* si detecta presencia clara de un único producto pero no puede leer una cantidad explícita,  
* no debe devolver `0` por defecto,  
* y debe priorizar coherencia semántica entre evidencia y cantidad.

Sin embargo, el prompt no debe considerarse control suficiente por sí solo. La protección definitiva debe estar en backend.

### **9.3 Revisión del parseo**

El parser debe distinguir entre distintos escenarios:

* campo `qty` ausente,  
* campo `qty = null`,  
* campo `qty = 0`,  
* campo `qty` inválido por tipo o formato,  
* campo `qty` válido positivo.

Además, debe permitir registrar un estado intermedio como:

* `raw_qty`,  
* `normalized_qty`,  
* `qty_parse_status`.

Esto facilita debugging y auditoría.

### **9.4 Punto recomendado para aplicar la regla**

La inferencia mínima debería aplicarse en la capa de **consolidación de dominio** o en una función central de normalización previa al DTO final.

Razones:

* es el punto donde ya existe contexto suficiente para saber si la evidencia es válida,  
* evita aplicar inferencia demasiado temprano,  
* evita depender del comportamiento exacto del proveedor,  
* y garantiza consistencia para persistencia y frontend.

### **9.5 Persistencia de metadata**

La persistencia debe registrar no solo el valor final, sino también su procedencia.

Campos recomendados:

* `qty_final`  
* `qty_source` → `detected | inferred`  
* `qty_inference_reason` → string o enum  
* `raw_qty` → opcional  
* `qty_confidence` → opcional si aplica

### **9.6 DTO final**

El contrato hacia frontend debe ser explícito y estable. El DTO debería exponer como mínimo:

* `qty`  
* `qtySource`  
* `qtyInferenceReason` (nullable)

Con esto, el frontend deja de tener que adivinar si un `1` es detectado o inferido.

---

## **10\. Modelo de datos recomendado**

A nivel conceptual, se recomienda el siguiente esquema lógico:

```ts
qty: number
qtySource: 'detected' | 'inferred'
qtyInferenceReason?: 'valid_evidence_without_explicit_quantity' | null
rawQty?: number | null
```

Si el backend trabaja en Python, la misma idea debería reflejarse en los modelos internos y DTOs serializados.

### **10.1 Recomendaciones de validación**

* `qty` final no debería ser `null` si la entidad consolidada fue aceptada como válida y visible para frontend.  
* `qty` final no debería ser negativo.  
* `qtySource = inferred` requiere `qty >= 1`.  
* `qtyInferenceReason` debería ser obligatorio cuando `qtySource = inferred`.

## **11\. Estrategia de implementación**

### **11.1 Paso 1 — Auditoría del flujo actual de `qty`**

Mapear con precisión:

* dónde entra `qty`,  
* cómo llega desde el modelo,  
* dónde se transforma,  
* dónde puede convertirse en `null` o `0`,  
* y qué capa construye el DTO final.

### **11.2 Paso 2 — Hardening del parser**

Agregar normalización explícita de salida del modelo y captura de estados inválidos.

### **11.3 Paso 3 — Introducción de regla central de inferencia mínima**

Implementar una función única de negocio, por ejemplo:

```
resolve_final_qty(...)
```

que decida el valor final y su metadata asociada.

### **11.4 Paso 4 — Ajuste de persistencia**

Guardar origen y motivo de la cantidad.

### **11.5 Paso 5 — Ajuste de DTOs / contratos**

Exponer al frontend los nuevos campos y garantizar consistencia.

### **11.6 Paso 6 — Testing**

Cubrir con tests unitarios y de integración los casos más importantes.

---

## **12\. Estrategia de testing**

## **12.1 Unit tests**

Se deben cubrir como mínimo estos escenarios:

1. Evidencia válida \+ `qty = null` → `qty_final = 1`  
2. Evidencia válida \+ `qty` ausente → `qty_final = 1`  
3. Evidencia válida \+ `qty = 0` injustificado → `qty_final = 1`  
4. Evidencia válida \+ `qty = 2` → `qty_final = 2`  
5. Sin evidencia válida \+ `qty = null` → no inferir  
6. `qtySource` y `qtyInferenceReason` correctos en cada caso

### **12.2 Integration tests**

Casos end-to-end o semi integrados:

* respuesta del modelo sin cantidad explícita,  
* parseo correcto,  
* consolidación correcta,  
* persistencia correcta,  
* DTO final correcto hacia frontend.

### **12.3 Regression tests**

Agregar cobertura para casos históricos donde hoy se observa:

* `qty = null` con SKU correcto,  
* `qty = 0` en posiciones con evidencia,  
* diferencia entre ramas o prompts previos.

---

## **13\. Riesgos y mitigaciones**

### **13.1 Riesgo: sobreconteo en casos ambiguos**

**Descripción:** inferir `1` donde la evidencia no era suficientemente fuerte.

**Mitigación:** aplicar la regla solo cuando la entidad ya haya sido validada como evidencia confiable.

### **13.2 Riesgo: mezclar detección con identificación confiable**

**Descripción:** asumir presencia real cuando solo existe una hipótesis débil.

**Mitigación:** separar claramente umbrales de validez de evidencia y umbrales de identificación.

### **13.3 Riesgo: duplicación de lógica entre backend y frontend**

**Descripción:** que frontend vuelva a inferir o reinterpretar cantidad.

**Mitigación:** el backend debe exponer el valor final y su source de forma explícita.

### **13.4 Riesgo: valores inconsistentes entre persistencia y DTO**

**Descripción:** persistir una cosa y devolver otra.

**Mitigación:** reutilizar una única función de resolución final antes de persistir y serializar.

---

## **14\. Dependencias**

Esta versión conviene ejecutarse después de **3.2.1** por orden de endurecimiento funcional, pero técnicamente puede desarrollarse en paralelo si:

* el contrato interno de consolidación ya está razonablemente estable,  
* y la capa de DTO no está siendo modificada de forma incompatible.

---

## **15\. Entregables**

La versión 3.2.2 debe dejar los siguientes entregables:

1. **Regla de negocio documentada** para cantidad mínima inferida.  
2. **Cambios en pipeline/backend** para inferencia mínima consistente.  
3. **Persistencia del motivo de inferencia** y origen del valor.  
4. **DTOs finales endurecidos** con `qty` consistente.  
5. **Tests unitarios e integración** para escenarios conocidos.  
6. **Criterios de observabilidad/debug** para rastrear por qué un `qty` fue inferido.

---

## **16\. Criterios de aceptación**

La versión se considera aceptada si se cumplen todos los siguientes puntos:

1. Si hay una única evidencia válida sin cantidad explícita, entonces `qty_final = 1`.  
2. No aparecen `null` injustificados en resultados con evidencia válida.  
3. No aparecen `0` injustificados cuando existe presencia válida del producto.  
4. El frontend recibe siempre un valor consistente de cantidad para resultados válidos.  
5. Se puede distinguir si la cantidad fue detectada o inferida.  
6. La lógica queda centralizada en backend y no depende del frontend.  
7. Existen tests que cubren casos reales conocidos.

---

## **17\. Recomendaciones de implementación**

Para mantener el cambio controlado y profesional, se recomienda:

* centralizar la decisión en una única función de dominio,  
* no depender exclusivamente del prompt,  
* no permitir que el frontend reconstruya la semántica de `qty`,  
* registrar metadata suficiente para observabilidad,  
* y dejar trazabilidad clara en logs o estructuras persistidas cuando se aplique inferencia mínima.

---

## **18\. Conclusión**

La versión **3.2.2** no busca contar mejor en términos avanzados, sino contar de forma **más coherente, robusta y consumible** cuando ya existe evidencia válida de producto.

El principio central es simple:

si el sistema pudo afirmar presencia válida, no debe devolver una cantidad vacía o contradictoria.

Formalizar esta regla mejora:

* la integridad del pipeline,  
* la calidad de los datos consumidos por frontend,  
* la experiencia operativa de revisión,  
* y la trazabilidad del comportamiento del sistema.

En términos de producto, esta versión reduce ruido, elimina resultados semánticamente incorrectos y prepara mejor el terreno para futuras mejoras de conteo más sofisticadas sin romper la consistencia actual.
