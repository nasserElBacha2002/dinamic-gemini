# **Dinamic Inventory v3.0 — Reglas de Estilo del Proyecto**

## **1\. Objetivo**

Este documento define las reglas de estilo visual, estructural y de implementación del frontend de **Dinamic Inventory v3.0**.

Su propósito es asegurar consistencia en:

* la interfaz web,  
* la experiencia del usuario,  
* el uso de componentes,  
* la estructura visual,  
* la mantenibilidad del código,  
* y la escalabilidad del proyecto.

La librería principal de interfaz y estilos del proyecto es **Material UI (MUI)**.

---

## **2\. Principios generales de estilo**

El frontend debe seguir estos principios:

### **2.1 Claridad operativa**

La interfaz debe priorizar comprensión rápida y lectura clara. Este sistema está orientado a operación y revisión de inventarios, no a una experiencia visual ornamental.

### **2.2 Consistencia**

Componentes equivalentes deben verse y comportarse de la misma manera en toda la aplicación.

### **2.3 Jerarquía visual simple**

La información más importante debe destacarse sin sobrecargar la pantalla. Debe existir una jerarquía visual clara entre:

* navegación,  
* contexto de página,  
* métricas,  
* tabla de resultados,  
* acciones principales,  
* acciones secundarias.

### **2.4 Escalabilidad**

Las decisiones de estilo no deben depender de pantallas aisladas. Cada componente debe poder reutilizarse en nuevas vistas sin redefinir estilos desde cero.

### **2.5 Accesibilidad básica**

La interfaz debe mantener contrastes adecuados, tamaños legibles, foco visible y estados claros.

---

## **3\. Stack visual oficial**

El stack oficial del frontend es:

* **React**  
* **TypeScript**  
* **Material UI**  
* **MUI Data Grid** para tablas complejas  
* **React Router**  
* **TanStack Query**

No se debe usar **TanStack Table** como sustituto de MUI Data Grid en vistas operativas (posiciones, inventarios, pasillos, productos, historial de revisión); para esas vistas el estándar es **MUI Data Grid**. No se deben introducir nuevas librerías de estilos sin justificación técnica clara.

---

## **4\. Regla principal de estilos**

### **4.1 Material UI como fuente principal de UI**

Todos los componentes visuales deben construirse prioritariamente con **Material UI**.

Esto incluye:

* layout,  
* inputs,  
* botones,  
* tablas,  
* chips,  
* diálogos,  
* cards,  
* drawers,  
* tabs,  
* loaders,  
* alerts.

### **4.2 Evitar mezclar múltiples sistemas de estilos**

No se debe mezclar indiscriminadamente:

* CSS Modules,  
* Tailwind,  
* Styled Components,  
* Emotion custom excesivo,  
* hojas globales extensas,  
* librerías visuales paralelas.

La regla es:

* usar **MUI** como base,  
* usar `sx` para personalizaciones simples y locales,  
* usar `theme` para estilos globales y tokens compartidos,  
* crear wrappers propios cuando se repite una misma configuración.

---

## **5\. Estrategia de estilado**

### **5.1 Prioridad de uso**

Orden recomendado:

1. Componente nativo de MUI.  
2. Props estándar del componente.  
3. `sx` para ajustes locales simples.  
4. `theme` para configuración global y tokens.  
5. Wrapper reutilizable del proyecto.

### **5.2 Cuándo usar `sx`**

Usar `sx` para:

* espaciado puntual,  
* alineaciones,  
* tamaños simples,  
* ajustes visuales locales,  
* variantes pequeñas de layout.

No usar `sx` para reescribir completamente el estilo de un componente en cada pantalla.

### **5.3 Cuándo usar wrappers**

Si una misma configuración se repite en más de una vista, debe abstraerse en un componente propio.

Ejemplos recomendados:

* `StatusChip`  
* `PageSection`  
* `PageHeader`  
* `MetricCard`  
* `LoadingBlock`  
* `EmptyState`  
* `ReviewDialog`  
* `EvidenceGallery`

---

## **6\. Tema global**

Debe existir un `theme` centralizado del proyecto.

## **6.1 Qué debe definir el theme**

Como mínimo:

* paleta de colores,  
* tipografía,  
* espaciado base,  
* radios de borde,  
* elevaciones,  
* overrides de componentes,  
* estados visuales comunes.

## **6.2 Regla de consistencia**

No se deben hardcodear colores, sombras, radios o tamaños repetidos en múltiples componentes si ya pueden vivir en el theme.

---

## **7\. Colores**

## **7.1 Principio**

La aplicación debe tener una identidad visual profesional y limpia, orientada a operación. El color debe usarse para comunicar estado y prioridad, no para decorar.

## **7.2 Uso recomendado del color**

### **Primario**

Para:

* acciones principales,  
* navegación activa,  
* elementos de foco,  
* acciones de confirmación no destructiva.

### **Secundario / neutros**

Para:

* bordes,  
* fondos suaves,  
* separaciones,  
* superficies.

### **Colores semánticos**

Para estados:

* éxito,  
* advertencia,  
* error,  
* información.

## **7.3 Regla**

No usar colores arbitrarios por pantalla. Los colores deben salir del theme o de los tokens semánticos definidos.

---

## **8\. Tipografía**

## **8.1 Principio**

La tipografía debe priorizar legibilidad, estructura y ritmo visual.

## **8.2 Reglas**

* Evitar tamaños demasiado pequeños.  
* Mantener consistencia entre títulos, subtítulos, labels y texto de tabla.  
* No usar muchos pesos tipográficos distintos en una misma vista.  
* Los títulos deben ser claros pero sobrios.

## **8.3 Jerarquía mínima sugerida**

* Título de página  
* Subtítulo o contexto  
* Título de sección  
* Texto de tabla  
* Texto auxiliar  
* Caption o metadata

---

## **9\. Espaciado**

## **9.1 Principio**

El espaciado debe ser consistente y predecible.

## **9.2 Regla**

Usar el sistema de spacing de MUI. Evitar márgenes y paddings aleatorios hardcodeados sin criterio común.

## **9.3 Aplicación**

El espaciado debe ayudar a separar:

* navegación y contenido,  
* métricas y tablas,  
* acciones primarias y secundarias,  
* grupos de formulario,  
* bloques visuales relacionados.

---

## **10\. Bordes, radios y elevación**

## **10.1 Regla general**

Los radios, bordes y sombras deben ser moderados. La UI no debe sentirse recargada ni excesivamente decorativa.

## **10.2 Aplicación**

* Cards con elevación sutil.  
* Diálogos claros y limpios.  
* Inputs consistentes.  
* Superficies bien separadas pero sin exceso de profundidad.

---

## **11\. Layout**

## **11.1 Principio general**

El layout debe estar pensado para una herramienta operativa, con foco en productividad y revisión.

## **11.2 Estructura recomendada**

Cada página debería tender a esta estructura:

1. Contexto superior de la página  
2. Acciones principales  
3. Resumen o métricas si aplica  
4. Contenido principal  
5. Paneles secundarios o detalle

## **11.3 Componentes sugeridos**

* `AppBar`  
* `Drawer`  
* `Container`  
* `Box`  
* `Stack`  
* `Grid`  
* `Card`  
* `Paper`

## **11.4 Regla**

Evitar layouts profundamente anidados y estructuras difíciles de mantener.

---

## **12\. Tablas y grillas de datos**

## **12.1 Regla principal**

Las tablas complejas deben implementarse con **MUI Data Grid**.

## **12.2 Casos recomendados para Data Grid**

* resultados por posición,  
* inventarios,  
* pasillos,  
* productos detectados,  
* historial de revisiones.

## **12.3 Reglas visuales de tablas**

* columnas claras y con nombres consistentes,  
* evitar exceso de información en una sola celda,  
* usar chips para estados,  
* usar miniaturas controladas para evidencia,  
* acciones visibles pero no invasivas,  
* filtros cuando aporten valor operativo.

## **12.4 Regla de rendimiento**

No renderizar tablas manuales complejas si Data Grid ya resuelve el caso.

---

## **13\. Estados visuales**

## **13.1 Estados de negocio**

Los estados del sistema deben representarse de forma consistente mediante componentes reutilizables.

Ejemplo recomendado:

* `StatusChip`

## **13.2 Estados mínimos a contemplar**

* loading  
* empty  
* error  
* success  
* processing  
* queued  
* completed  
* deleted  
* needs review

## **13.3 Regla**

Nunca resolver estados críticos solo con texto plano. Debe haber apoyo visual consistente.

---

## **14\. Formularios**

## **14.1 Principio**

Los formularios deben ser simples, directos y fáciles de completar.

## **14.2 Reglas**

* Labels claros.  
* Validaciones visibles.  
* Mensajes de error concretos.  
* Agrupación lógica de campos.  
* Botones primarios y secundarios diferenciados.

## **14.3 Componentes sugeridos**

* `TextField`  
* `Select`  
* `Autocomplete`  
* `Checkbox`  
* `RadioGroup`  
* `Dialog`  
* `Button`

---

## **15\. Botones y acciones**

## **15.1 Regla general**

Debe existir una jerarquía clara entre acciones primarias, secundarias y destructivas.

## **15.2 Guía**

### **Acción primaria**

Usar para la acción principal de la pantalla.

### **Acción secundaria**

Usar para acciones de apoyo o navegación.

### **Acción destructiva**

Usar variantes claras de error o warning para eliminación o descarte.

## **15.3 Regla**

No debe haber múltiples botones compitiendo visualmente como si todos fueran primarios.

---

## **16\. Diálogos, drawers y modales**

## **16.1 Uso recomendado**

### **Dialog**

Para confirmar acciones o editar datos puntuales.

### **Drawer**

Para ver detalle contextual sin perder la tabla principal.

### **Modal**

Solo cuando realmente aporte y no exista mejor alternativa.

## **16.2 Regla**

No abusar de overlays. La revisión debe sentirse fluida.

---

## **17\. Composición de componentes**

## **17.1 Regla general**

Los componentes visuales deben tener responsabilidad clara.

## **17.2 Separación sugerida**

### **Componentes de presentación**

Solo renderizan UI.

### **Componentes contenedores o hooks**

Resuelven fetching, coordinación y estado de interacción.

## **17.3 Ejemplos recomendados**

* `PositionsTable`  
* `PositionDetailPanel`  
* `MetricCard`  
* `EvidenceGallery`  
* `StatusChip`  
* `ReviewHistoryList`

---

## **18\. Reglas de código de estilos**

## **18.1 No duplicar configuración visual**

Si el mismo bloque de estilos aparece repetido, debe abstraerse.

## **18.2 Evitar estilos inline arbitrarios**

No usar estilos inline desorganizados salvo casos mínimos y excepcionales.

## **18.3 Preferir tokens del theme**

Antes de escribir un color o spacing manual, verificar si ya existe en el theme.

## **18.4 Mantener nombres semánticos**

Los componentes y wrappers deben llamarse según su función, no según su apariencia visual pasajera.

Correcto:

* `StatusChip`  
* `MetricCard`  
* `PageHeader`

Evitar:

* `BlueBox`  
* `BigTableWrapper`  
* `FancyCard`

---

## **19\. Reglas de consistencia visual por módulo**

## **19.1 Inventarios**

Vista orientada a navegación y resumen.

## **19.2 Pasillos**

Vista orientada a estado de procesamiento y acceso a resultados.

## **19.3 Posiciones**

Vista orientada a revisión detallada y evidencia.

## **19.4 Métricas**

Vista orientada a lectura rápida de KPIs.

## **19.5 Revisión**

Vista orientada a precisión operativa, no a complejidad visual.

---

## **20\. Accesibilidad mínima**

Se deben respetar como mínimo estas reglas:

* foco visible,  
* contraste suficiente,  
* labels correctos,  
* botones distinguibles,  
* estados no dependientes solo del color,  
* tamaños razonables para lectura y clic.

---

## **21\. Responsabilidad arquitectónica y SOLID en UI**

## **21.1 Single Responsibility**

Un componente no debe encargarse simultáneamente de:

* renderizar UI compleja,  
* resolver llamadas HTTP,  
* transformar datos de dominio,  
* manejar reglas de negocio.

## **21.2 Open/Closed**

Los componentes base deben poder extenderse mediante props y composición sin tener que reescribirse.

## **21.3 Interface Segregation**

No crear componentes gigantes que intenten cubrir demasiados casos de uso.

## **21.4 Dependency Inversion**

La UI debe depender de servicios, hooks o adaptadores claros, no de detalles internos de implementación del backend.

---

## **22\. Componentes base recomendados del proyecto**

Se recomienda crear una capa de componentes compartidos:

* `PageLayout`  
* `PageHeader`  
* `PageSection`  
* `StatusChip`  
* `MetricCard`  
* `LoadingBlock`  
* `ErrorState`  
* `EmptyState`  
* `EvidenceGallery`  
* `ConfirmDialog`  
* `ReviewDialog`

Esto permitirá mantener coherencia sin repetir definiciones visuales.

---

## **23\. Reglas de evolución futura**

Cuando se incorpore nueva funcionalidad:

* primero debe evaluarse si puede resolverse con componentes existentes,  
* luego si requiere extender un componente compartido,  
* y solo en último lugar crear un patrón visual completamente nuevo.

La evolución de la UI debe ser acumulativa y consistente.

---

## **24\. Conclusión**

La guía de estilos del proyecto se resume en estas reglas centrales:

* **Material UI es la base oficial del frontend.**  
* **El theme centraliza la identidad visual.**  
* **La UI debe ser clara, operativa y consistente.**  
* **Los componentes deben ser reutilizables y de responsabilidad acotada.**  
* **Las decisiones visuales deben escalar con el sistema.**  
* **La implementación debe acompañar los principios SOLID también en la capa de interfaz.**

Este documento debe utilizarse como referencia para toda nueva pantalla, componente o refactor visual dentro de Dinamic Inventory v3.0.
