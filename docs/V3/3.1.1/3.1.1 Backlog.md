# **Backlog técnico — Dinamic Inventory v3.1.1**

## **1\. Objetivo del backlog**

Implementar la versión **v3.1.1** para consolidar el flujo de revisión alrededor de **Results**, eliminando la dependencia operativa de **Entities** como vista separada, rediseñando la UI principal, enriqueciendo el detalle del resultado y agregando navegación secuencial entre resultados.

---

# **2\. Estrategia de implementación**

La implementación conviene hacerla en este orden:

1. **Definir nuevo contrato funcional**  
2. **Adaptar backend DTOs y endpoints**  
3. **Refactorizar frontend data layer**  
4. **Rediseñar Results list**  
5. **Rediseñar Result detail**  
6. **Agregar navegación previous/next**  
7. **Deprecar Entities view**  
8. **QA funcional y regresión**

---

# **3\. Épicas**

## **Épica 1 — Consolidación del modelo visible de revisión**

Objetivo: convertir `Result` en la única unidad principal visible de revisión en frontend.

## **Épica 2 — Nuevo contrato backend para list/detail**

Objetivo: exponer desde backend todo lo necesario para que Result list y Result detail funcionen sin depender de una pantalla de Entities.

## **Épica 3 — Rediseño UI de Results**

Objetivo: simplificar la tabla principal y mejorar escaneabilidad.

## **Épica 4 — Rediseño UI de Result Detail**

Objetivo: concentrar evidencia, acciones y metadata en una sola pantalla usable.

## **Épica 5 — Navegación secuencial dentro del detalle**

Objetivo: permitir revisar resultados en continuidad.

## **Épica 6 — Deprecación de View Entities**

Objetivo: remover la vista separada de Entities del flujo principal.

## **Épica 7 — QA, hardening y rollout**

Objetivo: validar funcionalidad, estabilidad y compatibilidad con datos reales.

### **Historia 2.3 — Agregar contexto de navegación previous/next**

**Como** frontend  
**Quiero** conocer el anterior y siguiente resultado dentro de un conjunto filtrado  
**Para** navegar sin volver a la tabla

### **Alternativas**

#### **Opción A**

Resolver previous/next desde frontend usando el listado ya cargado.  
Más simple y recomendable para v3.1.1.

#### **Opción B**

Endpoint backend que reciba:

* current result id  
* current filters  
* current sort  
  y devuelva:  
* previousId  
* nextId

### **Recomendación**

Empezar con **Opción A** y escalar a B solo si aparece paginación compleja o filtros remotos.

### **Código sugerido — helper frontend**

```ts
export interface ResultNavigationContext {
  ids: string[];
  currentIndex: number;
  previousId: string | null;
  nextId: string | null;
}

export function buildResultNavigationContext(
  ids: string[],
  currentId: string
): ResultNavigationContext {
  const currentIndex = ids.findIndex((id) => id === currentId);

  return {
    ids,
    currentIndex,
    previousId: currentIndex > 0 ? ids[currentIndex - 1] : null,
    nextId: currentIndex >= 0 && currentIndex < ids.length - 1 ? ids[currentIndex + 1] : null,
  };
}
```

### **Código sugerido — función de agregación**

```ts
import type { ResultSummary } from './types';

export interface ResultsKpis {
  total: number;
  needsReview: number;
  validTraceability: number;
  missingTraceability: number;
  invalidTraceability: number;
  zeroQty: number;
}

export function buildResultsKpis(rows: ResultSummary[]): ResultsKpis {
  return rows.reduce<ResultsKpis>(
    (acc, row) => {
      acc.total += 1;
      if (row.needsReview) acc.needsReview += 1;
      if (row.traceabilityStatus === 'VALID') acc.validTraceability += 1;
      if (row.traceabilityStatus === 'MISSING') acc.missingTraceability += 1;
      if (row.traceabilityStatus === 'INVALID') acc.invalidTraceability += 1;
      if ((row.detectedQty ?? 0) === 0) acc.zeroQty += 1;
      return acc;
    },
    {
      total: 0,
      needsReview: 0,
      validTraceability: 0,
      missingTraceability: 0,
      invalidTraceability: 0,
      zeroQty: 0,
    }
  );
}
```

### **Historia 3.3 — Filtros rápidos en Results**

**Como** operador  
**Quiero** filtrar el listado  
**Para** trabajar más rápido sobre los casos relevantes

### **Filtros mínimos**

* All  
* Needs review  
* Valid traceability  
* Missing traceability  
* Qty 0  
* Low confidence

### **Código sugerido**

```ts
export type ResultsFilter =
  | 'ALL'
  | 'NEEDS_REVIEW'
  | 'TRACEABILITY_VALID'
  | 'TRACEABILITY_MISSING'
  | 'ZERO_QTY'
  | 'LOW_CONFIDENCE';

export function filterResults(
  rows: ResultSummary[],
  filter: ResultsFilter
): ResultSummary[] {
  switch (filter) {
    case 'NEEDS_REVIEW':
      return rows.filter((r) => r.needsReview);
    case 'TRACEABILITY_VALID':
      return rows.filter((r) => r.traceabilityStatus === 'VALID');
    case 'TRACEABILITY_MISSING':
      return rows.filter((r) => r.traceabilityStatus === 'MISSING');
    case 'ZERO_QTY':
      return rows.filter((r) => (r.detectedQty ?? 0) === 0);
    case 'LOW_CONFIDENCE':
      return rows.filter((r) => (r.confidence ?? 1) < 0.9);
    default:
      return rows;
  }
}
```

# **11\. Criterios globales de aceptación de la versión**

La v3.1.1 se considera completa cuando:

1. **Results** es la única lista principal visible para revisión.  
2. **View Entities** deja de ser necesaria para operar.  
3. El detalle del resultado concentra:  
   * evidencia  
   * metadata relevante  
   * acciones de revisión  
   * historial  
4. El usuario puede navegar entre resultados con **anterior/siguiente**.  
5. La tabla prioriza visualmente **SKU, qty y traceability**.  
6. La UI es más limpia y con menos ruido técnico.  
7. El flujo funciona con datos reales sin regresiones críticas.

* **no borrar Entity del backend todavía**  
* **sí borrarlo del flujo principal de frontend**

* **sí rediseñar fuerte la experiencia de revisión**
