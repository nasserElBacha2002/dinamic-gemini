# **Documento Técnico de Implementación — Dinamic Inventory v3.1**

## **Trazabilidad de Evidencia por Imagen**

**Versión:** 1.0  
**Estado:** Propuesta técnica de implementación  
**Base:** Extensión incremental sobre v3.0  
**Fecha:** 11 de marzo de 2026

---

## **1\. Objetivo técnico**

Implementar en Dinamic Inventory v3.1 una capacidad de **trazabilidad explícita entre imágenes de entrada y resultados devueltos por la API de análisis**, de forma que cada conteo, hallazgo o resultado estructurado pueda indicar **qué imagen lo respalda**.

La implementación debe:

* reutilizar la arquitectura de v3.0,  
* no romper contratos existentes innecesariamente,  
* introducir metadata de imagen de forma consistente,  
* extender prompt, parsing, persistencia y salida,  
* y dejar preparada la base para futuras mejoras de consolidación avanzada.

---

## **2\. Objetivo funcional traducido a implementación**

Dado un job con múltiples imágenes:

1. cada imagen debe recibir un identificador técnico único dentro del job,  
2. ese identificador debe incorporarse al contexto enviado a la API,  
3. la API debe devolver, por cada resultado, un `source_image_id` o equivalente,  
4. el backend debe validar esa referencia,  
5. el resultado estructurado debe persistir esa asociación,  
6. y el sistema debe poder exponerla en respuesta, revisión y exportación.

---

## **3\. Principios de diseño**

### **3.1 Extensión incremental**

La v3.1 debe ser una extensión de v3.0. No se debe rediseñar el pipeline si la funcionalidad puede agregarse sobre los contratos actuales.

### **3.2 Compatibilidad hacia atrás**

Los resultados de v3.0 deben seguir siendo procesables aunque no contengan trazabilidad por imagen.

### **3.3 Trazabilidad como dato de dominio**

La referencia a imagen no debe tratarse como un dato auxiliar o de debugging, sino como parte del resultado funcional.

### **3.4 Validación explícita**

Toda referencia a imagen devuelta por la API debe validarse contra las imágenes realmente cargadas en el job.

### **3.5 Preparación para futuro multi-evidencia**

Aunque la primera versión pueda implementar `source_image_id`, la estructura debe permitir evolucionar a `source_image_ids` y `primary_image_id`.

---

## **4\. Alcance técnico**

### **Incluido en esta implementación**

* identificación de imágenes dentro del job,  
* metadata estructurada de evidencias,  
* cambios en prompt/payload para informar IDs,  
* cambio en contrato esperado de respuesta de la API,  
* parsing y validación de `source_image_id`,  
* persistencia de evidencia por resultado,  
* exposición del dato en DTOs y salidas,  
* flags o warnings de inconsistencia de trazabilidad.

### **Excluido de esta implementación principal**

* rediseño total del frontend,  
* scoring avanzado de calidad de evidencia,  
* consolidación multi-imagen sofisticada,  
* snapshots visuales enriquecidos,  
* motores de ranking de evidencia,  
* reentrenamiento del sistema de visión.

---

## **5\. Arquitectura objetivo**

La capacidad nueva atraviesa estas capas:

1. **Input / Upload Layer**  
   Registro de imágenes cargadas y asignación de identificador interno.  
2. **Analysis Preparation Layer**  
   Construcción de contexto para la API, incluyendo mapping de imágenes.  
3. **Provider Layer**  
   Ajuste del contrato esperado con la API para recibir `source_image_id`.  
4. **Parsing / Normalization Layer**  
   Parseo de resultados y validación de referencias.  
5. **Domain / Application Layer**  
   Incorporación de evidencia por imagen al modelo de resultado.  
6. **Persistence Layer**  
   Guardado de imágenes y su relación con resultados.  
7. **Output / API Layer**  
   Exposición de trazabilidad en responses, revisión y exportaciones.

---

## **6\. Diseño de datos**

## **6.1 Entidad nueva o extendida: ImageEvidence**

Debe existir una representación técnica por cada imagen cargada.

### **Campos mínimos recomendados**

* `image_id: str`  
* `job_id: str`  
* `original_filename: str`  
* `upload_order: int`  
* `storage_path: str`  
* `mime_type: Optional[str]`  
* `created_at: datetime`  
* `checksum: Optional[str]`  
* `width: Optional[int]`  
* `height: Optional[int]`

### **Observaciones**

* `image_id` debe ser estable dentro del job.  
* No debe depender exclusivamente del nombre del archivo.  
* Puede ser algo del estilo:  
  * `img_001`  
  * `img_002`  
  * `img_003`  
* o un UUID corto interno si ya hay convención para ello.

Para revisión humana, conviene conservar también `original_filename`.

---

## **6.2 Extensión de entidad de resultado**

La entidad que hoy represente un resultado contado debe extenderse con trazabilidad.

### **Opción mínima**

* `source_image_id: Optional[str]`

### **Opción recomendada para futuro**

* `source_image_id: Optional[str]`  
* `source_image_ids: list[str] = []`  
* `primary_image_id: Optional[str]`  
* `traceability_status: Literal["valid", "missing", "invalid"]`  
* `traceability_warning: Optional[str]`

La implementación inicial puede persistir solo `source_image_id`, pero la capa de dominio debería contemplar fácilmente la extensión futura.

---

## **6.3 Inconsistencias de trazabilidad**

Agregar un mecanismo estructurado para registrar problemas de referencia.

### **Posibles estados**

* `valid`: la referencia existe y coincide con una imagen del job  
* `missing`: no se devolvió referencia  
* `invalid`: se devolvió una referencia inexistente  
* `ambiguous`: reservado para futuro

---

## **7\. Contratos técnicos**

## **7.1 Contrato interno de imágenes para análisis**

Antes de llamar a la API, debe existir una colección estructurada como esta:

```json
{
  "images": [
    {
      "image_id": "img_001",
      "original_filename": "rack_a_01.jpg",
      "upload_order": 1
    },
    {
      "image_id": "img_002",
      "original_filename": "rack_a_02.jpg",
      "upload_order": 2
    }
  ]
}
```

No necesariamente se debe enviar exactamente así al proveedor, pero sí debe existir internamente en esta forma o equivalente.

---

## **7.2 Contrato esperado de salida del proveedor**

Cada resultado devuelto por la API debe incluir un identificador de imagen.

### **Ejemplo mínimo esperado**

```json
{
  "results": [
    {
      "label": "SKU_A",
      "count": 12,
      "source_image_id": "img_002"
    }
  ]
}
```

### **Ejemplo preparado para futuro**

```json
{
  "results": [
    {
      "label": "SKU_A",
      "count": 12,
      "source_image_ids": ["img_001", "img_002"],
      "primary_image_id": "img_002"
    }
  ]
}
```

---

## **8\. Cambios por capa**

## **8.1 Upload / Ingestion Layer**

### **Objetivo**

Asignar IDs únicos a las imágenes y persistir metadata básica antes del análisis.

### **Cambios requeridos**

* interceptar el lote de imágenes cargadas,  
* generar `image_id` por cada una,  
* asociarlas al `job_id`,  
* registrar orden de carga,  
* persistir referencia al archivo almacenado.

### **Recomendación**

Si v3.0 ya tiene una estructura de assets del job, extender esa capa en vez de crear una nueva paralela.

---

## **8.2 Prompt Builder / Request Builder**

### **Objetivo**

Hacer explícita la identidad de cada imagen en el contexto enviado a la API.

### **Cambios requeridos**

* extender el builder del prompt,  
* enumerar cada imagen con su `image_id`,  
* explicitar que toda respuesta debe indicar el `source_image_id` por resultado.

### **Lineamiento de prompt**

La instrucción debe ser inequívoca. No conviene dejarlo como sugerencia.

### **Ejemplo conceptual**

* “Each image in this batch has a unique identifier.”  
* “For every counted result, return the exact `source_image_id` of the image used as evidence.”  
* “Do not invent IDs.”  
* “Only use image IDs explicitly provided in the input.”

---

## **8.3 Provider Adapter**

### **Objetivo**

Ajustar la expectativa sobre la respuesta del proveedor.

### **Cambios requeridos**

* actualizar el schema o parser esperado,  
* tolerar respuestas sin referencia si el proveedor falla,  
* propagar warnings en vez de romper siempre el job.

### **Decisión recomendada**

La ausencia de `source_image_id` no debería tumbar toda la ejecución salvo que el negocio lo exija. Mejor marcar inconsistencia.

---

## **8.4 Parsing / Normalization Layer**

### **Objetivo**

Validar y normalizar la referencia de imagen devuelta por la API.

### **Reglas**

* si `source_image_id` está presente y existe en el job → `valid`  
* si falta → `missing`  
* si está presente pero no existe → `invalid`

### **Cambios requeridos**

* agregar set o índice rápido de `image_id`s válidos del job,  
* validar cada resultado contra ese conjunto,  
* anotar estado de trazabilidad,  
* normalizar el resultado para persistencia.

---

## **8.5 Domain Layer**

### **Objetivo**

Incorporar la trazabilidad por imagen al modelo de dominio.

### **Cambios requeridos**

* extender DTOs o entidades de resultado,  
* agregar campos de evidencia,  
* exponer API clara para leer la imagen fuente,  
* mantener compatibilidad con resultados heredados sin trazabilidad.

### **Recomendación**

No esconder la lógica de validación dentro del parser solamente. El dominio debe conocer el concepto de trazabilidad.

---

## **8.6 Persistence Layer**

### **Objetivo**

Guardar tanto las imágenes del job como la relación de cada resultado con su evidencia.

### **Opciones**

### **Opción A — Embebido en JSON de resultados**

Guardar `source_image_id` dentro del payload JSON del resultado.

**Ventajas**

* implementación rápida,  
* menor impacto de schema.

**Desventajas**

* peor consultabilidad SQL,  
* más débil para reporting.

### **Opción B — Columnas dedicadas**

Agregar campos concretos en la tabla de resultados.

Ejemplo:

* `source_image_id`  
* `traceability_status`  
* `traceability_warning`

**Ventajas**

* filtros y queries más simples,  
* mejor auditoría.

**Desventajas**

* requiere migración de schema.

### **Opción C — Tabla relacional de vínculo**

Crear tabla específica para evidence mapping.

Ejemplo:

* `result_evidence_links`  
  * `id`  
  * `job_id`  
  * `result_id`  
  * `image_id`  
  * `is_primary`  
  * `created_at`

**Ventajas**

* diseño más correcto para multi-evidencia,  
* escalable a futuro.

**Desventajas**

* más trabajo ahora.

### **Recomendación técnica**

Si buscás equilibrio entre velocidad y evolución:

* en **v3.1 inicial**, usar **columnas dedicadas o JSON enriquecido**,  
* pero diseñar el dominio de forma compatible con una futura tabla relacional.

Si ya estás trabajando con una base sólida en v3.0, lo más prolijo es:

* tabla de `job_images`,  
* `source_image_id` en la entidad/tabla principal de resultado,  
* y más adelante migrar a vínculo N:N si hiciera falta.

---

## **8.7 Output / Response Layer**

### **Objetivo**

Exponer la trazabilidad al consumidor del backend.

### **Cambios requeridos**

* extender response DTOs,  
* incluir `source_image_id`,  
* opcionalmente incluir `original_filename`,  
* exponer `traceability_status`.

### **Ejemplo de respuesta enriquecida**

```json
{
  "result_id": "res_001",
  "label": "SKU_A",
  "count": 12,
  "source_image_id": "img_002",
  "traceability_status": "valid"
}
```

---

## **9\. Diseño de persistencia recomendado**

## **9.1 Tabla `job_images`**

Nueva tabla para registrar imágenes cargadas.

### **Campos sugeridos**

* `id` o `image_id`  
* `job_id`  
* `original_filename`  
* `upload_order`  
* `storage_path`  
* `mime_type`  
* `checksum`  
* `width`  
* `height`  
* `created_at`

### **Índices sugeridos**

* índice por `job_id`  
* índice único por `(job_id, image_id)`

---

## **9.2 Extensión de tabla de resultados**

Agregar:

* `source_image_id` nullable  
* `traceability_status` nullable o con default  
* `traceability_warning` nullable

### **Clave foránea**

Si la base actual lo permite, `source_image_id` debería referenciar `job_images.image_id` en combinación lógica con `job_id`.

Si esto complica demasiado el schema actual, al menos mantener validación en aplicación.

---

## **10\. Validación y manejo de errores**

## **10.1 Casos a cubrir**

### **Caso 1 — Resultado válido**

`source_image_id` existe y pertenece al job.

Acción:

* persistir,  
* marcar `valid`.

### **Caso 2 — Resultado sin referencia**

No viene `source_image_id`.

Acción:

* persistir resultado,  
* marcar `missing`,  
* registrar warning.

### **Caso 3 — Referencia inválida**

Viene `source_image_id`, pero no existe en el job.

Acción:

* persistir resultado,  
* marcar `invalid`,  
* registrar warning detallado.

### **Caso 4 — Formato incorrecto**

Viene tipo incorrecto o estructura inconsistente.

Acción:

* intentar normalización si es trivial,  
* si no, marcar `invalid` y registrar warning.

---

## **11\. Cambios en exportaciones**

Aunque puede quedar en una segunda subetapa, la implementación debe dejar preparado que CSV/Excel/JSON incluyan al menos:

* `result_id`  
* `label`  
* `count`  
* `source_image_id`  
* `traceability_status`

Opcionalmente:

* `original_filename`

Esto es importante porque uno de los valores de negocio de v3.1 está en poder auditar resultados fuera del sistema.

---

## **12\. Cambios en revisión y UI**

A nivel backend, conviene dejar listo que el frontend pueda:

* mostrar la imagen fuente asociada a un resultado,  
* filtrar resultados con trazabilidad faltante o inválida,  
* ordenar por `traceability_status`,  
* navegar desde un resultado hacia su evidencia.

No hace falta implementar toda la UX ahora, pero los endpoints y DTOs deben permitirlo.

---

## **13\. Estrategia de implementación incremental**

## **Fase 1 — Metadata de imágenes**

Implementar registro de `job_images` y generación de `image_id`.

### **Entregable**

El job ya conoce todas sus imágenes con identidad propia.

---

## **Fase 2 — Prompt enriquecido**

Extender el prompt builder para enviar IDs y exigir `source_image_id`.

### **Entregable**

El proveedor recibe el batch con identidad explícita.

---

## **Fase 3 — Parsing y validación**

Extender parser y normalización para leer `source_image_id`, validarlo y marcar estado.

### **Entregable**

Los resultados internos ya incluyen trazabilidad validada.

---

## **Fase 4 — Persistencia**

Guardar metadata de imágenes y referencia en resultados.

### **Entregable**

La trazabilidad queda almacenada en DB o storage estructurado.

---

## **Fase 5 — Exposición en API / export**

Extender DTOs, endpoints y exportaciones.

### **Entregable**

La funcionalidad es visible y consumible.

---

## **14\. Criterios técnicos de aceptación**

La implementación se considera técnicamente correcta cuando:

1. cada imagen de un job tiene `image_id` único;  
2. el prompt o payload incorpora esos IDs;  
3. el parser acepta `source_image_id` por resultado;  
4. el sistema valida que ese ID exista en el job;  
5. el resultado persistido incluye referencia de imagen y estado de trazabilidad;  
6. las respuestas del backend exponen esta información;  
7. los resultados heredados sin trazabilidad siguen siendo compatibles;  
8. la ausencia de referencia no rompe toda la ejecución, salvo configuración explícita;  
9. el diseño permite evolucionar a múltiples imágenes por resultado sin refactor mayor.

## **15\. Testing strategy**

## **15.1 Unit tests**

### **Prompt builder**

* genera IDs correctamente,  
* incluye instrucciones de trazabilidad,  
* mantiene formato esperado.

### **Parser**

* procesa `source_image_id` válido,  
* maneja missing,  
* detecta invalid,  
* marca estado correcto.

### **Domain normalization**

* transforma respuestas en resultados persistibles,  
* mantiene compatibilidad con payloads viejos.

---

## **15.2 Integration tests**

* job con múltiples imágenes,  
* provider mock que devuelve referencias válidas,  
* provider mock que devuelve referencias faltantes,  
* provider mock que devuelve referencias inválidas,  
* persistencia completa de `job_images` \+ resultados.

---

## **15.3 Regression tests**

* ejecución heredada de v3.0 sin `source_image_id`,  
* exportación antigua no rota,  
* endpoints existentes siguen funcionando.

---

## **16\. Riesgos técnicos**

## **16.1 Inconsistencia del proveedor**

El proveedor puede devolver IDs incorrectos o incompletos aunque el prompt sea correcto.

### **Mitigación**

Validación local \+ warnings \+ métricas.

---

## **16.2 Acoplamiento excesivo al prompt**

Si la estructura depende solo de instrucciones libres, puede haber variabilidad.

### **Mitigación**

Usar schema lo más estructurado posible en el output esperado.

---

## **16.3 Persistencia insuficiente**

Guardar solo JSON puede dificultar reporting futuro.

### **Mitigación**

Elegir desde ahora una persistencia razonablemente consultable.

---

## **16.4 Escalado a multi-evidencia**

Una implementación demasiado rígida con un solo `source_image_id` puede trabar futuras mejoras.

### **Mitigación**

Diseñar dominio y DTOs con apertura a `source_image_ids`.

---

## **17\. Recomendación de diseño final**

Para una v3.1 sólida, sin sobrediseñar:

* crear entidad/tabla `job_images`,  
* generar `image_id` por imagen en el momento de ingestión,  
* enriquecer prompt con ese mapping,  
* exigir `source_image_id` en resultados,  
* validar en backend,  
* persistir `source_image_id` \+ `traceability_status`,  
* exponer esos campos en responses y exportaciones,  
* dejar preparado el dominio para `source_image_ids` y `primary_image_id` más adelante.

Esa combinación da un buen balance entre:

* rapidez de implementación,  
* orden arquitectónico,  
* auditabilidad,  
* y capacidad de evolución.

---

## **18\. Resumen técnico ejecutivo**

La v3.1 debe implementarse como una extensión transversal sobre v3.0 que agregue identidad a cada imagen del job y trazabilidad a cada resultado del análisis. Técnicamente, esto requiere incorporar una representación persistida de imágenes cargadas, enriquecer el prompt hacia la API con IDs explícitos, extender el parser para recibir y validar `source_image_id`, persistir esa relación junto con el resultado y exponerla en los contratos del backend. La implementación debe tolerar respuestas incompletas, registrar inconsistencias de trazabilidad y dejar preparada la arquitectura para una futura consolidación multi-evidencia.
