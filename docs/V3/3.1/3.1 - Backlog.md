# **Plan de Implementación — Dinamic Inventory v3.1**

## **Trazabilidad de Evidencia por Imagen**

**Versión:** 1.0  
**Fecha:** 11 de marzo de 2026  
**Base:** Extensión incremental sobre v3.0

---

## **1\. Objetivo del plan**

Implementar en v3.1 la capacidad de **registrar de qué imágenes la API hizo el conteo**, sin reescribir la base de v3.0 y manteniendo compatibilidad con la arquitectura actual.

El plan está organizado para avanzar de forma incremental, con entregables verificables en cada etapa, minimizando riesgo técnico y dejando preparada la base para mejoras futuras de consolidación avanzada, revisión y exportaciones enriquecidas.

---

## **2\. Resultado esperado al finalizar v3.1**

Al terminar esta etapa, el sistema debe poder:

* asignar un `image_id` único a cada imagen cargada en un job,  
* incluir esos IDs en el contexto enviado a la API,  
* recibir por cada resultado un `source_image_id`,  
* validar si esa referencia existe dentro del job,  
* persistir la relación resultado ↔ imagen,  
* exponer esa trazabilidad en responses y salidas estructuradas,  
* marcar inconsistencias cuando falte o sea inválida la referencia.

---

## **3\. Estrategia general**

La implementación se apoya en cinco principios:

### **3.1 Extensión sobre v3.0**

No rediseñar el flujo principal existente. Los cambios deben integrarse sobre las abstracciones y contratos ya creados.

### **3.2 Incrementalidad**

Cada etapa debe dejar una mejora concreta y testeable, evitando grandes cambios indivisibles.

### **3.3 Compatibilidad**

El sistema debe seguir funcionando aunque un resultado no traiga trazabilidad.

### **3.4 Validación local fuerte**

La API puede fallar o devolver referencias erróneas; el backend debe validarlas y clasificar su estado.

### **3.5 Preparación para evolución**

Aunque la primera implementación use `source_image_id`, el diseño debe dejar abierta una futura expansión a múltiples imágenes soporte.

---

## **4\. Roadmap propuesto**

Se propone dividir la v3.1 en **6 épicas técnicas**:

1. **Épica 1 — Registro e identidad de imágenes**  
2. **Épica 2 — Prompt y contrato de análisis enriquecido**  
3. **Épica 3 — Parsing, validación y normalización de trazabilidad**  
4. **Épica 4 — Persistencia de evidencia y resultados**  
5. **Épica 5 — Exposición en API, revisión y exportaciones**  
6. **Épica 6 — Testing, métricas y hardening**

---

# **ÉPICA 1 — Registro e identidad de imágenes**

## **Objetivo**

Incorporar una representación técnica explícita de cada imagen cargada en un job.

## **Problema que resuelve**

Hoy las imágenes pueden existir como archivos o inputs del job, pero no necesariamente como entidades identificadas y persistidas con una identidad funcional propia.

## **Entregable**

Cada imagen del job queda registrada con un `image_id` único y metadata básica.

## **Historias técnicas**

### **US-1.1 — Generación de `image_id`**

Como sistema, necesito generar un identificador único por imagen dentro del job para poder referenciarla durante el análisis y en los resultados.

**Tareas**

* definir convención de `image_id`,  
* implementar generador estable por lote,  
* asegurar unicidad dentro del job.

### **US-1.2 — Registro de metadata básica**

Como sistema, necesito almacenar metadata mínima de cada imagen para poder rastrearla luego.

**Tareas**

* registrar `job_id`,  
* registrar `original_filename`,  
* registrar `upload_order`,  
* registrar `storage_path`,  
* registrar `created_at`,  
* opcionalmente checksum, mime type, dimensiones.

### **US-1.3 — Integración con flujo de ingestión existente**

Como backend, necesito integrar el registro de imágenes al flujo actual sin romper la lógica de carga existente.

**Tareas**

* ubicar el punto de entrada exacto donde hoy se reciben/guardan imágenes,  
* insertar la lógica de creación de `ImageEvidence`,  
* mantener compatibilidad con el pipeline actual.

## **Criterios de aceptación**

* todo job con múltiples imágenes registra un `image_id` por imagen,  
* los IDs son únicos dentro del job,  
* la metadata básica queda disponible para capas posteriores,  
* el flujo actual de carga sigue funcionando.

---

# **ÉPICA 2 — Prompt y contrato de análisis enriquecido**

## **Objetivo**

Hacer explícita la identidad de las imágenes en el llamado a la API y exigir trazabilidad en la respuesta.

## **Problema que resuelve**

La API hoy analiza un conjunto de imágenes, pero no necesariamente devuelve desde cuál hizo cada conteo.

## **Entregable**

El prompt/payload enviado al proveedor incluye IDs de imágenes y una instrucción explícita para devolver `source_image_id`.

## **Historias técnicas**

### **US-2.1 — Enriquecimiento del prompt**

Como sistema, necesito incluir la lista de imágenes identificadas en el contexto de análisis para que la API pueda referenciarlas correctamente.

**Tareas**

* extender el prompt builder,  
* inyectar listado de imágenes con `image_id`,  
* definir formato claro y consistente,  
* evitar ambigüedad entre nombre de archivo e identificador interno.

### **US-2.2 — Instrucción obligatoria de trazabilidad**

Como sistema, necesito que el prompt requiera explícitamente `source_image_id` por cada resultado para poder mapear evidencia con hallazgos.

**Tareas**

* agregar instrucción firme, no sugerida,  
* aclarar que no se inventen IDs,  
* restringir referencias a IDs del input.

### **US-2.3 — Ajuste del contrato esperado del proveedor**

Como backend, necesito extender el contrato esperado de respuesta para contemplar `source_image_id`.

**Tareas**

* actualizar schema de salida esperado,  
* revisar adaptadores del proveedor,  
* definir comportamiento ante ausencia del campo.

## **Criterios de aceptación**

* el prompt o payload contiene IDs de imágenes,  
* la instrucción de trazabilidad queda explícita,  
* el proveedor mock o adaptador puede devolver `source_image_id`,  
* no se rompe el flujo normal de análisis.

---

# **ÉPICA 3 — Parsing, validación y normalización de trazabilidad**

## **Objetivo**

Procesar técnicamente la referencia de imagen devuelta por la API y clasificar su validez.

## **Problema que resuelve**

Aunque el proveedor devuelva `source_image_id`, el sistema necesita verificar si realmente corresponde a una imagen del job.

## **Entregable**

Cada resultado procesado queda normalizado con un estado de trazabilidad.

## **Historias técnicas**

### **US-3.1 — Parseo de `source_image_id`**

Como parser, necesito leer el campo `source_image_id` de cada resultado del proveedor para incorporarlo al modelo interno.

**Tareas**

* extender parser de resultados,  
* soportar respuestas enriquecidas,  
* mantener compatibilidad con respuestas heredadas.

### **US-3.2 — Validación contra imágenes del job**

Como backend, necesito validar que cada `source_image_id` exista en el conjunto de imágenes del job para evitar referencias inválidas.

**Tareas**

* construir lookup rápido de `image_id`s válidos,  
* validar cada referencia,  
* clasificar estado.

### **US-3.3 — Normalización de estado de trazabilidad**

Como dominio, necesito representar de forma consistente la calidad de la referencia devuelta.

**Estados sugeridos**

* `valid`  
* `missing`  
* `invalid`

**Tareas**

* definir enum o constante,  
* asociar warnings,  
* agregar mensaje de inconsistencia cuando aplique.

### **US-3.4 — Compatibilidad con resultados legacy**

Como sistema, necesito seguir procesando resultados sin trazabilidad para no romper flujos existentes.

**Tareas**

* permitir `source_image_id` opcional,  
* marcar `missing` sin fallar,  
* conservar compatibilidad con v3.0.

## **Criterios de aceptación**

* un resultado con `source_image_id` válido queda marcado como `valid`,  
* un resultado sin referencia queda marcado como `missing`,  
* una referencia inexistente queda marcada como `invalid`,  
* el pipeline no falla por ausencia de referencia salvo regla explícita futura.

---

# **ÉPICA 4 — Persistencia de evidencia y resultados**

## **Objetivo**

Guardar de forma estructurada tanto las imágenes del job como la relación entre cada resultado y su evidencia fuente.

## **Problema que resuelve**

Sin persistencia, la trazabilidad solo existe durante la ejecución y no sirve para auditoría, revisión ni exportación.

## **Entregable**

La base de datos o el storage estructurado guardan `job_images` y la referencia `source_image_id` por resultado.

## **Historias técnicas**

### **US-4.1 — Persistencia de imágenes del job**

Como sistema, necesito guardar cada imagen del job como entidad consultable.

**Tareas**

* definir tabla o estructura `job_images`,  
* crear migración si aplica,  
* guardar metadata mínima,  
* indexar por `job_id`.

### **US-4.2 — Persistencia de referencia por resultado**

Como sistema, necesito persistir `source_image_id` y el estado de trazabilidad en los resultados del job.

**Tareas**

* definir cambio de schema en tabla de resultados o JSON persistido,  
* guardar `source_image_id`,  
* guardar `traceability_status`,  
* guardar warning si corresponde.

### **US-4.3 — Migración segura**

Como equipo técnico, necesitamos introducir cambios de persistencia sin romper jobs ya existentes.

**Tareas**

* definir campos nullable,  
* asegurar compatibilidad hacia atrás,  
* validar lectura de registros legacy.

## **Criterios de aceptación**

* las imágenes del job quedan persistidas,  
* cada resultado guarda su estado de trazabilidad,  
* los datos heredados siguen siendo legibles,  
* no se rompe el acceso a jobs anteriores.

---

# **ÉPICA 5 — Exposición en API, revisión y exportaciones**

## **Objetivo**

Volver visible y consumible la trazabilidad por imagen para frontend, revisión operativa y exportación.

## **Problema que resuelve**

Persistir el dato no alcanza; debe poder consultarse y usarse.

## **Entregable**

Los DTOs y salidas del sistema exponen `source_image_id` y `traceability_status`.

## **Historias técnicas**

### **US-5.1 — Extensión de DTOs y responses**

Como consumidor del backend, necesito recibir en cada resultado la información de trazabilidad para poder mostrarla o analizarla.

**Tareas**

* extender DTOs,  
* incluir `source_image_id`,  
* incluir `traceability_status`,  
* opcionalmente incluir `original_filename`.

### **US-5.2 — Exposición para revisión interna**

Como sistema de revisión, necesito poder filtrar o identificar resultados con trazabilidad faltante o inválida.

**Tareas**

* exponer flags de inconsistencia,  
* preparar respuesta para filtros de revisión,  
* facilitar navegación resultado → imagen.

### **US-5.3 — Exportaciones enriquecidas**

Como usuario de negocio o auditoría, necesito exportar resultados con la imagen fuente asociada.

**Tareas**

* agregar columnas a export CSV/Excel/JSON,  
* incluir al menos `source_image_id` y `traceability_status`,  
* opcionalmente incluir nombre de archivo original.

## **Criterios de aceptación**

* las respuestas API exponen trazabilidad,  
* los resultados pueden identificarse como válidos, faltantes o inválidos,  
* las exportaciones contienen la nueva información,  
* la funcionalidad es visible sin romper contratos críticos.

---

# **ÉPICA 6 — Testing, métricas y hardening**

## **Objetivo**

Cerrar la v3.1 con cobertura suficiente, métricas útiles y comportamiento robusto.

## **Problema que resuelve**

Sin validación fuerte, la nueva trazabilidad puede quedar inconsistente o degradarse silenciosamente.

## **Entregable**

La funcionalidad queda testeada, monitoreable y lista para iterar.

## **Historias técnicas**

### **US-6.1 — Unit tests**

Como equipo técnico, necesitamos validar de forma aislada generación de IDs, prompt, parser y normalización.

**Tareas**

* tests de `image_id`,  
* tests de prompt enriquecido,  
* tests de parser con caso válido,  
* tests de parser con missing,  
* tests de parser con invalid.

### **US-6.2 — Integration tests**

Como equipo técnico, necesitamos validar el flujo end-to-end dentro del backend.

**Tareas**

* job con múltiples imágenes,  
* provider mock con referencias válidas,  
* provider mock con referencias faltantes,  
* provider mock con referencias inválidas,  
* verificación de persistencia final.

### **US-6.3 — Regression tests**

Como equipo técnico, necesitamos asegurar que v3.0 no se rompa por la nueva capacidad.

**Tareas**

* testear jobs antiguos,  
* testear respuestas sin trazabilidad,  
* testear exportaciones previas.

### **US-6.4 — Métricas operativas de trazabilidad**

Como producto/ingeniería, necesitamos medir la calidad de esta nueva capacidad.

**Métricas sugeridas**

* porcentaje de resultados con `traceability_status = valid`,  
* porcentaje `missing`,  
* porcentaje `invalid`,  
* cantidad de resultados por imagen,  
* distribución de uso de evidencia.

## **Criterios de aceptación**

* existe cobertura de tests para casos principales,  
* el sistema sigue siendo compatible con flujos legacy,  
* las métricas permiten monitorear adopción y calidad.

---

## **5\. Orden recomendado de ejecución**

El orden ideal para implementar es:

### **Paso 1**

Épica 1 — Registro e identidad de imágenes

### **Paso 2**

Épica 2 — Prompt y contrato enriquecido

### **Paso 3**

Épica 3 — Parsing y validación

### **Paso 4**

Épica 4 — Persistencia

### **Paso 5**

Épica 5 — Exposición y exportaciones

### **Paso 6**

Épica 6 — Testing y hardening

Este orden minimiza retrabajo porque primero se define la identidad de imágenes, luego se usa en el análisis, luego se procesa, luego se guarda y finalmente se expone.

---

## **6\. Dependencias entre épicas**

* **Épica 2** depende de **Épica 1**  
* **Épica 3** depende de **Épica 1** y **Épica 2**  
* **Épica 4** depende de **Épica 3**  
* **Épica 5** depende de **Épica 4**  
* **Épica 6** atraviesa todas, pero conviene consolidarla al cierre

---

## **7\. Riesgos y mitigaciones**

### **Riesgo 1 — La API no devuelve consistentemente `source_image_id`**

**Mitigación:** no hacer fallar el job; registrar `missing` y warning estructurado.

### **Riesgo 2 — IDs inválidos devueltos por el proveedor**

**Mitigación:** validar localmente contra las imágenes del job y marcar `invalid`.

### **Riesgo 3 — Sobrediseño temprano para multi-evidencia**

**Mitigación:** implementar un modelo inicial simple con apertura a evolución futura.

### **Riesgo 4 — Impacto de schema en base de datos**

**Mitigación:** usar campos nullable, migraciones compatibles y pruebas sobre registros legacy.

### **Riesgo 5 — Exposición incompleta en frontend**

**Mitigación:** asegurar primero backend y contratos; la visualización puede escalar después.

---

## **8\. Definition of Done de v3.1**

La v3.1 puede considerarse terminada cuando:

* existe una entidad o representación persistida de las imágenes del job,  
* cada imagen tiene `image_id` único,  
* el prompt o payload incorpora dichos IDs,  
* la respuesta del proveedor puede mapear cada resultado a un `source_image_id`,  
* el backend valida y clasifica esa referencia,  
* la persistencia guarda la trazabilidad,  
* la API la expone,  
* las exportaciones pueden incluirla,  
* hay tests de unidad, integración y regresión,  
* y el sistema sigue siendo compatible con resultados de v3.0.

---

## **9\. Recomendación de implementación práctica**

Para avanzar con buena relación entre velocidad y calidad:

### **Primera iteración recomendable**

* implementar `job_images`,  
* agregar `source_image_id`,  
* agregar `traceability_status`,  
* enriquecer prompt,  
* validar referencias,  
* exponer el dato en responses.

### **Segunda iteración opcional**

* agregar `original_filename` en responses,  
* enriquecer exportaciones,  
* agregar métricas,  
* preparar `source_image_ids` y `primary_image_id`.

---

## **10\. Resumen ejecutivo del plan**

La implementación de v3.1 debe ejecutarse como una extensión incremental en seis épicas: primero se da identidad a cada imagen del job, luego esa identidad se incorpora al análisis, después se valida la respuesta del proveedor, se persiste la relación entre resultados e imágenes, se expone esa trazabilidad al sistema y finalmente se endurece con pruebas y métricas. Esto permite resolver el problema principal de negocio —saber desde qué imágenes la API hizo el conteo— sin reescribir la base establecida en v3.0.
