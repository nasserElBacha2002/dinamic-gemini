# **Documento Funcional — Dinamic Inventory v3.1**

## **Trazabilidad de Evidencia por Imagen**

**Versión:** 1.0  
**Estado:** Propuesta funcional  
**Base:** Extensión sobre Dinamic Inventory v3.0  
**Fecha:** 11 de marzo de 2026

---

## **1\. Resumen ejecutivo**

La versión **3.1** de Dinamic Inventory tiene como objetivo incorporar **trazabilidad explícita entre las imágenes cargadas por el usuario y los resultados generados por la API de análisis**.

En la versión 3.0, el sistema ya permite procesar múltiples fotos y obtener resultados consolidados. Sin embargo, el resultado final no siempre deja asentado **de qué imagen proviene cada conteo, hallazgo o evidencia utilizada por la API**. Esto limita la capacidad de revisión, auditoría, explicación operativa y mejora continua del sistema.

La versión 3.1 resuelve este problema mediante un flujo en el que:

* cada imagen cargada es identificada antes del análisis,  
* esa identificación se incorpora al prompt enviado a la API,  
* y la respuesta debe incluir, para cada resultado contado, una referencia explícita a la imagen fuente utilizada.

Esto convierte a la evidencia visual en un elemento trazable y auditable dentro del pipeline, sin reescribir la arquitectura base de v3.0.

---

## **2\. Objetivo de negocio**

Permitir que el sistema pueda responder con claridad y consistencia:

**“¿Desde qué imagen o imágenes se realizó este conteo?”**

Esto busca mejorar:

* la auditabilidad del sistema,  
* la confiabilidad de la revisión humana,  
* la explicación de los resultados,  
* la identificación de errores,  
* y la calidad de la consolidación futura.

---

## **3\. Objetivo funcional**

Agregar a la plataforma la capacidad de:

1. **identificar unívocamente cada imagen cargada**,  
2. **enviar esa identificación como parte del contexto de análisis**,  
3. **exigir a la API que devuelva el identificador de imagen por cada resultado contado**,  
4. **persistir esa relación en los resultados del job**,  
5. **exponer esa trazabilidad para revisión, exportación y análisis posterior**.

---

## **4\. Principio rector**

La versión 3.1 debe **extender v3.0 sin reescribir sus bases**.

Toda mejora debe apoyarse en las abstracciones ya creadas en v3.0 para incorporar nuevas capacidades de trazabilidad, revisión y consolidación, evitando acoplamientos excesivos o rediseños innecesarios del núcleo ya estable.

---

## **5\. Problema actual**

En el estado actual del producto:

* el usuario puede cargar múltiples imágenes en una misma operación,  
* la API puede devolver resultados correctos o parcialmente correctos,  
* pero no siempre queda registrada la relación exacta entre cada resultado y la imagen que lo originó.

Esto genera varias limitaciones:

* dificultad para revisar un conteo puntual,  
* imposibilidad de saber qué imagen respaldó una respuesta,  
* menor capacidad para detectar errores de evidencia,  
* poca transparencia para debugging funcional,  
* escasa base para métricas avanzadas de calidad.

---

## **6\. Alcance funcional de v3.1**

La v3.1 se enfoca en la **trazabilidad de evidencia por imagen**.

### **Incluye**

* identificación previa de cada imagen cargada,  
* incorporación de esa identificación al contexto del análisis,  
* respuesta enriquecida con referencia a imagen fuente,  
* persistencia de la relación resultado ↔ imagen,  
* exposición de esta información en los resultados del job,  
* base funcional para revisión avanzada y exportaciones futuras.

### **Puede incluir en esta etapa o quedar preparada**

* consolidación avanzada de evidencias,  
* selección de evidencia principal,  
* score de calidad de evidencia,  
* filtros avanzados de revisión,  
* mejoras de asociación etiqueta/producto,  
* snapshots pre y post revisión,  
* métricas por error y por pasillo,  
* exportaciones CSV, Excel y JSON enriquecidas.

### **No incluye como objetivo principal**

* rediseño completo del pipeline,  
* cambio de proveedor de API,  
* reentrenamiento de modelos,  
* rediseño integral del frontend,  
* reestructuración profunda de la persistencia existente.

---

## **7\. Definición conceptual de la etapa**

La versión 3.1 introduce una nueva capa funcional:

### **Evidence-to-Result Mapping**

Cada resultado devuelto por la API debe poder vincularse explícitamente con la imagen que lo sustenta.

El sistema deja de tratar el lote de imágenes como un conjunto anónimo y empieza a manejarlas como una colección de evidencias identificadas, donde cada una tiene identidad propia dentro del análisis.

---

## **8\. Flujo funcional esperado**

### **8.1 Carga de imágenes**

El usuario carga múltiples imágenes en la plataforma como parte de un job de inventario o revisión.

### **8.2 Identificación de imágenes**

Antes de realizar el llamado a la API, el sistema asigna a cada imagen un identificador único dentro del contexto del job.

Ese identificador debe existir independientemente del nombre original del archivo, para evitar ambigüedades.

### **8.3 Preparación de contexto**

Al construir el prompt o payload de análisis, el sistema informa explícitamente a la API qué imágenes forman parte del análisis y con qué identificador debe referenciarlas.

### **8.4 Análisis por API**

La API procesa el lote de imágenes según el prompt enriquecido.

### **8.5 Respuesta enriquecida**

Por cada resultado contado, hallazgo o agrupación relevante, la respuesta debe incluir el identificador de imagen asociado.

En escenarios más avanzados, un resultado podría referenciar más de una imagen.

### **8.6 Persistencia**

El sistema guarda esa relación como parte del resultado procesado del job.

### **8.7 Revisión**

La plataforma puede mostrar o utilizar esa trazabilidad para revisión manual, debugging, filtros, snapshots y exportaciones.

---

## **9\. Requerimientos funcionales**

### **RF-01 — Identificación única de imágenes**

El sistema debe asignar un identificador único a cada imagen cargada dentro de un job.

### **RF-02 — Persistencia de metadata de imagen**

El sistema debe registrar, al menos, la relación entre:

* job,  
* imagen,  
* identificador interno,  
* nombre original del archivo,  
* orden o posición dentro del lote.

### **RF-03 — Inclusión de IDs en el análisis**

El sistema debe incorporar el identificador de cada imagen dentro del prompt o payload enviado a la API.

### **RF-04 — Instrucción explícita al proveedor**

El prompt debe requerir de forma explícita que cada resultado contado incluya la referencia a la imagen fuente.

### **RF-05 — Respuesta con trazabilidad**

El sistema debe poder recibir, validar y procesar respuestas donde cada resultado incluya un campo de referencia de imagen.

### **RF-06 — Almacenamiento de relación resultado ↔ imagen**

La relación entre resultado y evidencia debe quedar persistida como parte del resultado estructurado del job.

### **RF-07 — Exposición de trazabilidad**

La trazabilidad por imagen debe estar disponible para consumo interno del sistema y para revisión posterior.

### **RF-08 — Compatibilidad con consolidación futura**

La estructura funcional debe permitir evolucionar hacia resultados respaldados por múltiples imágenes.

### **RF-09 — Manejo de ausencia de referencia**

Si la API no devuelve referencia de imagen para un resultado, el sistema debe registrar esa situación como inconsistencia o dato incompleto.

### **RF-10 — Base para exportaciones**

La información de trazabilidad debe quedar estructurada de forma que pueda exportarse en CSV, Excel o JSON en etapas posteriores.

---

## **10\. Reglas funcionales**

### **RN-01**

Cada imagen debe tener un ID único dentro del job. No puede haber dos imágenes con el mismo identificador en la misma ejecución.

### **RN-02**

El identificador de imagen no debe depender exclusivamente del nombre original del archivo.

### **RN-03**

Todo resultado contado debe intentar asociarse al menos a una imagen fuente.

### **RN-04**

Si una respuesta de API devuelve un ID inexistente, el sistema debe marcarlo como inválido.

### **RN-05**

Si la API devuelve resultados sin referencia de imagen, el job no debe necesariamente fallar, pero sí debe registrar el incidente para revisión.

### **RN-06**

La trazabilidad de evidencia debe considerarse parte del resultado funcional, no un dato accesorio.

### **RN-07**

La solución debe ser compatible con futuros escenarios donde:

* un resultado tenga múltiples imágenes soporte,  
* una imagen respalde múltiples resultados,  
* exista una imagen principal y otras secundarias.

---

## **11\. Entidades funcionales involucradas**

### **11.1 Job**

Representa la ejecución general de análisis.

### **11.2 Image Evidence**

Representa cada imagen cargada como una evidencia identificable dentro del job.

A nivel funcional, debería contemplar al menos:

* `image_id`  
* `job_id`  
* `original_filename`  
* `upload_order`  
* `storage_reference`  
* `created_at`

### **11.3 Count Result / Finding**

Representa cada resultado devuelto por la API.

A nivel funcional, debería poder incluir:

* identificador del resultado,  
* producto / etiqueta / clase detectada,  
* cantidad,  
* referencia a imagen fuente,  
* opcionalmente referencias múltiples,  
* flags de revisión o consistencia.

---

## **12\. Estructura funcional esperada del resultado**

A nivel conceptual, cada resultado debería poder verse así:

```json
{
  "result_id": "res_001",
  "product_label": "SKU_A",
  "count": 12,
  "source_image_id": "img_003"
}
```

Y, en una evolución más robusta:

```json
{
  "result_id": "res_001",
  "product_label": "SKU_A",
  "count": 12,
  "source_image_ids": ["img_003", "img_004"],
  "primary_image_id": "img_003"
}
```

Esto no obliga todavía a una implementación exacta, pero sí define la dirección funcional correcta.

---

## **13\. Casos de uso principales**

### **CU-01 — Revisar de qué imagen salió un conteo**

Como usuario revisor, quiero ver qué imagen respaldó un resultado, para poder validar si el conteo fue correcto.

### **CU-02 — Auditar una inconsistencia**

Como usuario operativo o técnico, quiero rastrear un resultado hasta la imagen fuente, para investigar errores o comportamientos dudosos.

### **CU-03 — Comparar calidad de evidencias**

Como analista del sistema, quiero conocer qué imágenes fueron usadas como respaldo, para entender si el proveedor está apoyándose en evidencia clara o deficiente.

### **CU-04 — Exportar resultados trazables**

Como usuario de negocio, quiero descargar resultados con referencia a sus imágenes fuente, para compartir evidencia con terceros o generar auditoría.

### **CU-05 — Mejorar prompts y revisión**

Como equipo de producto, queremos registrar esta trazabilidad para mejorar futuras iteraciones del prompt, consolidación y criterios de revisión.

## **14\. Escenarios funcionales**

### **Escenario A — Conteo simple con una sola imagen fuente**

El usuario carga cinco imágenes. La API devuelve tres resultados. Cada resultado indica un único `source_image_id`.

Resultado esperado: el sistema puede mostrar y guardar de qué imagen provino cada conteo.

### **Escenario B — Resultado sin referencia**

La API devuelve un resultado válido en cantidad, pero sin `source_image_id`.

Resultado esperado: el sistema conserva el resultado, pero marca la trazabilidad como incompleta.

### **Escenario C — Referencia inválida**

La API devuelve `source_image_id = img_999`, pero esa imagen no existe dentro del job.

Resultado esperado: el sistema marca el resultado como inconsistente o inválido a nivel de trazabilidad.

### **Escenario D — Soporte múltiple**

En una futura ampliación, la API utiliza más de una imagen para respaldar una decisión.

Resultado esperado: la estructura funcional ya puede soportar múltiples referencias sin rediseño.

---

## **15\. Impacto esperado en revisión operativa**

La incorporación de trazabilidad por imagen permitirá:

* revisar conteos de manera más rápida,  
* entender por qué la API llegó a un resultado,  
* detectar imágenes débiles o confusas,  
* priorizar mejoras en la captura,  
* construir métricas de calidad por imagen,  
* y preparar mecanismos más avanzados de consolidación.

---

## **16\. Dependencias funcionales**

La etapa depende de que v3.0 ya cuente con:

* flujo estable de carga de múltiples imágenes,  
* construcción de prompt o payload hacia la API,  
* procesamiento estructurado de respuesta,  
* persistencia mínima de jobs y resultados,  
* arquitectura extensible que permita enriquecer metadata sin romper contratos principales.

---

## **17\. Riesgos funcionales**

### **17.1 Respuesta incompleta del proveedor**

La API podría no devolver consistentemente el identificador solicitado.

### **17.2 Ambigüedad en resultados consolidados**

Un resultado consolidado puede haber surgido por comparación entre varias imágenes y no por una sola.

### **17.3 Sobredependencia del prompt**

Si el proveedor depende únicamente del prompt para devolver esta estructura, puede haber variaciones en consistencia.

### **17.4 Revisión parcial en frontend**

Puede implementarse la trazabilidad en backend, pero no verse todavía reflejada en revisión visual si el frontend no la expone.

---

## **18\. Criterios de aceptación funcional**

La etapa 3.1 se considera funcionalmente cumplida cuando:

1. un job con múltiples imágenes registra un identificador único por cada imagen;  
2. esos identificadores se incorporan al análisis enviado a la API;  
3. la respuesta puede incluir referencia a imagen por cada resultado;  
4. el sistema guarda esa relación de manera estructurada;  
5. puede conocerse, para un resultado dado, cuál fue su imagen fuente;  
6. los casos sin referencia o con referencia inválida quedan registrados como inconsistencias de trazabilidad;  
7. la solución no rompe el comportamiento base de v3.0.

---

## **19\. Métricas sugeridas para esta etapa**

Aunque no todas deben implementarse ya, esta etapa habilita medir:

* porcentaje de resultados con trazabilidad válida,  
* porcentaje de resultados sin `source_image_id`,  
* porcentaje de referencias inválidas,  
* cantidad de resultados por imagen,  
* distribución de uso de imágenes dentro de un job,  
* imágenes más utilizadas como evidencia principal,  
* tasa de revisión manual por imagen,  
* errores por pasillo o lote vinculados a evidencia.

---

## **20\. Evolución natural posterior**

Una vez implementada esta base funcional, la v3.1 puede expandirse hacia:

* selección automática de evidencia principal,  
* score de calidad por imagen,  
* consolidación multi-evidencia,  
* filtros de revisión avanzados,  
* snapshots pre y post revisión,  
* exportaciones enriquecidas,  
* explicación visual de resultados,  
* ranking de imágenes por valor probatorio.

---

## **21\. Definición final de la etapa**

La versión 3.1 de Dinamic Inventory incorpora una capacidad nueva y clave para la madurez operativa del producto: **la trazabilidad de evidencia por imagen**. A partir de esta etapa, cada imagen deja de ser solo una entrada visual y pasa a ser una evidencia identificada, referenciable y auditable dentro del proceso de conteo. Esto fortalece la revisión humana, la explicación del sistema, la depuración de errores y la evolución futura hacia consolidaciones más inteligentes.

---

## **22\. Resumen corto para alineación interna**

**v3.1 no cambia la base de v3.0; la extiende.**  
Su objetivo principal es registrar de qué imágenes la API hizo el conteo. Para eso, cada imagen se identifica antes del análisis, esa identidad se envía en el prompt y cada resultado debe volver asociado a una imagen fuente. Esto mejora trazabilidad, revisión, auditoría y calidad de evidencia..
