# V3.0 — Notas de alineación de documentación

Este documento registra las correcciones aplicadas durante la alineación de la documentación v3.0 y las decisiones abiertas que siguen pendientes de confirmación.

---

## 1. Resumen de correcciones aplicadas

* **Documento técnico 3.0:** Se añadieron transiciones de estado formales para Inventory, Aisle y Position (§7.1, 7.2, 7.4). Se definió la **definición canónica de Position** (unidad revisable = pallet lógico detectado) y el significado de los estados reviewed/corrected/deleted. Se añadieron las secciones **§9.4 Contrato de salida del pipeline para el mapeador**, **§9.5 Modelo de error operativo**, **§9.6 Métricas canónicas** y **§9.7 Paginación y filtros** para `GET /aisles/{aisleId}/positions`. Se actualizó el stack de frontend (§13) a React, TypeScript, MUI, React Router, TanStack Query y **MUI Data Grid** como estándar para tablas operativas; se eliminaron "TanStack Table" y "biblioteca UI liviana". Se añadió la ubicación del frontend (`frontend/` en la raíz) en §5.2 y §13. La Fase 8 de métricas ahora referencia §9.6.

* **V3.0.md:** Se alineó el stack de la Fase 6 con MUI y MUI Data Grid; se eliminó "TanStack Table o grilla similar". Se reforzó la definición de Position (§4.3) y se añadieron referencias al Documento técnico para transiciones, métricas (§9.6), paginación/filtros (§9.7) y frontend (§13). La sección Frontend (§8) y la Fase 8 de métricas referencian el Documento técnico.

* **V3.0 - Backlog.md:** La estructura del proyecto ya no incluye `interfaces/web/`; el frontend se documenta como app en **`frontend/`** en la raíz, con referencia a FRONTEND_ESTRUCTURA.md. Los ejemplos de UI se pasaron a MUI: EvidenceGallery (Grid, Box), MetricsCards (Grid, Card, CardContent), PositionsTable (DataGrid), ProductsTable (TableContainer, Table, TableHead, TableBody, TableRow, TableCell), PositionDetailPage (Stack), AisleStatusBadge (Chip). Se añadieron referencias al Documento técnico para el contrato del mapper (§9.4), métricas (§9.6), paginación/filtros (§9.7) y se indicó que la tabla de posiciones debe usar **MUI Data Grid**.

* **FRONTEND_ESTRUCTURA.md:** Se eliminó "(o TanStack Table)" como alternativa; MUI Data Grid queda como estándar para tablas operativas complejas. Se reforzó que la ubicación canónica del frontend es **`frontend/`** en la raíz. Se añadió la fila "Tablas complejas" en la tabla resumen.

* **V3.0 - FrontEnd - Regla de estilos.md:** Se añadió la regla explícita de no usar TanStack Table como sustituto de MUI Data Grid en vistas operativas (posiciones, inventarios, pasillos, productos, historial de revisión).

## 2. Inconsistencias resueltas

| Inconsistencia | Resolución |
|----------------|------------|
| TanStack Table vs MUI Data Grid | MUI Data Grid es el estándar oficial para tablas operativas complejas en todos los documentos. |
| "UI liviana" / "componentes propios simples" en Documento técnico | Sustituido por Material UI y MUI Data Grid. |
| Ubicación del frontend (`interfaces/web/` en Backlog) | Eliminado; frontend solo en **`frontend/`** en la raíz; Backlog referencia FRONTEND_ESTRUCTURA.md. |
| Ejemplos con Tailwind / `className` (grid-cols-3, etc.) en Backlog | Sustituidos por componentes MUI (Grid, Box, Card, Table, DataGrid, Chip, Stack). |
| Position ambiguo (posición vs pallet vs slot físico) | Definición canónica en Documento técnico §7.4: unidad revisable = pallet lógico detectado; no slot físico. |
| Transiciones de estado no documentadas | Añadidas en Documento técnico §7.1, 7.2, 7.4 para Inventory, Aisle y Position. |
| Significado de reviewed / corrected / deleted | Aclarado en Documento técnico §7.4 (reviewed = sin cambios, corrected = con cambio de datos, deleted = eliminación lógica). |
| Contrato pipeline → dominio | Nueva sección §9.4 con estructura mínima de positions[], products[], evidencias y campos obligatorios/opcionales. |
| Métricas sin definición canónica | Nueva sección §9.6 con total_reviewed_positions, auto_accepted_positions, success_rate, correction_rate, deletion_rate. |
| Modelo de error operativo | Nueva sección §9.5 con error_code, error_message, retryable y reflejo en Aisle. |
| Paginación y filtros de GET positions | Nueva sección §9.7 con page, page_size, status, needs_review, min_confidence, sku. |

## 3. Decisiones abiertas que requieren confirmación

Las siguientes decisiones se dejaron con **recomendación** en la documentación; si producto o arquitectura confirman otra opción, debe actualizarse el Documento técnico y, si aplica, este archivo.

1. **Cuándo Inventory pasa a `processing`**  
   Recomendación documentada: cuando **al menos un** pasillo del inventario pasa a `queued` o `processing`. ¿Confirmado o debe ser "todos los pasillos encolados"?

2. **Cierre de inventario**  
   Recomendación documentada: paso a `completed` por **acción explícita** de "Cerrar inventario" en la UI (no automático cuando todos los pasillos están completed). ¿Confirmado?

3. **Paginación por defecto**  
   Recomendación documentada: `page=1`, `page_size=25` (o 50) para `GET /aisles/{aisleId}/positions`. ¿Valores definitivos?

4. **Lista cerrada de `error_code`**  
   En §9.5 se documentan ejemplos propuestos (ASSET_UPLOAD_FAILED, PIPELINE_FAILED, TIMEOUT, VALIDATION_ERROR). ¿Se adopta una lista cerrada oficial o queda abierto a códigos por implementación?

Cuando se confirmen, conviene actualizar el Documento técnico 3.0 en las secciones correspondientes (§7 para transiciones, §9.7 para paginación, §9.5 para error_code) y anotar aquí la decisión final.
