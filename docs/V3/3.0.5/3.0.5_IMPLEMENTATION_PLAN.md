# v3.0.5 Implementation Plan — Technical Hardening and UI Foundation

This document is the **implementation plan** for Dinamic Inventory v3.0.5, derived from the product plan in `3.0.5.md`. It includes a repository assessment, sprint breakdown, and execution order.

---

## 1. Repository Assessment

### 1.1 Current frontend structure

- **Entry:** `main.tsx` — React 18, ReactRouter, MUI ThemeProvider, CssBaseline. No global state library.
- **Routes:** `App.tsx` — four routes: InventoriesList, InventoryDetail, AislePositionsPage, PositionDetailPage.
- **API layer:** `api/client.ts` — plain `fetch` functions returning Promises; all types from `api/types.ts`.
- **Types:** Single file `api/types.ts` (~217 lines) — DTOs, status unions, `ApiError` class, and shared shapes (Inventory, Aisle, PositionSummary, etc.). No separation between API DTOs and UI/domain types.
- **Pages:** Four main pages; each owns local state (`useState`), `useEffect` + `useCallback` for fetch and retry. Loading/error/empty handled inline with MUI `CircularProgress`, `Alert`, and `Paper`/typography.
- **Components:** `CreateInventoryDialog`, `CreateAisleDialog` — form dialogs that call API directly and invoke `onSuccess`/`onError` callbacks.
- **Utils:** `apiErrors.ts`, `formatDate.ts`, `jobStatus.ts`, `aisleStatus.ts`, `positionStatus.ts`, `resultRoutes.ts` — small, focused helpers.

**Conclusion:** Flat structure; no TanStack Query; no domain-oriented modules; types are centralized but monolithic; data fetching is imperative and duplicated (load, setLoading, setError, try/catch in every page).

### 1.2 Current data-fetching patterns

- **InventoriesList:** `useState` (inventories, loading, error) + `useCallback(load)` + `useEffect(() => load(), [load])`. Calls `getInventories()`. Create flow: dialog calls `createInventory()`, then `onSuccess` navigates or calls `load()`.
- **InventoryDetail:** Multiple sequential/parallel fetches: `getInventory` → `getAisles` → `fetchAssetCountsForAisles` (N× `getAisleAssets`) → `getInventoryMetrics`. Heavy use of `useState` (inventory, aisles, metrics, multiple loading/error flags), `cancelledRef` for cleanup. Mutations: `startAisleProcessing`, `uploadAisleAssets`, `createAisle` — each triggers manual `loadAisles()` or `loadInitial()`.
- **AislePositionsPage:** Same pattern: `getAislePositions` in `useEffect` + load/retry.
- **PositionDetailPage:** `getPositionDetail` in `useEffect`; `submitReviewAction` in callbacks with manual `load()` after each action.

**Conclusion:** No cache, no shared server state, no invalidation strategy. Every screen re-fetches on mount; mutations trigger ad-hoc refetch or local callback. Ideal for TanStack Query migration.

### 1.3 Current type organization

- **Single file:** `frontend/src/api/types.ts` contains: status const arrays and types, API DTOs (Inventory, Aisle, PositionSummary, etc.), request/response types, `ApiError` class. Used by `client.ts` and all pages/components.
- **No separation** between backend response shape and UI-only or domain models. No `modules/*/types` or `shared/types` yet.

**Conclusion:** Types are consistent and aligned with backend but monolithic. Sprint B will split by domain and layer (DTOs vs UI/shared).

### 1.4 Current repeated UI patterns

- **Loading:** `<Box sx={{ display: 'flex', justifyContent: 'center', py: 3/4 }}><CircularProgress /></Box>` repeated in every page.
- **Error:** `<Alert severity="error" ... action={<Button onClick={retry}>Retry</Button>}>` with slight variations.
- **Empty state:** `<Paper sx={{ p: 3/4 }}><Typography color="text.secondary">...</Typography></Paper>`.
- **Buttons:** MUI Button with variant/size; no shared wrapper.
- **Tables:** TableContainer + Table + TableHead/TableBody; no shared table shell.
- **Dialogs:** MUI Dialog with DialogTitle, DialogContent, DialogActions; CreateInventoryDialog and CreateAisleDialog are similar in structure.

**Conclusion:** Clear repetition for loading, error, empty; good candidates for shared components in Sprint C without building a full design system.

### 1.5 Backend `any` hotspots

- **API routes:** `jobs.py` — `_db_enabled(settings: Any)`, `_progress_dict(record: Any)`, form/settings in handlers. `inventories_v3.py` — `_parse_summary_quantity(raw: Any)`.
- **Repositories:** `sql_position_repository.py`, `sql_review_action_repository.py`, `sql_job_repository.py` — `_row_to_*(row: Any)` helpers. `database/repository.py` — `_row_to_iso(row: Any, attr)`.
- **Pipeline/context:** `run_context.py` — `settings: Any`; `contracts/stage.py` — `run(..., data: Any) -> Any`; `hybrid_inventory_pipeline.py` — `logger: Any`.
- **Frames/config:** `normalize.py` — settings and config snapshot use `Any`.
- **LLM/reid:** Providers take `settings: Any`; `clip_embedder.py`, `phash.py` — `sig: Any`.
- **Parsing/validation:** `global_analysis_parser.py` — `_safe_* (v: Any)`; `global_analysis_schema.py` — `_check_bbox(obj: Any, ...)`.
- **Ports:** `job_repository.py` — `update(..., **updates: Any)`.

**Conclusion:** Priorities for Sprint D: (1) API route handlers and DTOs, (2) repository row mappers (typed row/tuple), (3) settings type (single interface), (4) pipeline/context and ports.

### 1.6 Technical constraints

- **React 18**, **TypeScript ~5.3**, **Vite 5**, **MUI 5**. No existing query/cache library.
- **API:** REST under `/api/v3/`; Vite proxy to backend; no OpenAPI-generated client.
- **Backend:** FastAPI; Pydantic schemas; no change to API contracts in 3.0.5.
- **Tests:** Backend pytest; frontend has no test suite referenced in this repo — validation will be manual and smoke.

## 2. v3.0.5 Implementation Plan (by sprint)

### Sprint 3.0.5.A — Frontend Data Layer Foundation

**Goals**

- Install and configure TanStack Query (React Query).
- Introduce QueryClient and a single provider in the app tree.
- Define query key convention and a small key factory.
- Migrate main list/detail screens to query hooks; move mutations into mutation hooks with explicit invalidation.
- Keep API client as-is (fetch in `api/client.ts`); hooks call client and are used by pages.

**Affected files/folders**

- **New:** `frontend/src/api/queryKeys.ts` — key factory and constants.
- **New:** `frontend/src/hooks/useInventories.ts` — useInventoriesList, useInventoryDetail (inventory + aisles + metrics), and optionally useInventoryMetrics.
- **New:** `frontend/src/hooks/useAisles.ts` — useAislesList, useAisleAssets (if needed for asset counts).
- **New:** `frontend/src/hooks/usePositions.ts` — useAislePositions, usePositionDetail.
- **New:** `frontend/src/hooks/useMutations.ts` (or split per domain) — useCreateInventory, useCreateAisle, useStartAisleProcessing, useUploadAisleAssets, useSubmitReviewAction.
- **Modified:** `frontend/package.json` — add `@tanstack/react-query`.
- **Modified:** `frontend/src/main.tsx` — wrap app with QueryClientProvider.
- **Modified:** `frontend/src/pages/InventoriesList.tsx` — use useInventoriesList + useCreateInventory; remove local fetch state.
- **Modified:** `frontend/src/pages/InventoryDetail.tsx` — use useInventoryDetail, useInventoryMetrics (or integrated), and mutation hooks; remove manual load/refetch and most loading/error state.
- **Modified:** `frontend/src/pages/AislePositionsPage.tsx` — use useAislePositions; remove local fetch state.
- **Modified:** `frontend/src/pages/PositionDetailPage.tsx` — use usePositionDetail + useSubmitReviewAction mutation; remove local fetch/refetch.
- **Dialogs:** CreateInventoryDialog and CreateAisleDialog stay presentational; parents pass mutation.mutate or a callback that triggers invalidation/refetch (via mutation onSuccess).

**User stories (from plan)**

- US-A1: QueryClient and provider — done by main.tsx + default options.
- US-A2: Query key convention — done by queryKeys.ts and usage in hooks.
- US-A3: List screens use TanStack Query — InventoriesList, AislePositionsPage.
- US-A4: Detail screens use query hooks — InventoryDetail, PositionDetailPage.
- US-A5: Mutations with explicit invalidation — all create/process/upload/review mutations.

**Dependencies**

- Query keys and client must exist before hooks. Hooks must exist before page migration. No dependency on Sprint B/C/D.

**Risks**

- InventoryDetail currently does N+1 for asset counts; can remain a separate useQueries or a single endpoint later; for 3.0.5 keep existing behavior (fetch counts in hook or in page via hook).
- Over-invalidation: prefer invalidating specific keys (e.g. one inventory’s aisles) rather than global refetch.

**Validation criteria**

- No direct fetch in page components for list/detail data; data comes from useQuery hooks.
- Mutations invalidate only the necessary keys (e.g. ['inventories'], ['inventories', inventoryId, 'aisles']).
- Loading and error states still work; retry uses query refetch.
- Manual smoke: list → detail → create inventory → create aisle → process → positions → position detail → review action.

### Sprint 3.0.5.B — Frontend Type Structure and Contracts

**Goals**

- Split types into API DTOs vs UI/domain vs shared.
- Organize by domain (inventories, aisles, positions, reviews) under something like `src/types/` or `src/modules/*/types`.
- Reduce monolithic `api/types.ts`; keep a single re-export or thin barrel if needed for backward compatibility during migration.
- Centralize shared types (ids, status enums, pagination if any).

**Affected files/folders**

- **New:** `frontend/src/types/api/` or `frontend/src/types/dto/` — request/response DTOs (by resource).
- **New:** `frontend/src/types/domain/` or per-module types — UI-facing models if different from DTOs.
- **New:** `frontend/src/types/shared/` — status unions, ids, common fields.
- **Modified:** `frontend/src/api/types.ts` — re-export from new structure or split and re-export for minimal churn.
- **Modified:** `frontend/src/api/client.ts` — import from new type locations (or keep importing from types.ts that re-exports).
- **Modified:** All pages/hooks that import from `api/types` — switch to new imports or keep using barrel.

**User stories**

- US-B1: DTOs vs UI models separation.
- US-B2: Modularization by domain.
- US-B3: Shared contracts (status, ids).
- US-B4: Naming consistency.

**Dependencies**

- Best done after Sprint A so hooks and pages already use a single types entry point; then split types without changing behavior.

**Risks**

- Large file moves can cause merge conflicts; do in small steps (one domain at a time).

**Validation criteria**

- No monolithic types file; clear separation by layer and domain.
- No duplicate definitions of the same contract; imports are consistent.

### Sprint 3.0.5.C — UI Foundation / Reusable Components Base

**Goals**

- Add minimal shared components: LoadingState, ErrorState, EmptyState, and optionally a small PageContainer.
- Use them in the four main pages to replace repeated loading/error/empty markup.
- Do not introduce a full design system or tokens; keep styling to MUI sx or minimal variants.

**Affected files/folders**

- **New:** `frontend/src/components/shared/LoadingState.tsx`, `ErrorState.tsx`, `EmptyState.tsx` (and optionally `PageContainer.tsx`).
- **Modified:** InventoriesList, InventoryDetail, AislePositionsPage, PositionDetailPage — use shared components where loading/error/empty are shown.

**User stories**

- US-C1: Base shared components.
- US-C2: Consistent loading/error/empty.
- US-C3: Optional action/container wrappers if clearly repeated.
- US-C4: Base ready for 4.0 evolution.

**Dependencies**

- After Sprint A (and preferably B) so pages are already refactored to hooks; then replace only the presentational parts.

**Risks**

- Scope creep: avoid building more than loading/error/empty and one container if needed.

**Validation criteria**

- At least LoadingState, ErrorState, EmptyState exist and are used in key pages; duplication of the same markup is reduced.

### Sprint 3.0.5.D — Backend Type Hardening

**Goals**

- Replace high-impact `any` usages: route handlers (request/response), repository row mappers, settings type.
- Introduce explicit types (TypedDict, Protocol, or small interfaces) for rows, settings, and pipeline context where appropriate.
- No change to runtime behavior or API contracts.

**Affected files/folders**

- **Priority 1:** `src/api/routes/inventories_v3.py`, `src/api/routes/jobs.py` — handler signatures and DTOs.
- **Priority 2:** `src/infrastructure/repositories/sql_*.py` — row type (e.g. Row or tuple) and mapper signatures.
- **Priority 3:** `src/config.py` or a small `settings.py` interface — single Settings type used by routes and services.
- **Priority 4:** `src/pipeline/context/run_context.py`, `src/jobs/ports/job_repository.py` — context and port signatures.
- **Lower priority:** LLM providers, parsing helpers — replace `Any` with minimal interfaces or leave documented.

**User stories**

- US-D1: Inventory of `any` by priority.
- US-D2: Endpoints typed.
- US-D3: Services typed.
- US-D4: Adapters/repositories typed.
- US-D5: Incremental, safe removal.

**Dependencies**

- Independent of frontend sprints; can run in parallel after plan is agreed.

**Risks**

- Row types may depend on DB driver (e.g. Cursor vs Row); use typing that matches actual usage.
- Settings type must avoid circular imports (e.g. use TYPE_CHECKING and interface in a small module).

**Validation criteria**

- Critical endpoints and repository mappers have explicit types; no new `any` in new code in those areas; tests and manual smoke pass.

---

## 3. Recommended implementation order

1. **Sprint A** — Frontend data layer (this document’s first implementation step).
2. **Sprint B** — Frontend types (after A is merged and stable).
3. **Sprint C** — UI shared components (after B, or in parallel with B if types change is small).
4. **Sprint D** — Backend typing (can start after or in parallel with A; no dependency on frontend).

---

## 4. Success metrics (from plan)

- Fetch no longer done directly in main page components.
- QueryClient and query key convention in place.
- Mutations invalidate cache explicitly.
- Types no longer in a single monolithic file (after B).
- Shared loading/error/empty components exist and are used (after C).
- Critical backend endpoints and repositories have explicit types (after D).
