# **Dinamic Inventory v3.0.5 — Technical Hardening and UI Foundation Plan**

## **1\. Summary**

La versión **3.0.5** tiene como objetivo fortalecer la base técnica de **Dinamic Inventory v3.0** sin modificar su comportamiento funcional principal, incorporando además una **base mínima de componentes reutilizables** en frontend para preparar una evolución más ordenada hacia la futura versión 4.0.

Esta versión actúa como una iteración de **hardening técnico \+ UI foundation**, enfocada en cuatro líneas de trabajo:

1. **Frontend Data Layer** mediante adopción de **TanStack Query**.  
2. **Modularización de tipos en frontend** para mejorar mantenibilidad y claridad de contratos.  
3. **Eliminación progresiva de `any` y tipado ambiguo en backend**, reemplazándolos por tipos explícitos de dominio, DTOs y contratos mejor definidos.  
4. **Construcción de una base mínima de componentes reutilizables en frontend**, orientada a consistencia estructural y reutilización, sin abordar todavía un rediseño UX/UI completo.

La premisa principal es **extender la base de v3.0 sin reescribirla**, reduciendo deuda técnica y preparando el sistema para futuras capacidades funcionales y visuales de las líneas 3.1 y 4.0.

---

## **2\. Objective**

Fortalecer la plataforma v3.0 a nivel técnico y estructural, mejorando robustez, mantenibilidad, claridad de contratos, previsibilidad del flujo de datos y consistencia básica de interfaz, sin introducir cambios funcionales mayores en la experiencia de usuario ni en la lógica de negocio principal.

---

## **3\. Guiding Principle**

**Mantener la funcionalidad existente de v3.0 mientras se mejora la estructura técnica interna del frontend y backend, incorporando una base mínima de UI reutilizable, evitando refactors innecesariamente invasivos y preservando compatibilidad con los flujos actuales.**

---

## **4\. Version Positioning**

La versión 3.0.5 debe entenderse como una iteración intermedia entre:

* **v3.0**, que consolidó la base funcional actual,  
* **v3.1**, que podrá enfocarse en mejoras funcionales avanzadas,  
* **v4.0**, que tendrá espacio para una mejora significativa de UX/UI, diseño visual, consistencia global y evolución del sistema de componentes.

En este contexto:

* **3.0.5 no busca rediseñar la experiencia visual completa**,  
* pero **sí debe sentar una base de componentes reutilizables y contratos limpios** para que la futura 4.0 no tenga que partir desde una UI desordenada o duplicada.  
  ---

  ## **5\. Scope**

  ### **In scope**

  #### **Frontend**

* Integración de **TanStack Query** como capa estándar de fetch, cache, invalidación y mutaciones.  
* Reemplazo progresivo de fetch imperativo dentro de componentes por hooks especializados.  
* Definición de una convención formal de **query keys**, invalidaciones, manejo de errores y estados de carga.  
* Modularización de tipos frontend por dominio y por capa.  
* Separación explícita entre:  
  * API DTOs,  
  * tipos de dominio/UI,  
  * tipos compartidos.  
* Creación de una **base mínima de componentes reutilizables** para patrones ya repetidos en el sistema.  
* Estandarización de estados de interfaz frecuentes como loading, error, empty state, badges de estado y contenedores base.

  #### **Backend**

* Inventario y eliminación progresiva de `any` y otros tipos ambiguos de alta criticidad.  
* Tipado explícito de inputs, outputs, servicios y adapters.  
* Mejora de contratos internos sin alterar innecesariamente endpoints existentes.  
* Refuerzo de tipos en modelos de request/response y estructuras de dominio ya consolidadas.  
  ---

  ## **6\. Out of scope**

Queda fuera de la versión 3.0.5:

* consolidación avanzada de evidencias,  
* multi-image consolidation flow,  
* score de calidad de evidencia,  
* snapshots pre/post review,  
* nuevas métricas funcionales de revisión,  
* rediseño visual profundo de la plataforma,  
* definición completa de design system,  
* tokens visuales extensivos de color, tipografía, spacing y motion,  
* accesibilidad exhaustiva de nivel de sistema completo,  
* reescritura completa del frontend o backend,  
* cambios de comportamiento del producto que alteren reglas de negocio existentes.

Estos puntos pertenecen al roadmap de **v3.1** o **v4.0**.

---

## **7\. Why this version exists**

La versión 3.0.5 existe para resolver una necesidad concreta: **mejorar la calidad interna de la plataforma antes de avanzar con nuevas features funcionales o con un rediseño UX/UI más ambicioso**.

Sin esta capa de hardening, la evolución hacia 3.1 y 4.0 puede apoyarse sobre:

* fetches dispersos,  
* contratos poco claros,  
* tipos desorganizados,  
* puntos del backend con ambigüedad de datos,  
* componentes duplicados,  
* patrones UI repetidos sin estandarización,  
* mayor riesgo de errores silenciosos,  
* mayor costo futuro de rediseño visual.

La 3.0.5 reduce ese riesgo y deja una base más estable para el roadmap posterior.

## **8\. Technical and Structural Goals**

### **8.1 Frontend goals**

* Centralizar el acceso a datos.  
* Reducir lógica de networking en componentes.  
* Mejorar consistencia de loading / error / success.  
* Facilitar invalidación de datos tras mutaciones.  
* Organizar los tipos siguiendo límites de dominio.  
* Reducir acoplamiento entre componentes, API y estado remoto.  
* Introducir componentes reutilizables donde ya exista repetición clara.  
* Unificar estructuras base de UI sin bloquear decisiones visuales futuras.

  ### **8.2 Backend goals**

* Aumentar seguridad de tipos.  
* Reducir uso de estructuras implícitas o ambiguas.  
* Mejorar claridad de contratos entre capas.  
* Detectar errores en tiempo de desarrollo en lugar de en runtime.  
* Preparar el sistema para futuras extensiones de dominio sin propagar deuda técnica.

  ### **8.3 UI foundation goals**

* Reducir duplicación de markup y patrones base.  
* Facilitar evolución visual futura sobre una estructura más limpia.  
* Crear una primera capa de componentes compartidos de bajo riesgo.  
* Mejorar consistencia estructural entre pantallas existentes.

  ## **9\. TanStack Query Adoption Criteria**

La adopción de TanStack Query debe regirse por criterios técnicos explícitos. No se incorpora solo como librería de conveniencia, sino como estándar de acceso a estado remoto.

### **9.1 General rules**

* Todo dato remoto consumido por pantallas del producto debe obtenerse mediante TanStack Query, salvo casos excepcionalmente justificados.  
* Los componentes no deben contener lógica de fetch directo salvo en wrappers muy específicos y temporales durante migración.  
* Las llamadas HTTP deben vivir en una capa de API/clients separada de los componentes.  
* Los hooks basados en TanStack Query deben representar casos de uso del dominio y no detalles de implementación de bajo nivel.

  ### **9.2 Query keys criteria**

Las query keys deben:

* ser determinísticas,  
* reflejar el dominio y no detalles visuales,  
* permitir invalidación precisa,  
* soportar filtros y parámetros relevantes,  
* seguir una convención compartida.

  #### **Convención propuesta**

* `['reviews']`  
* `['reviews', 'list', filters]`  
* `['reviews', 'detail', reviewId]`  
* `['reviews', 'artifacts', reviewId]`  
* `['reviews', 'metrics', reviewId]`

  ### **9.3 Query hook criteria**

Cada hook debe:

* tener responsabilidad clara,  
* devolver datos ya tipados,  
* encapsular errores del fetch,  
* evitar transformar excesivamente el payload salvo que la transformación sea parte clara del caso de uso.

Ejemplos:

* `useReviewList(filters)`  
* `useReviewDetail(reviewId)`  
* `useReviewArtifacts(reviewId)`  
* `useReviewMetrics(reviewId)`

  ### **9.4 Mutation criteria**

Toda mutación debe:

* vivir en un hook específico,  
* declarar explícitamente qué keys invalida,  
* manejar estados de pending/success/error,  
* centralizar side effects derivados,  
* evitar lógica repetida por pantalla.

Ejemplos:

* `useUploadReview()`  
* `useApproveFinding()`  
* `useRejectFinding()`  
* `useRetryReviewProcess()`

  ### **9.5 Error handling criteria**

* Los errores de red, validación o negocio deben normalizarse en una capa compartida.  
* Las pantallas no deben interpretar manualmente múltiples formas de error provenientes de distintos endpoints.  
* Los hooks deben exponer un contrato consistente de error.

  ### **9.6 Loading and stale data criteria**

* Debe existir una política explícita de `staleTime` por tipo de recurso.  
* Recursos muy estables pueden tener un `staleTime` mayor.  
* Detalles que cambian tras acciones de revisión deben invalidarse con precisión.  
* No se debe abusar de refetch global cuando alcanza con invalidación localizada.

  ### **9.7 Retry criteria**

* `retry` debe configurarse según criticidad del endpoint.  
* Mutaciones sensibles no deben reintentarse ciegamente.  
* Queries de lectura tolerantes pueden reintentar de forma moderada.

  ### **9.8 Cache ownership criteria**

* TanStack Query debe manejar **server state**.  
* El estado puramente visual/local debe permanecer fuera de TanStack Query.  
* No debe usarse como reemplazo genérico de todo estado de UI.

  ## **10\. Frontend Type Modularization Criteria**

La modularización de tipos debe responder a límites de dominio y capas técnicas, no solo a división arbitraria de archivos.

### **10.1 Mandatory type layers**

Deben distinguirse al menos estas capas:

#### **API DTOs**

Representan request y response del backend.

#### **Domain/UI Models**

Representan estructuras adaptadas al uso interno de la UI.

#### **Shared/Common Types**

Representan tipos reutilizables transversales como paginación, ids, status, metadata común.

### **10.2 Organizational criteria**

Se recomienda una organización por módulos de dominio, por ejemplo:

* `src/modules/reviews/api/`  
* `src/modules/reviews/types/`  
* `src/modules/reviews/hooks/`  
* `src/modules/shared/types/`  
* `src/modules/shared/components/`

O una variante equivalente que respete separación clara.

### **10.3 Rules**

* No mezclar DTOs de API con tipos de rendering si sus responsabilidades son distintas.  
* Evitar archivos monolíticos de tipos sin frontera clara.  
* Reducir imports cruzados entre módulos no relacionados.  
* Unificar naming conventions.  
* Evitar redefiniciones duplicadas del mismo contrato en distintos lugares.  
  ---

  ## **11\. Reusable Components Foundation Criteria**

La incorporación de componentes reutilizables en 3.0.5 debe ser **selectiva, estructural y de bajo riesgo**. No se busca construir todavía un design system completo, sino establecer una base de UI compartida para patrones repetidos.

### **11.1 General principle**

Solo deben abstraerse componentes cuya reutilización sea clara y que ya estén presentes en múltiples pantallas o flujos del producto.

### **11.2 What should be included**

Pueden incluirse componentes base como:

* botones compartidos,  
* inputs base,  
* selects base,  
* modales base,  
* badges de estado,  
* loaders,  
* empty states,  
* error states,  
* cards contenedoras,  
* wrappers de acciones repetidas,  
* tablas o shells de listados si la abstracción es evidente.

  ### **11.3 What should not be forced in 3.0.5**

No debe intentarse en esta versión:

* un sistema completo de variantes visuales avanzadas,  
* una librería de componentes exhaustiva,  
* una definición final de identidad visual,  
* microinteracciones complejas,  
* decisiones visuales profundas que dependan de la futura UX/UI 4.0.

  ### **11.4 Abstraction criteria**

Un componente reusable debe crearse cuando:

* ya aparece repetido en más de un punto del sistema,  
* su contrato de props es claro,  
* aporta reducción de duplicación,  
* no depende de una decisión visual aún incierta,  
* mejora legibilidad o mantenibilidad del frontend.

  ### **11.5 Design constraint for this version**

Los componentes de 3.0.5 deben priorizar:

* consistencia estructural,  
* simplicidad,  
* contratos de props claros,  
* bajo acoplamiento,  
* facilidad de evolución futura.  
  ---

  ## **12\. Backend Typing Criteria**

  ### **12.1 General principle**

Eliminar `any` no es un objetivo cosmético. El objetivo es establecer contratos explícitos y verificables.

### **12.2 Priority order**

La remoción de `any` debe priorizarse en este orden:

1. endpoints y contratos públicos,  
2. DTOs/request/response,  
3. servicios de aplicación,  
4. adapters e integraciones,  
5. helpers internos.

   ### **12.3 Allowed replacements**

Según el caso, el reemplazo puede ser mediante:

* interfaces,  
* type aliases,  
* modelos de dominio,  
* DTOs,  
* unions discriminadas,  
* clases solo cuando exista comportamiento asociado,  
* estructuras validadas con el mecanismo tipado que ya use el proyecto.

  ### **12.4 Rules**

* No reemplazar `any` por tipos igual de ambiguos salvo justificación temporal.  
* No introducir clases artificiales donde basta un contrato de datos.  
* Tipar explícitamente estructuras intermedias relevantes del pipeline si afectan contratos o lógica crítica.  
* Los adapters deben exponer tipos concretos y predecibles.

  ## **13\. Architecture Impact**

  ### **13.1 Expected frontend impact**

* Menor acoplamiento entre vistas y networking.  
* Mejor control del estado remoto.  
* Pantallas más declarativas.  
* Menor duplicación de lógica de fetch.  
* Tipos más localizables y mantenibles.  
* Base inicial de componentes compartidos reutilizables.  
* Mejor preparación para una evolución de UX/UI en 4.0.

  ### **13.2 Expected backend impact**

* Mayor claridad en contratos internos.  
* Menos errores silenciosos por estructuras no tipadas.  
* Refactors futuros más seguros.  
* Mejor base para nuevas capacidades funcionales.  
  ---

  ## **14\. Risks**

  ### **14.1 Frontend risks**

* Migrar a TanStack Query sin una convención clara de keys.  
* Duplicar lógica entre hooks nuevos y fetches viejos durante transición.  
* Mezclar estado remoto y estado local de UI.  
* Modularizar tipos sin criterio de dominio.  
* Intentar crear demasiados componentes reutilizables sin suficiente estabilidad de diseño.

  ### **14.2 Backend risks**

* Reemplazar `any` por tipos superficiales sin valor real.  
* Tocar demasiados archivos de bajo impacto y no priorizar zonas críticas.  
* Romper contratos internos por falta de validación incremental.

  ### **14.3 Delivery risks**

* Convertir 3.0.5 en una reescritura encubierta.  
* Mezclar esta versión con funcionalidades de 3.1 o decisiones visuales profundas de 4.0.  
* Subestimar el esfuerzo de estandarización de contratos y componentes base.  
  ---

  ## **15\. Acceptance Criteria**

La versión 3.0.5 se considerará aceptada si se cumplen al menos los siguientes criterios:

### **Frontend**

* Las pantallas críticas ya no realizan fetch imperativo directo en componentes principales.  
* Existe un `QueryClient` configurado y una convención clara de query keys.  
* Las mutaciones principales invalidan o actualizan cache de forma consistente.  
* Los tipos ya no viven en un archivo monolítico sin separación por dominio/capa.  
* Los módulos principales exponen tipos y hooks claros y reutilizables.  
* Existe una base inicial de componentes reutilizables para patrones repetidos del sistema.  
* Los estados comunes de interfaz muestran mayor consistencia estructural.

  ### **Backend**

* Los `any` de alta criticidad fueron removidos o encapsulados con contrato temporal explícito.  
* Los endpoints y servicios principales tienen tipos de entrada/salida definidos.  
* Los adapters/integraciones críticas ya no dependen de payloads ambiguos.  
* El tipado refuerza contratos sin cambiar la lógica funcional principal.

  ### **Global**

* No hubo regresiones funcionales relevantes.  
* La plataforma mantiene compatibilidad con los flujos actuales de v3.0.  
* La base queda preparada para iniciar 3.1 con menor deuda técnica y 4.0 con mejor estructura visual y de componentes.

  # **Sprint 3.0.5.A — Frontend Data Layer Foundation**

  ## **Sprint objective**

Introducir TanStack Query como capa estándar de acceso a datos en frontend para los flujos principales del sistema.

## **Scope**

* instalación y configuración de TanStack Query,  
* creación de `QueryClient`,  
* definición de estrategia de query keys,  
* migración inicial de endpoints/pantallas prioritarias,  
* creación de hooks de lectura y mutación más importantes,  
* estandarización de loading/error/success.

  ## **User Stories**

  ### **US-A1 — Configuración base de TanStack Query**

**Como** desarrollador frontend  
**quiero** tener TanStack Query configurado globalmente  
**para** estandarizar el manejo de estado remoto en toda la aplicación.

#### **Acceptance Criteria**

* existe configuración global de `QueryClient`,  
* el provider envuelve correctamente la aplicación,  
* la configuración base define defaults razonables de retry y caching.  
  ---

  ### **US-A2 — Convención de query keys**

**Como** equipo técnico  
**quiero** definir una convención formal para query keys  
**para** invalidar y leer cache de forma consistente.

#### **Acceptance Criteria**

* existe convención documentada,  
* las nuevas queries respetan esa convención,  
* las keys son determinísticas y previsibles.  
  ---

  ### **US-A3 — Migración de listados principales**

**Como** usuario interno de la plataforma  
**quiero** que las pantallas de listado carguen datos con una capa de fetch más robusta  
**para** tener una experiencia más estable y consistente.

#### **Acceptance Criteria**

* el listado prioritario migra a TanStack Query,  
* desaparece el fetch manual principal del componente,  
* loading/error/empty state siguen funcionando correctamente.  
  ---

  ### **US-A4 — Migración de detalles principales**

**Como** usuario interno  
**quiero** que las pantallas de detalle usen hooks de query centralizados  
**para** reducir inconsistencias y duplicación de lógica.

#### **Acceptance Criteria**

* el detalle prioritario usa un hook dedicado,  
* el hook retorna datos tipados,  
* se manejan correctamente estados de error y carga.  
  ---

  ### **US-A5 — Mutaciones con invalidación explícita**

**Como** desarrollador frontend  
**quiero** encapsular mutaciones relevantes en hooks con invalidación explícita  
**para** mantener la cache consistente tras cambios de estado.

#### **Acceptance Criteria**

* las mutaciones principales viven fuera del componente,  
* invalidan keys puntuales,  
* no dependen de refetch manual global salvo justificación.  
  ---

  ## **Sprint deliverables**

* base de TanStack Query integrada,  
* query keys normalizadas,  
* hooks iniciales de lectura y mutación,  
* primeras pantallas críticas migradas.  
  ---

  # **Sprint 3.0.5.B — Frontend Type Structure and Contracts**

  ## **Sprint objective**

Reorganizar los tipos del frontend para reflejar mejor el dominio, separar capas y reducir deuda de contratos.

## **Scope**

* modularización de `types`,  
* separación de DTOs, domain models y shared types,  
* limpieza de imports cruzados,  
* unificación de convenciones de naming,  
* reducción de duplicación de contratos.

  ## **User Stories**

  ### **US-B1 — Separación de DTOs y modelos de UI**

**Como** desarrollador frontend  
**quiero** diferenciar claramente los tipos de API de los tipos usados por la UI  
**para** evitar acoplamiento innecesario entre backend y rendering.

#### **Acceptance Criteria**

* los DTOs de request/response tienen ubicación clara,  
* los modelos de UI no están mezclados arbitrariamente con DTOs,  
* existe trazabilidad entre contrato API y consumo interno.  
  ---

  ### **US-B2 — Modularización por dominio**

**Como** equipo técnico  
**quiero** organizar los tipos por módulos de dominio  
**para** mejorar mantenibilidad y navegación del código.

#### **Acceptance Criteria**

* los tipos principales dejan de depender de un archivo monolítico,  
* cada módulo principal expone sus tipos relevantes,  
* disminuyen imports desordenados o indirectos.  
  ---

  ### **US-B3 — Contratos compartidos reutilizables**

**Como** desarrollador  
**quiero** tener tipos compartidos centralizados para estructuras comunes  
**para** evitar redefiniciones inconsistentes.

#### **Acceptance Criteria**

* tipos como status, ids, paginación o metadata común se centralizan,  
* no existen duplicados innecesarios de estos contratos.  
  ---

  ### **US-B4 — Nomenclatura consistente**

**Como** equipo técnico  
**quiero** que los tipos sigan una convención homogénea  
**para** facilitar lectura y mantenimiento.

#### **Acceptance Criteria**

* se aplica convención consistente de nombres,  
* los nuevos módulos respetan esa convención,  
* disminuye ambigüedad entre request, response, model y payload.  
  ---

  ## **Sprint deliverables**

* estructura de tipos modularizada,  
* DTOs separados de modelos de UI,  
* tipos compartidos centralizados,  
* imports y naming más claros.

#### **Acceptance Criteria**

**Como** equipo frontend  
**quiero** abstraer wrappers repetidos de acciones o contenedores

**Como** equipo de producto

  # **Sprint 3.0.5.D — Backend Type Hardening**

  ## **Sprint objective**

Reducir ambigüedad tipada en backend, removiendo `any` de mayor riesgo y reforzando contratos críticos.

## **Scope**

* inventario y clasificación de `any`,  
* priorización por criticidad,  
* tipado de endpoints y DTOs,  
* tipado de servicios/aplicación,  
* refuerzo de adapters críticos,  
* limpieza controlada sin alterar lógica funcional.

  ## **User Stories**

  ### **US-D1 — Inventario de `any` críticos**

**Como** equipo backend  
**quiero** identificar y clasificar los `any` existentes por criticidad  
**para** priorizar el hardening de las zonas con mayor riesgo.

#### **Acceptance Criteria**

* existe inventario inicial de usos de `any`,  
* cada caso relevante tiene criticidad asignada,  
* la priorización favorece contratos públicos y flujos centrales.  
  ---

  ### **US-D2 — Tipado de endpoints principales**

**Como** desarrollador backend  
**quiero** que los endpoints principales tengan tipos explícitos de entrada y salida  
**para** reforzar el contrato del sistema.

#### **Acceptance Criteria**

* los endpoints prioritarios ya no exponen payloads ambiguos,  
* requests y responses tienen contratos definidos,  
* el tipado no rompe compatibilidad funcional.  
  ---

  ### **US-D3 — Tipado de servicios de aplicación**

**Como** equipo backend  
**quiero** que los servicios principales usen tipos explícitos  
**para** reducir errores por estructuras implícitas y mejorar refactorabilidad.

#### **Acceptance Criteria**

* los servicios prioritarios ya no dependen de `any` en sus entradas/salidas clave,  
* existe trazabilidad clara entre DTOs, dominio y resultado.  
  ---

  ### **US-D4 — Refuerzo de adapters críticos**

**Como** desarrollador backend  
**quiero** tipar adapters e integraciones críticas  
**para** encapsular mejor datos provenientes de fuentes externas o capas adyacentes.

#### **Acceptance Criteria**

* los adapters prioritarios exponen contratos concretos,  
* la ambigüedad se reduce en zonas de integración,  
* se evita propagación de tipos inseguros hacia capas superiores.  
  ---

  ### **US-D5 — Eliminación progresiva y segura**

**Como** equipo técnico  
**quiero** que la remoción de `any` sea incremental y segura  
**para** mejorar robustez sin introducir regresiones por refactor excesivo.

#### **Acceptance Criteria**

* el hardening se realiza por zonas priorizadas,  
* los cambios tienen alcance controlado,  
* la funcionalidad observable se mantiene estable.  
  ---

  ## **Sprint deliverables**

* inventario priorizado de `any`,  
* endpoints críticos tipados,  
* servicios principales endurecidos,  
* adapters prioritarios con contratos explícitos.  
  ---

  ## **17\. Recommended Delivery Sequence**

1. **Sprint A primero**, porque establece la base de acceso a datos en frontend.  
2. **Sprint B después**, para ordenar contratos y tipos del frontend ya migrado.  
3. **Sprint C luego**, para construir una base reusable sobre una estructura frontend más limpia.  
4. **Sprint D al final**, enfocando backend con el contexto ya estabilizado del frontend y los contratos visibles.

Esta secuencia reduce retrabajo y mejora coherencia entre capas.

## **18\. Success Metrics**

Aunque la 3.0.5 no es una versión funcional, sí puede medirse por señales técnicas:

* reducción de fetch directo en componentes,  
* reducción del archivo monolítico de tipos,  
* cantidad de hooks de data layer estandarizados,  
* cantidad de componentes base reutilizables creados y usados,  
* reducción de markup duplicado en pantallas clave,  
* porcentaje de endpoints críticos con contratos explícitos,  
* reducción de `any` en zonas de alta criticidad,  
* disminución de lógica duplicada de carga/error en frontend.

  ## **19\. Final Note**

La versión 3.0.5 debe entenderse como una **versión de estabilización técnica y preparación estructural de interfaz**. No busca cambiar qué hace el producto, sino **mejorar cómo está construido** para habilitar una evolución más segura tanto en la línea funcional 3.1 como en la línea visual/UX 4.0.

En términos estratégicos:

* **v3.0** consolidó la base funcional,  
* **v3.0.5** endurece la plataforma y crea una UI foundation mínima,  
* **v3.1** podrá enfocarse en nuevas capacidades funcionales con menor deuda técnica,  
* **v4.0** podrá abordar mejoras significativas de UX/UI apoyándose en una estructura más limpia y reutilizable.
