# **Dinamic Inventory — Roadmap de Implementación**

# **Versión 3.0 — Base operativa web del sistema**

## **1\. Objetivo general de la 3.0**

La versión 3.0 debe transformar el sistema actual en una **plataforma operativa de inventario**, donde el usuario pueda:

* crear un inventario,  
* cargar evidencia de un pasillo,  
* procesar múltiples fotos o videos,  
* visualizar resultados estructurados,  
* revisar pallets/posiciones detectadas,  
* corregir conteos o productos,  
* eliminar detecciones erróneas,  
* medir el desempeño del inventario.

La 3.0 no debe intentar resolver toda la complejidad futura del producto. Su meta es dejar una **primera versión realmente utilizable**, con frontend web en React, backend con dominio claro, persistencia, revisión humana y métricas básicas.

---

# **2\. Objetivos específicos de la 3.0**

## **Objetivos funcionales**

* Representar el inventario mediante entidades reales del negocio.  
* Procesar evidencia agrupada por pasillo.  
* Soportar múltiples fotos y video por pasillo.  
* Mostrar resultados en tabla para revisión manual.  
* Permitir correcciones y eliminación lógica de pallets mal detectados.  
* Calcular métricas simples de éxito del inventario.

## **Objetivos técnicos**

* Separar el dominio de negocio del pipeline técnico.  
* Dejar `Job` como entidad interna del backend.  
* Persistir inputs, resultados, evidencias y correcciones.  
* Exponer una API clara para ser consumida por React.  
* Reutilizar el pipeline actual, pero mapeando sus salidas al nuevo dominio.

## **Objetivos de producto**

* Dejar de pensar el sistema como “un proceso que corre”.  
* Empezar a pensarlo como “una operación de inventario revisable”.

---

# **3\. Alcance de la 3.0**

## **Incluido**

* Inventarios  
* Pasillos  
* Assets de entrada: fotos, video o mixto  
* Procesamiento asociado a pasillos  
* Resultados estructurados por posición/pallet  
* Evidencias visuales asociadas  
* Frontend web básico en React  
* Revisión manual básica  
* Métricas básicas  
* Borrado lógico y auditoría mínima de correcciones

## **No incluido**

* Roles/permisos avanzados  
* Comparativas históricas entre inventarios  
* Reprocesamiento selectivo desde la UI  
* Normalización avanzada de etiquetas  
* Mapeo físico exacto del depósito  
* Integraciones con WMS externo  
* Merge/split manual de pallets  
* Flujo multiusuario complejo

---

# **4\. Modelo conceptual de la 3.0**

## **4.1 Inventory**

Es la entidad principal de negocio.  
Representa una sesión de inventario completa.

Contiene:

* datos generales  
* estado  
* pasillos asociados  
* métricas globales

## **4.2 Aisle**

Es la unidad operativa principal.  
Cada pasillo se procesa de manera independiente.

Contiene:

* assets de entrada  
* estado de procesamiento  
* resultados detectados  
* métricas por pasillo

## **4.3 Position**

Es la **unidad revisable** de la 3.0. En la práctica representa una **unidad de inventario tipo pallet detectada por el pipeline**, no un slot físico del depósito ya mapeado. Contiene:

* evidencia principal  
* productos detectados  
* estado  
* confianza  
* bandera de revisión

Para estados, transiciones y definición canónica, ver **Documento técnico 3.0 §7.4**.

## **4.4 ProductRecord**

Representa un producto/SKU detectado dentro de una posición.

Contiene:

* sku  
* descripción  
* cantidad detectada  
* cantidad corregida  
* confianza

## **4.5 Evidence**

Representa una evidencia visual asociada a una entidad.

Puede corresponder a:

* imagen original,  
* frame,  
* crop de pallet,  
* crop de producto,  
* crop de etiqueta,  
* imagen anotada.

## **4.6 Job**

Es una entidad técnica, no visible como concepto principal al usuario.

Se asocia a:

* `Aisle`

Y representa:

* ingestión,  
* extracción,  
* detección,  
* clasificación,  
* consolidación.

# **5\. Principios de diseño de la 3.0**

## **5.1 El usuario opera inventarios, no jobs**

La UI debe mostrar:

* inventarios,  
* pasillos,  
* posiciones,  
* revisión,  
* métricas.

No debe obligar al usuario a pensar en la estructura interna del pipeline.

## **5.2 Un pasillo es la unidad natural de procesamiento**

Un llamado operativo debe corresponder a un pasillo.

Ventajas:

* menor complejidad,  
* mejor trazabilidad,  
* reintento más simple,  
* revisión más clara.

## **5.3 La evidencia debe ser una entidad de primera clase**

No solo importa el resultado; importa qué imagen lo respalda.

## **5.4 Las correcciones no deben sobrescribir silenciosamente**

Hay que mantener:

* valor detectado,  
* valor corregido,  
* historial básico de revisión.

## **5.5 La primera versión debe ser funcional, no perfecta**

La 3.0 debe priorizar:

* flujo completo usable,  
* claridad,  
* trazabilidad básica,  
* revisión efectiva.

# **6\. Fases de implementación de la 3.0**

---

## **FASE 0 — Definición funcional y técnica**

### **Objetivo**

Cerrar el contrato del sistema antes de empezar a implementar.

### **Entregables**

* documento de alcance 3.0,  
* modelo de entidades,  
* estados del sistema,  
* endpoints iniciales,  
* definición del flujo frontend/backend.

### **Decisiones que deben quedar cerradas**

* `Inventory` es la entidad raíz.  
* `Aisle` es la unidad operativa.  
* `Position` es la unidad revisable.  
* `Job` se asocia a `Aisle`.  
* el frontend será React consumiendo API REST.  
* la salida del pipeline debe persistirse en el nuevo modelo.

### **Resultado esperado**

Una base clara que evite retrabajo en fases posteriores.

---

## **FASE 1 — Modelo de dominio y persistencia**

### **Objetivo**

Crear la estructura backend mínima para representar el negocio.

### **Entidades mínimas**

* `inventories`  
* `aisles`  
* `source_assets`  
* `jobs`  
* `positions`  
* `products`  
* `evidences`  
* `review_actions`

### **Implementación**

* definir esquema SQL,  
* crear repositorios o capa de acceso a datos,  
* crear modelos de dominio,  
* establecer relaciones entre entidades,  
* definir estados válidos.

### **Estados sugeridos**

#### **Inventory**

* `draft`  
* `processing`  
* `in_review`  
* `completed`  
* `failed`

#### **Aisle**

* `created`  
* `assets_uploaded`  
* `queued`  
* `processing`  
* `processed`  
* `in_review`  
* `completed`  
* `failed`

#### **Position**

* `detected`  
* `reviewed`  
* `corrected`  
* `deleted`

### **Criterios de aceptación**

* se puede crear un inventario,  
* se puede crear un pasillo,  
* se puede asociar un pasillo a un inventario,  
* se puede crear un job asociado a un pasillo,  
* se pueden persistir resultados estructurados.

Para transiciones de estado de Inventory, Aisle y Position, ver **Documento técnico 3.0 §7**.

---

## **FASE 2 — Ingesta de pasillos y assets**

### **Objetivo**

Permitir que un pasillo reciba múltiples imágenes, video o mezcla.

### **Implementación**

* endpoint para crear pasillo,  
* endpoint para subir assets,  
* almacenamiento físico de archivos,  
* metadata por archivo,  
* validación de formatos soportados,  
* asociación de cada archivo al pasillo.

### **Entidad SourceAsset**

Campos sugeridos:

* `id`  
* `aisle_id`  
* `type` (`photo`, `video`)  
* `original_filename`  
* `storage_path`  
* `mime_type`  
* `uploaded_at`  
* `metadata_json`

### **Reglas**

* un pasillo puede tener 1..N assets,  
* los assets pueden procesarse juntos como una misma unidad lógica,  
* el estado del pasillo cambia según el avance del procesamiento.

### **Criterios de aceptación**

* el backend acepta múltiples fotos,  
* el backend acepta video,  
* el backend guarda metadata,  
* los archivos quedan trazables desde el pasillo.

---

## **FASE 3 — Orquestación backend y jobs internos**

### **Objetivo**

Conectar la creación de pasillos con el disparo del pipeline.

### **Implementación**

* al cargar assets, crear uno o más jobs internos,  
* asociar jobs al pasillo,  
* definir cola o mecanismo de ejecución,  
* registrar estado, timestamps, errores y artifacts técnicos.

### **Recomendación**

No exponer los jobs como recurso principal al frontend.  
El frontend debe consultar el estado del pasillo.

### **Posibles tipos de job**

* `ingestion`  
* `frame_extraction`  
* `detection`  
* `classification`  
* `consolidation`

### **Criterios de aceptación**

* al cargar un pasillo, se dispara procesamiento,  
* el pasillo refleja estado de ejecución,  
* si un job falla, el pasillo refleja error y contexto mínimo.

---

## **FASE 4 — Adaptación del pipeline al nuevo modelo**

### **Objetivo**

Mapear la salida técnica del pipeline actual al nuevo dominio.

### **Implementación**

El pipeline debe generar, por cada pasillo:

* posiciones detectadas,  
* productos asociados por posición,  
* evidencias por posición/producto,  
* confianza,  
* flags de revisión.

### **Qué hay que resolver**

* cómo consolidar múltiples frames/fotos de una misma posición,  
* cómo elegir la evidencia principal,  
* cómo persistir crops útiles,  
* cómo dejar resultados estructurados.

### **Salida esperada**

Por cada `Position`:

* `primary_evidence`  
* `confidence`  
* `needs_review`  
* lista de `ProductRecord`

Por cada `ProductRecord`:

* sku,  
* descripción,  
* cantidad detectada,  
* confianza,  
* evidencia opcional.

### **Criterios de aceptación**

* al terminar el procesamiento, existen posiciones persistidas,  
* cada posición tiene al menos una evidencia asociada,  
* los productos quedan vinculados a su posición,  
* el pasillo puede consultarse sin depender de archivos intermedios.

---

## **FASE 5 — API REST de consulta y revisión**

### **Objetivo**

Exponer una API clara para que React pueda operar el sistema.

### **Endpoints mínimos**

#### **Inventarios**

* `POST /inventories`  
* `GET /inventories`  
* `GET /inventories/{inventoryId}`

#### **Pasillos**

* `POST /inventories/{inventoryId}/aisles`  
* `GET /inventories/{inventoryId}/aisles`  
* `GET /aisles/{aisleId}`

#### **Assets**

* `POST /aisles/{aisleId}/assets`

#### **Resultados**

* `GET /aisles/{aisleId}/positions`  
* `GET /positions/{positionId}`

#### **Revisión**

* `PATCH /positions/{positionId}`  
* `POST /positions/{positionId}/reviews`

#### **Métricas**

* `GET /inventories/{inventoryId}/metrics`

### **Consideraciones**

* respuestas paginadas; parámetros y filtros según **Documento técnico 3.0 §9.7** (`page`, `page_size`, `status`, `needs_review`, `min_confidence`, `sku`),  
* URLs de evidencia o endpoints dedicados para servir imágenes.

### **Criterios de aceptación**

* React puede listar inventarios,  
* React puede listar pasillos,  
* React puede consultar posiciones,  
* React puede modificar una posición,  
* React puede obtener métricas.

## **FASE 7 — Revisión manual básica**

### **Objetivo**

Completar el loop humano del inventario.

### **Acciones mínimas**

* confirmar una posición sin cambios,  
* corregir cantidad,  
* corregir SKU,  
* eliminar lógicamente una posición.

### **Modelo de revisión**

Cada acción debe generar un registro en `review_actions`.

Campos sugeridos:

* `id`  
* `position_id`  
* `action_type`  
* `before_json`  
* `after_json`  
* `created_at`  
* `user_id` opcional para futuro

### **Reglas**

* `deleted` no borra físicamente,  
* los valores detectados deben conservarse,  
* el valor final debe poder reconstruirse.

### **Criterios de aceptación**

* un usuario puede modificar una posición desde la web,  
* queda registro de la corrección,  
* la tabla se actualiza correctamente,  
* la posición puede marcarse como eliminada.

---

## **FASE 8 — Métricas básicas y cierre**

### **Objetivo**

Medir el resultado del inventario y permitir marcarlo como finalizado.

### **Métricas mínimas por inventario**

* total de pasillos,  
* total de posiciones detectadas,  
* posiciones corregidas, eliminadas, aceptadas sin cambios,  
* success rate, correction rate, deletion rate (definición canónica en **Documento técnico 3.0 §9.6**),  
* tiempo promedio por pasillo,  
* total de productos unknown.

### **Fórmula de éxito**

Ver **Documento técnico 3.0 §9.6** para la definición canónica de `success_rate`, `total_reviewed_positions`, `auto_accepted_positions`, etc.

### **Cierre operativo**

* un pasillo puede marcarse `completed`,  
* un inventario pasa a `completed` cuando todos sus pasillos cerraron.

### **Criterios de aceptación**

* se pueden consultar métricas consolidadas,  
* el sistema refleja porcentaje de éxito,  
* el usuario puede identificar rápidamente dónde hubo fallas.

# **7\. Historias de usuario clave para la 3.0**

## **HU1 — Crear inventario**

Como operador, quiero crear un inventario nuevo para organizar la ejecución de una sesión completa.

## **HU2 — Crear pasillo**

Como operador, quiero crear un pasillo dentro de un inventario para procesarlo de forma independiente.

## **HU3 — Subir evidencia**

Como operador, quiero subir múltiples fotos o un video de un pasillo para que el sistema procese esa evidencia.

## **HU4 — Ver estado del pasillo**

Como operador, quiero consultar si un pasillo está en cola, procesando, listo o falló.

## **HU5 — Consultar resultados**

Como revisor, quiero ver una tabla de posiciones detectadas con sus productos y evidencias para validar el inventario.

## **HU6 — Ver detalle de posición**

Como revisor, quiero abrir una posición puntual para revisar su evidencia y los productos detectados.

## **HU7 — Corregir conteo**

Como revisor, quiero corregir la cantidad detectada de un producto cuando el sistema se equivoque.

## **HU8 — Corregir identificación**

Como revisor, quiero corregir un SKU si el sistema asignó un producto incorrecto.

## **HU9 — Eliminar detección errónea**

Como revisor, quiero marcar como eliminada una posición/pallet mal detectado.

## **HU10 — Medir desempeño**

Como responsable del inventario, quiero ver métricas de éxito para saber cuántos resultados fueron correctos sin intervención manual.

# **8\. Arquitectura sugerida para la 3.0**

## **Frontend**

Aplicación React en **`frontend/`** en la raíz del repositorio. Ver **FRONTEND_ESTRUCTURA.md** para estructura y stack (React, TypeScript, MUI, MUI Data Grid, etc.).

## **Backend API**

Capa que:

* expone recursos del dominio,  
* dispara jobs,  
* persiste resultados,  
* sirve evidencias y métricas.

## **Pipeline**

Capa interna especializada en:

* extracción,  
* detección,  
* clasificación,  
* consolidación.

## **Persistencia**

* DB para entidades del negocio,  
* filesystem o storage para evidencias y artifacts.

## **Regla importante**

La base de datos debe ser la fuente de verdad del negocio.  
El storage debe contener archivos, no el estado principal.

# **9\. Riesgos de la 3.0**

## **Riesgo 1 — Acoplar demasiado la UI al pipeline actual**

Mitigación:

* exponer recursos de dominio, no artifacts internos.

## **Riesgo 2 — No definir bien la unidad revisable**

Mitigación:

* usar `Position` como unidad revisable equivalente al pallet lógico detectado en 3.0 (definición canónica en **Documento técnico 3.0 §7.4**).

## **Riesgo 3 — Falta de evidencia clara**

Mitigación:

* garantizar evidencia principal por posición.

## **Riesgo 4 — Correcciones que pisan el resultado original**

Mitigación:

* guardar detected vs corrected \+ review action.

## **Riesgo 5 — Exceso de alcance**

Mitigación:

* reservar varias mejoras para 3.1 y 3.2.

Subir precisión operativa, mejorar la asociación entre evidencia, pallet y producto, y hacer más eficiente la revisión humana.

## **Posibles ejes**

* consolidación avanzada de evidencia,  
* mejor selección de imagen principal,  
* score de calidad de evidencia,  
* filtros avanzados en la tabla de revisión,  
* mejora del prompt/salida estructurada,  
* exportación de resultados,  
* métricas más detalladas.

## **Fases resumidas**

* **3.1.A** consolidación avanzada de evidencias,  
* **3.1.B** UX de revisión más rica,
