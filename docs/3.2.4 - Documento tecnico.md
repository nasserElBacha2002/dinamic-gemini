# **RFC técnica — v3.2.4**

## **Referencias visuales opcionales por inventario en arquitectura multi-provider**

## **1\. Estado**

**Propuesto**

---

## **2\. Resumen**

La versión **v3.2.4** introduce soporte para **referencias visuales opcionales por inventario**, cargadas desde el frontend durante el flujo de creación del inventario, con el fin de enriquecer el contexto visual del análisis automático de pasillos.

La solución debe integrarse con la arquitectura actual basada en **strategy pattern para múltiples proveedores de análisis**, por lo que las referencias visuales **no deben diseñarse como una feature específica de Gemini**. En cambio, deben modelarse como parte del **Analysis Context** del dominio/aplicación, dejando que cada proveedor decida cómo consumirlas según sus capacidades.

En proveedores multimodales, estas referencias podrán enviarse como imágenes de contexto. En proveedores no multimodales, podrán degradarse a metadata textual o ignorarse explícitamente de manera controlada.

---

## **3\. Motivación**

Actualmente, el análisis de un pasillo depende principalmente de:

* imágenes/evidencia principal del pasillo,  
* prompt textual del análisis,  
* reglas y parsing del proveedor elegido.

Este enfoque puede ser insuficiente cuando el estándar visual del inventario no está completamente representado en el prompt textual. Esto afecta especialmente casos donde:

* el formato de etiqueta esperado es específico del cliente o inventario,  
* existen diferencias sutiles entre etiquetas válidas y no válidas,  
* hay ruido, desenfoque o ambigüedad en la evidencia capturada,  
* el proveedor necesita contexto adicional para inferir mejor qué buscar.

Las referencias visuales por inventario buscan resolver esto mediante una mejora simple: permitir al usuario cargar imágenes representativas del estándar esperado y hacerlas disponibles como contexto del análisis.

---

## **4\. Objetivos**

### **4.1 Objetivos funcionales**

* permitir que el usuario cargue imágenes de referencia al crear un inventario;  
* hacer que este paso sea opcional;  
* persistir esas referencias asociadas al inventario;  
* exponerlas al frontend para visualización básica;  
* incorporarlas al contexto de análisis cuando se ejecute un job sobre un pasillo de ese inventario;  
* dejar trazabilidad de qué referencias fueron usadas.

### **4.2 Objetivos de arquitectura**

* mantener independencia del proveedor de análisis;  
* no acoplar la solución a Gemini;  
* integrar las referencias dentro del contrato de contexto de análisis común;  
* permitir que cada estrategia/proveedor consuma o ignore las referencias según soporte;  
* evitar que la lógica viva dentro del route handler o dentro de un proveedor específico.

---

## **5\. No objetivos**

Esta versión no incluye:

* entrenamiento o fine-tuning de modelos;  
* clasificación automática del tipo de referencia;  
* librería global compartida entre inventarios;  
* referencias multi-tenant complejas;  
* versionado avanzado de referencias;  
* scoring o ranking inteligente de referencias;  
* deduplicación avanzada;  
* edición compleja o administración enterprise;  
* catálogo global de estándares visuales.

---

## **6\. Principios de diseño**

### **6.1 Provider-agnostic**

La feature debe existir a nivel de dominio/aplicación, no a nivel del proveedor.

### **6.2 Optional by default**

La ausencia de referencias no altera el flujo actual.

### **6.3 Graceful degradation**

Si el proveedor no soporta imágenes de contexto, la feature no debe romper el análisis.

### **6.4 Minimal surface**

El release debe resolver el flujo principal sin abrir scope innecesario.

### **6.5 Traceability**

Debe quedar registro de si las referencias fueron resueltas y si efectivamente fueron utilizadas por el proveedor.

---

## **7\. Definición funcional**

## **7.1 Flujo de usuario**

### **Creación de inventario**

El usuario crea un inventario mediante un flujo de dos pasos:

#### **Paso 1**

Completa los datos básicos del inventario.

#### **Paso 2**

Puede opcionalmente cargar entre 1 y 3 imágenes de referencia.

El usuario puede:

* cargar referencias,  
* omitir el paso,  
* finalizar la creación del inventario en ambos casos.

### **Ejecución de análisis**

Cuando se ejecuta el análisis de un pasillo:

1. se resuelve el inventario asociado,  
2. se buscan sus referencias visuales,  
3. se construye el contexto de análisis,  
4. ese contexto se pasa a la estrategia/proveedor correspondiente,  
5. el proveedor decide cómo usar las referencias de acuerdo con sus capacidades.

---

## **8\. Arquitectura propuesta**

## **8.1 Idea central**

Las referencias visuales deben incorporarse al **AnalysisContext** o contrato equivalente usado por la capa de aplicación para invocar a cualquier proveedor.

La relación correcta es:

* el frontend carga referencias,  
* el backend las persiste,  
* la capa de aplicación las resuelve,  
* el contexto de análisis las contiene,  
* la estrategia/proveedor las consume según soporte.

No debe existir una dependencia del tipo:

* `if provider == gemini then attach references`

como lógica principal del sistema.

---

## **8.2 Capas afectadas**

### **Frontend**

* wizard de creación de inventario,  
* upload opcional de referencias,  
* preview/listado.

### **API / Application**

* endpoints para upload/listado,  
* caso de uso para resolver referencias del inventario,  
* composición del contexto de análisis.

### **Domain / Contracts**

* modelo de referencia visual,  
* extensión del contrato de análisis para incluir referencias.

### **Provider Strategies**

* adaptación de cada provider para consumir o no referencias.

### **Persistence / Storage**

* tabla de referencias por inventario,  
* almacenamiento de archivos,  
* metadata de referencias usadas por job.

---

## **9\. Modelo de dominio**

## **9.1 Entidad principal**

### **`InventoryVisualReference`**

Representa una imagen de referencia asociada a un inventario.

### **Campos mínimos sugeridos**

* `id`  
* `inventory_id`  
* `filename`  
* `storage_path`  
* `mime_type`  
* `file_size`  
* `created_at`

### **Campos opcionales**

* `width`  
* `height`  
* `created_by`  
* `is_deleted`  
* `display_name`  
* `checksum`

Para esta versión conviene mantenerlo simple.

---

## **10\. Persistencia**

## **10.1 Tabla propuesta**

### **`inventory_visual_references`**

Campos sugeridos:

* `id` UUID/string primary key  
* `inventory_id` UUID/string not null  
* `filename` text not null  
* `storage_path` text not null  
* `mime_type` text not null  
* `file_size` bigint not null  
* `created_at` timestamp not null

Índices sugeridos:

* índice por `inventory_id`

---

## **10.2 Convención de storage**

Path lógico sugerido:

`/inventories/{inventory_id}/visual_references/{reference_id}.{ext}`

### **Requisitos**

* no depender del nombre original como identificador;  
* separar display name de path real;  
* permitir futura migración de backend de storage;  
* validar extensión y mime type.

---

## **11\. Contrato de análisis multi-provider**

## **11.1 Problema a resolver**

Hoy el request hacia el proveedor puede estar centrado en:

* prompt,  
* imágenes principales,  
* configuraciones puntuales.

Con múltiples proveedores, eso ya no alcanza si queremos introducir referencias visuales sin acoplar a uno solo.

---

## **11.2 Extensión propuesta del contexto**

Debe existir una estructura común, por ejemplo:

### **`AnalysisContext`**

Con responsabilidades como:

* contener evidencia principal,  
* contener contexto visual opcional,  
* contener instrucciones de negocio,  
* contener metadata de ejecución.

Una forma conceptual:

```ts
type AnalysisContext = {
  primaryEvidence: AnalysisImage[]
  visualReferences?: VisualReferenceContext[]
  instructions: string[]
  metadata?: Record<string, unknown>
}
```

Y para referencias:

```ts
type VisualReferenceContext = {
  referenceId: string
  storagePath: string
  mimeType: string
  role: 'inventory_reference'
}
```

El punto importante no es el nombre exacto, sino que la referencia viva en el **contrato compartido** y no en una implementación concreta.

---

## **11.3 Capabilities del provider**

Cada estrategia debería poder expresar sus capacidades. Ejemplo conceptual:

```ts
type ProviderCapabilities = {
  supportsImageInputs: boolean
  supportsVisualReferenceContext: boolean
  supportsStructuredOutput: boolean
}
```

Entonces, la aplicación puede construir el contexto completo siempre, y el provider decidir:

* usar referencias como imágenes reales,  
* convertirlas en nota textual,  
* ignorarlas explícitamente.

---

## **12\. Comportamiento esperado por tipo de proveedor**

## **12.1 Proveedor multimodal con soporte completo**

Ejemplo: proveedor que acepta múltiples imágenes de contexto.

Comportamiento:

* recibe evidencia principal,  
* recibe referencias visuales,  
* recibe prompt enriquecido,  
* usa todo el material para responder.

## **12.2 Proveedor con soporte parcial**

Ejemplo: acepta imágenes pero no distingue bien roles.

Comportamiento:

* puede recibir referencias con instrucciones más explícitas,  
* o limitar cantidad,  
* o usar solo una parte del contexto.

## **12.3 Proveedor no multimodal**

Ejemplo: provider solo textual o estructurado.

Comportamiento:

* no recibe imágenes de referencia como input visual;  
* puede recibir solo metadata textual resumida;  
* o ignorar las referencias de forma explícita.

La aplicación no debe romperse en ninguno de estos casos.

---

## **13\. Strategy pattern: ajuste arquitectónico**

## **13.1 Situación deseada**

La selección del proveedor sigue ocurriendo mediante strategy pattern, pero ahora cada estrategia recibe un input común enriquecido.

Ejemplo conceptual:

```ts
interface AnalysisProviderStrategy {
  analyze(request: ProviderAnalysisRequest): Promise<ProviderAnalysisResult>
  getCapabilities(): ProviderCapabilities
}
```

Donde `ProviderAnalysisRequest` puede contener:

```ts
type ProviderAnalysisRequest = {
  analysisContext: AnalysisContext
  promptPayload: PromptPayload
}
```

La transformación de `AnalysisContext` al formato nativo del proveedor debe quedar **dentro de la estrategia**, no en el caso de uso.

---

## **13.2 Responsabilidades**

### **Application layer**

* resolver referencias visuales del inventario;  
* construir `AnalysisContext`;  
* persistir metadata de referencias resueltas;  
* invocar la estrategia correspondiente.

### **Provider strategy**

* leer el `AnalysisContext`;  
* decidir cómo mapear `visualReferences` a su formato nativo;  
* reportar si realmente las usó o no.

---

## **14\. Builder de contexto**

## **14.1 Necesidad**

Para no ensuciar el caso de uso ni los routes, conviene encapsular la construcción del contexto en un componente dedicado.

### **Propuesta**

`AnalysisContextBuilder`

Responsabilidades:

* recibir evidencia principal,  
* resolver referencias del inventario,  
* aplicar límites,  
* producir una estructura uniforme.

Ejemplo conceptual:

```ts
interface AnalysisContextBuilder {
  buildForAisleAnalysis(params: BuildAnalysisContextParams): Promise<AnalysisContext>
}
```

---

## **15\. Prompt enrichment provider-agnostic**

## **15.1 Principio**

El sistema no debe tener un “prompt Gemini de referencias”, sino una **instrucción de negocio común** que luego cada proveedor incorporará según corresponda.

Ejemplo de instrucción común:

Se incluyen imágenes de referencia del inventario que muestran ejemplos válidos del estándar visual esperado, como etiquetas o pallets típicos. Deben utilizarse como contexto comparativo para reconocer mejor formatos y elementos relevantes, sin reemplazar la evidencia primaria del pasillo analizado.

Esto puede formar parte de:

* un `PromptPayload`,  
* un `analysis instruction set`,  
* o un builder de instrucciones común.

---

## **15.2 Responsabilidad del provider**

Cada estrategia debe adaptar esta instrucción al formato del proveedor:

* mensaje system,  
* bloque instruction,  
* metadata,  
* structured input,  
* etc.

---

## **16\. Contrato de resultados del provider**

## **16.1 Necesidad**

No alcanza con registrar que el job tenía referencias disponibles. También conviene saber si el provider realmente las consumió.

### **Campo conceptual sugerido en el resultado del provider**

```ts
type ProviderAnalysisResult = {
  rawOutput: unknown
  parsedOutput: unknown
  providerMetadata?: {
    visualReferencesAvailable?: boolean
    visualReferencesConsumed?: boolean
    visualReferenceCount?: number
  }
}
```

Esto mejora trazabilidad y debugging.

---

## **17\. Metadata de job**

## **17.1 Recomendación**

Persistir dos niveles de información:

### **A. Referencias resueltas por la aplicación**

Qué referencias encontró y puso en el contexto.

### **B. Referencias consumidas por el provider**

Qué informó la estrategia/proveedor sobre su uso efectivo.

Ejemplo conceptual:

```json
{
  "visual_reference_context": {
    "resolved": true,
    "reference_ids": ["ref_1", "ref_2"],
    "resolved_count": 2,
    "provider_consumed": true
  }
}
```

Esto evita ambigüedad entre “existían” y “fueron usadas realmente”.

---

Responsabilidades:

* validar inventario,  
* validar archivos,  
* guardar en storage,  
* crear registros.

## **18.2 Listado de referencias**

### **`GET /api/v3/inventories/{inventory_id}/visual-references`**

Responsabilidades:

* listar referencias activas del inventario.

## **18.3 Creación de inventario**

El flujo recomendado sigue siendo:

1. crear inventario,  
2. subir referencias si el usuario eligió hacerlo.

Esto simplifica validaciones y rollback.

## **19\. Frontend design**

## **19.1 Flujo de creación**

### **Paso 1**

Datos del inventario.

### **Paso 2**

“Imágenes de referencia (opcional)”.

Requisitos:

* upload múltiple,  
* preview,  
* validación básica,  
* omitir sin fricción.

## **19.2 Mensaje UX**

Debe explicar que las referencias:

* son opcionales,  
* mejoran el contexto del análisis,  
* sirven como ejemplos visuales del estándar esperado.

---

## **20\. Validaciones**

## **20.1 Validaciones de negocio**

* máximo 3 referencias por inventario en esta versión;  
* permitir inventarios sin referencias;  
* asociar cada referencia a un único inventario.

## **20.2 Validaciones técnicas**

* formatos permitidos: jpg, jpeg, png, webp;  
* tamaño máximo por archivo;  
* inventario existente;  
* no exceder límite total.

---

## **21\. Manejo de errores**

## **21.1 Upload**

* archivo inválido,  
* exceso de tamaño,  
* storage failure,  
* error parcial en batch.

## **21.2 Resolución en análisis**

Si la aplicación esperaba referencias pero no puede resolverlas:

* registrar warning,  
* continuar sin referencias o fallar según política.

Para esta versión, conviene definir una política explícita. Mi recomendación:

* si el análisis principal puede seguir, continuar sin referencias y registrar warning.

## **21.3 Provider compatibility**

Si el provider seleccionado no soporta referencias visuales:

* no fallar;  
* degradar de forma explícita;  
* registrar `provider_consumed = false`.

---

## **22\. Riesgos**

### **22.1 Acoplamiento al provider dominante**

Riesgo de diseñar la feature alrededor de Gemini.

**Mitigación:** referencias como parte del `AnalysisContext`.

### **22.2 Ambigüedad entre referencias y evidencia real**

Riesgo de que el provider mezcle roles.

**Mitigación:** roles explícitos en contexto e instrucciones claras.

### **22.3 Sobrecarga del análisis**

Más imágenes pueden aumentar latencia/costo.

**Mitigación:** límites estrictos de cantidad y tamaño.

### **22.4 Pérdida de trazabilidad**

Riesgo de no saber si las referencias fueron realmente usadas.

**Mitigación:** metadata diferenciada entre resueltas y consumidas.

### **22.5 Divergencia entre proveedores**

Cada provider podría reaccionar distinto ante el mismo contexto.

**Mitigación:** contrato común \+ capabilities \+ adaptación por estrategia.

---

## **23\. Criterios de aceptación**

* el usuario puede crear un inventario sin referencias;  
* el usuario puede cargar referencias en un paso opcional;  
* las referencias quedan asociadas correctamente al inventario;  
* el backend puede resolver referencias del inventario durante el análisis;  
* las referencias forman parte del contexto común de análisis, no de una implementación específica;  
* al menos un provider multimodal puede consumirlas como imágenes de contexto;  
* un provider que no soporte referencias no rompe el flujo;  
* el prompt/instruction set se enriquece de forma provider-agnostic;  
* el job registra qué referencias fueron resueltas;  
* el resultado permite saber si el provider las consumió o no.

## **25\. Decisiones abiertas**

Antes de implementar conviene cerrar:

1. nombre exacto del contrato compartido (`AnalysisContext`, `ProviderAnalysisRequest`, etc.);  
2. política cuando hay referencias pero el provider no las soporta;  
3. política cuando la resolución de referencias falla;  
4. máximo exacto de imágenes y tamaño;  
5. si el resultado del provider reportará uso efectivo de referencias de forma obligatoria o opcional;  
6. si la metadata del job se guarda en JSON o esquema relacional adicional.

---

## **26\. Conclusión**

La v3.2.4 debe implementarse como una **feature de contexto visual por inventario**, opcional y compatible con una arquitectura multi-provider basada en strategy pattern.

El punto arquitectónico más importante es este:  
**las referencias visuales deben vivir en el contrato de análisis común y no en una implementación específica del proveedor**.

De esa manera:

* Gemini podrá consumirlas como contexto multimodal,  
* otros providers compatibles podrán hacer lo mismo,  
* providers no compatibles podrán degradar el comportamiento sin romper el sistema,  
* y la aplicación mantendrá una arquitectura limpia, extensible y consistente con el enfoque actual de strategies.
