# Dinamic Inventory v3.2.2 — Qty Flow Audit (as-built)

## 1) Where does `qty` first enter the system?

- **Hybrid pipeline (v2.1) provider output**: the LLM structured output includes `product_label_quantity` (and downstream derived `final_quantity`).
  - Parsing entrypoint: `backend/src/parsing/global_analysis_parser.py` (`parse_entities` sets `Entity.product_label_quantity`).
  - Validation entrypoint: `backend/src/validation/global_analysis_schema.py` (`validate_global_analysis_structure_v21` checks `product_label_quantity` is int-or-null-ish).

## 2) Which provider / parser / adapter handles it?

- **Parser**: `backend/src/parsing/global_analysis_parser.py`
  - Previously: coerced `product_label_quantity` via `_safe_int` and could collapse absent/null/invalid/zero into `None` or `0` without preserving state.
- **Pipeline decision**: `backend/src/decision/count_status.py`
  - Produces `Entity.count_status` and `Entity.final_quantity` (may be `None` for NEEDS_REVIEW).
- **Report output**: `backend/src/reporting/hybrid_report.py`
  - Serializes both `product_label_quantity` and `final_quantity` per entity.

## 3) Where could `qty` become missing/null/invalid/zero/unresolved (pre v3.2.2)?

- **Parser coercion**: missing vs null vs invalid were not explicitly represented downstream (only `None` or `int`).
- **Count status assignment**: `backend/src/decision/count_status.py` sets `final_quantity=None` in multiple “accepted presence but unresolved qty” states (e.g. some `NEEDS_REVIEW` paths).
- **v3 persistence mapping**: `backend/src/infrastructure/pipeline/v3_report_mapper.py` previously mapped missing/null/invalid qty to `detected_quantity=0`, which leaks an **unjustified 0** into persisted `product_records.detected_quantity`.
- **v3 API summary mapping**: `backend/src/api/routes/v3/shared.py` previously derived `detected_quantity` from `final_quantity` / `product_label_quantity` and returned `0` when unresolved, with no provenance.

## 4) Where is the final accepted entity produced?

- **Pipeline entity (v2.1)**: `src/domain/entity.py` instances are the “accepted” unit after:
  - `backend/src/pipeline/stages/entity_resolution_stage.py` → `assign_count_status` → `compute_entity_quality_score`.

## 5) Where is the final DTO/API response built?

- v3 endpoints:
  - `backend/src/api/routes/v3/positions.py` → calls
  - `backend/src/api/routes/v3/shared.py` → `position_to_summary` (DTO mapping).

## 6) Where is persistence performed?

- v3 persistence use case:
  - `backend/src/application/use_cases/persist_aisle_result.py` calls
  - `backend/src/infrastructure/pipeline/v3_report_mapper.py` (`map_hybrid_report_to_domain`) then saves via repositories.
- SQL repositories:
  - `backend/src/infrastructure/repositories/sql_position_repository.py`
  - `backend/src/infrastructure/repositories/sql_product_record_repository.py`

## 7) Best single decision point for final qty resolution (v3.2.2)

**As implemented for v3.2.2:**

- **Rule definition (single source of truth)**: `backend/src/domain/quantity/resolution.py`
- **Authoritative resolution moment**:
  - **Primary**: during v3 persistence mapping in `backend/src/infrastructure/pipeline/v3_report_mapper.py` (writes `qty_final`, `qty_source`, `qty_inference_reason`, `raw_qty`, `qty_parse_status` into persisted artifacts).
  - **Secondary / backward compatible**: v3 API mapping `backend/src/api/routes/v3/shared.py` reuses the same resolver for legacy rows missing the new persisted fields, ensuring the API cannot leak unjustified `null`/`0` when the position is considered product-present and has evidence.
