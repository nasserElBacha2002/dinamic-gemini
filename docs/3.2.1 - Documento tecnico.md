# **Dinamic Inventory v3.2.1 — Technical Design Document**

## **1\. Document Purpose**

This document defines the **technical design** for release **Dinamic Inventory v3.2.1 — Minimal Administrative Authentication**.

Its purpose is to translate the release requirements into an implementation-oriented specification covering:

* architecture decisions,  
* backend design,  
* frontend design,  
* API contracts,  
* configuration requirements,  
* security constraints,  
* testing strategy,  
* rollout considerations.

This document is intentionally limited to the **single-admin authentication model** required by v3.2.1.

---

## **2\. Technical Goals**

The technical goals of v3.2.1 are:

1. Introduce a minimal authentication boundary for the active v3 system.  
2. Protect all operational backend routes required by the application.  
3. Require login before the frontend can access protected flows.  
4. Keep the implementation small, explicit, and easy to extend.  
5. Avoid introducing unnecessary complexity such as multi-user data models or role systems.  
6. Prepare the system for deployment to a shared dev environment.

---

## **3\. High-Level Solution Overview**

The recommended solution for v3.2.1 is a **token-based authentication flow with a single administrator identity**.

### **Proposed model**

* The backend exposes a login endpoint.  
* The admin submits username and password.  
* The backend validates the password against a stored hash.  
* If valid, the backend issues a signed access token.  
* The frontend stores the authenticated session state.  
* The frontend sends the token with protected API requests.  
* Backend v3 routes validate the token before executing protected operations.

### **Why this model**

This approach is appropriate because it is:

* simple,  
* stateless at the API level,  
* easy to integrate with the current frontend/backend split,  
* compatible with a future dev deployment,  
* extensible later if multiple users are introduced.

---

## **4\. Proposed Authentication Architecture**

## **4.1 Authentication Components**

The auth solution should be split into the following technical components.

### **Backend**

* **Auth config loader**  
* **Password hash verifier**  
* **Token service**  
* **Auth schemas / DTOs**  
* **Auth route(s)**  
* **Current admin resolver**  
* **Route protection dependency or middleware**

### **Frontend**

* **Auth API client methods**  
* **Auth state store/provider**  
* **Login page**  
* **Protected route wrapper / guard**  
* **Unauthorized response handling**  
* **Logout action**

---

## **4.2 Identity Model**

This version should use a **single fixed administrator identity**.

A minimal internal identity model is sufficient. Example conceptual structure:

```ts
{
  id: "admin",
  username: string,
  role: "administrator"
}
```

A persistent user table is **not required** for v3.2.1 unless implementation convenience strongly justifies it.

The recommended default is:

* credentials stored in configuration,  
* authenticated principal resolved from configuration \+ token claims.

---

## **4.3 Auth Boundary Design Principle**

Authentication must be implemented as a **horizontal infrastructure concern**, not as ad hoc checks scattered across business routes.

This means:

* do not manually validate tokens inside each endpoint body,  
* use a shared protection mechanism,  
* centralize current-user resolution,  
* make unauthorized behavior consistent across the API surface.

---

## **5\. Backend Technical Design**

## **5.1 Backend Responsibilities**

The backend must provide:

1. Admin credential validation.  
2. Token issuance.  
3. Token validation.  
4. Route protection.  
5. Standard unauthorized responses.  
6. Safe configuration handling.

---

## **5.2 Suggested Backend Structure**

The exact folder layout can follow current project conventions, but conceptually the backend should gain an auth area similar to:

```
backend/
  src/
    auth/
      config.py
      schemas.py
      service.py
      security.py
      dependencies.py
      routes.py
```

### **Suggested responsibilities**

#### **`config.py`**

* load admin username,  
* load password hash,  
* load token secret,  
* load token expiration config.

#### **`schemas.py`**

* login request model,  
* login response model,  
* current-user response if `/auth/me` is implemented,  
* auth error response if standardized.

#### **`service.py`**

* credential validation use case,  
* creation of auth payload / principal,  
* orchestration of login flow.

#### **`security.py`**

* password hash verification,  
* token creation,  
* token decoding/validation,  
* token expiration rules.

#### **`dependencies.py`**

* reusable dependency for protected routes,  
* current admin resolution,  
* unauthorized error generation.

#### **`routes.py`**

* login endpoint,  
* optional auth verification endpoint.

---

## **5.3 Login Flow**

### **Request**

The frontend sends username and password to the backend login route.

### **Validation steps**

1. Validate request payload shape.  
2. Compare submitted username against configured admin username.  
3. Verify submitted password against configured password hash.  
4. If valid, generate signed access token.  
5. Return token and minimal principal metadata.

### **Failure behavior**

If credentials are invalid:

* return unauthorized response,  
* do not reveal whether username or password failed independently,  
* do not log the submitted password.

---

## **5.4 Token Design**

### **Recommended token type**

A signed bearer token is recommended.

JWT is acceptable for this release if implemented with:

* explicit expiration,  
* secret from configuration,  
* minimal claims,  
* signature validation on each protected request.

### **Suggested claims**

The token should remain minimal. For example:

* subject / principal id,  
* username,  
* role,  
* issued-at,  
* expiration.

Avoid unnecessary claims.

### **Token lifetime**

Use a reasonable dev-friendly lifetime, but not an unlimited token.

A practical starting point is a configurable expiration (for example several hours), with the exact value controlled by environment settings.

For v3.2.1, a refresh-token system is **not required**.

---

## **5.5 Route Protection Strategy**

### **Recommendation**

Use a reusable protection dependency (or framework-equivalent middleware mechanism) applied consistently to all protected v3 routes.

### **Protection target**

All operational v3 routes should require a valid admin token except explicitly public routes such as:

* healthcheck,  
* login,  
* optional auth verification.

### **Expected behavior**

If no token is present:

* return unauthorized.

If token is invalid:

* return unauthorized.

If token is expired:

* return unauthorized with a stable error contract.

---

## **5.6 Protected Route Coverage**

At minimum, the following v3 route groups should be protected:

* inventory creation and management,  
* asset upload,  
* job creation and job status inspection,  
* result retrieval,  
* review endpoints,  
* any active v3 operational route used by frontend flows.

A technical audit should confirm the final protected route surface.

---

## **5.7 Backend Error Contract**

Unauthorized/auth-related responses should be consistent.

Minimum cases to standardize:

* invalid login credentials,  
* missing token,  
* invalid token,  
* expired token.

Suggested response properties:

* machine-readable error code,  
* human-readable message,  
* appropriate HTTP status.

Do not leak sensitive internal details.

---

## **5.8 Password Security Requirements**

### **Mandatory requirements**

* Password must never be stored in plaintext.  
* Password must be verified against a secure hash.  
* Submitted passwords must never be logged.  
* Hash verification must be performed through a recognized password hashing approach.

### **Recommended hashing strategy**

Use a modern password hashing algorithm suitable for application authentication.

The exact library may depend on current backend stack, but the implementation should follow these rules:

* strong hash algorithm,  
* salt included as part of the stored hash format,  
* verification function provided by the hashing library.

---

## **5.9 Configuration Requirements**

The backend must receive authentication settings from runtime configuration.

### **Minimum required config**

* `ADMIN_USERNAME`  
* `ADMIN_PASSWORD_HASH`  
* `AUTH_TOKEN_SECRET`  
* `AUTH_TOKEN_EXPIRES_MINUTES` or equivalent

### **Additional recommended config**

* token issuer / audience if relevant,  
* auth enabled flag only if truly needed for controlled local development,  
* environment name.

### **Important rule**

Secrets must not be hardcoded inside route code or committed in plain text config intended for shared use.

---

## **5.10 Optional `/auth/me` Endpoint**

This endpoint is optional but recommended.

### **Purpose**

Allow the frontend to validate or hydrate the current authenticated identity on app boot.

### **Behavior**

* requires valid token,  
* returns minimal admin principal information.

This can simplify frontend session restoration and route guards.

---

## **6\. Frontend Technical Design**

## **6.1 Frontend Responsibilities**

The frontend must:

* present a login screen,  
* call the backend login endpoint,  
* store authenticated session state,  
* attach bearer token to protected requests,  
* block protected routes before login,  
* clear session on logout,  
* handle unauthorized responses gracefully.

---

## **6.2 Suggested Frontend Structure**

The exact layout depends on current frontend organization, but conceptually the auth feature should introduce:

```
frontend/
  src/
    features/auth/
      api.ts
      types.ts
      store.ts
      AuthProvider.tsx
      ProtectedRoute.tsx
      LoginPage.tsx
```

### **Suggested responsibilities**

#### **`api.ts`**

* login request,  
* optional current-user request.

#### **`types.ts`**

* login request type,  
* login response type,  
* auth principal type.

#### **`store.ts`**

* auth state,  
* token storage integration,  
* login/logout actions,  
* session initialization.

#### **`AuthProvider.tsx`**

* optional app-wide auth context/provider.

#### **`ProtectedRoute.tsx`**

* checks auth state,  
* redirects to login if missing.

#### **`LoginPage.tsx`**

* renders login form,  
* submits credentials,  
* handles loading and error states.

---

## **6.3 Login UX Requirements**

The login page should be intentionally minimal.

### **Required elements**

* username input,  
* password input,  
* submit button,  
* loading state,  
* invalid credentials error state.

### **UX behavior**

* submitting valid credentials transitions to the protected app,  
* invalid login displays a clear message,  
* repeated submission during loading should be prevented,  
* empty credentials should be validated before request.

---

## **6.4 Frontend Session Storage**

The frontend must store enough information to preserve authenticated state between page navigation and refresh where desired.

### **Practical recommendation**

Persist the access token in a controlled client-side storage mechanism compatible with the current app architecture.

Because this is a dev-oriented internal release, a pragmatic approach is acceptable, but it should still be implemented carefully.

### **Requirements**

* token retrievable on app initialization,  
* token removable on logout,  
* state synchronized with runtime auth store.

---

## **6.5 Protected Route Strategy**

Protected app sections must not render unless the auth state is valid.

### **Recommended behavior**

* unauthenticated user navigating to a protected route is redirected to login,  
* authenticated user can access the route,  
* invalid/expired token response from API should clear or invalidate local auth state and redirect appropriately.

---

## **6.6 API Client Integration**

The frontend API client layer must be extended so protected requests automatically include the token.

### **Requirements**

* attach `Authorization: Bearer <token>` header,  
* do so consistently across protected calls,  
* support centralized handling of unauthorized responses.

If the project already has a shared API client wrapper, auth should be integrated there instead of duplicating logic per request.

---

## **6.7 Logout Flow**

Logout must:

* clear token from storage,  
* clear current auth state,  
* redirect the user away from protected routes,  
* prevent further authenticated API use until login occurs again.

A backend logout endpoint is **not required** for this stateless version unless specifically desired for consistency.

---

## **7\. API Contract Proposal**

The final contract can be adapted to project conventions, but the following is a strong recommended baseline.

## **7.1 `POST /auth/login`**

### **Request body**

```json
{
  "username": "admin",
  "password": "<plaintext submitted by user>"
}
```

### **Success response**

```json
{
  "access_token": "<token>",
  "token_type": "bearer",
  "expires_in": 28800,
  "user": {
    "id": "admin",
    "username": "admin",
    "role": "administrator"
  }
}
```

### **Invalid credentials response**

```json
{
  "error": {
    "code": "INVALID_CREDENTIALS",
    "message": "Invalid credentials."
  }
}
```

---

## **7.2 `GET /auth/me` (optional but recommended)**

### **Request**

Requires bearer token.

### **Success response**

```json
{
  "id": "admin",
  "username": "admin",
  "role": "administrator"
}
```

### **Unauthorized response**

```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Authentication required."
  }
}
```

---

## **7.3 Unauthorized Response for Protected Routes**

### **Missing/invalid/expired token**

```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Authentication required."
  }
}
```

```

The exact message can vary, but the contract should remain consistent.

## 8. Security Design Notes

## 8.1 Security Level of This Release

This release provides **basic application access control**, not full enterprise security.

Its goal is to safely support:
- internal use,
- shared dev environment access,
- controlled testing.

It is not the final production security model.

---

## 8.2 Security Constraints

Mandatory security constraints:
- no plaintext password storage,
- no secret values committed in source code intended for shared environments,
- no credential logging,
- no open access to operational v3 routes,
- no frontend-only protection without backend enforcement.

---

## 8.3 Secret Management

For dev deployment, secrets may still come from environment variables or deployment-specific secure config.

The implementation should be compatible with future stronger secret management without major refactor.

---

## 8.4 Token Exposure Considerations

The frontend must minimize accidental token leakage.

At minimum:
- do not print token in logs,
- do not expose token in visible UI,
- clear token on logout,
- handle unauthorized responses centrally.

---

## 9. Migration and Integration Strategy

## 9.1 Integration Principle

Auth must be introduced with minimal disruption to the existing v3 business flows.

### Recommended sequence
1. implement backend auth primitives,
2. implement login endpoint,
3. protect a small subset of routes in development,
4. integrate frontend login and token storage,
5. protect the full active v3 route surface,
6. test end-to-end.

---

## 9.2 Backward Compatibility Note

Because v3.2.1 introduces a new access boundary, any existing local/manual API usage must be updated to include authentication when hitting protected routes.

This is an intentional breaking change at the access layer.

---

## 10. Testing Strategy

## 10.1 Backend Unit Tests

Should validate:
- correct password verification,
- invalid password rejection,
- token generation,
- token decoding and validation,
- token expiration behavior if enabled.

---

## 10.2 Backend API Tests

Should validate:
- successful login request,
- invalid login request,
- missing token on protected route,
- valid token on protected route,
- invalid token on protected route,
- expired token on protected route if expiration enforced in tests.

---

## 10.3 Frontend Tests

Should validate:
- login form behavior,
- successful auth state update,
- invalid credentials UI,
- protected route redirect,
- logout behavior,
- unauthorized API response handling.

---

## 10.4 Manual End-to-End Validation

A manual test pass must confirm the following scenario:

1. open app while unauthenticated,
2. observe login page,
3. submit valid admin credentials,
4. enter protected app,
5. create/use normal workflow,
6. refresh page and confirm expected session behavior,
7. logout,
8. confirm protected access is blocked again.

---

## 11. Rollout Considerations

## 11.1 Local Development

During implementation, developers may need a bootstrap process to generate the password hash for the configured admin user.

A small helper script or documented command is recommended.

---

## 11.2 Dev Environment Readiness

v3.2.1 should leave the system compatible with a later remote dev deployment by ensuring:
- secrets come from configuration,
- auth is enforced in backend,
- frontend uses runtime-compatible login behavior,
- no localhost-only auth assumptions exist.

---

## 12. Risks

### 12.1 Technical Risks
- partial route protection,
- inconsistent frontend token handling,
- brittle auth integration into API client,
- accidental credential/secret leakage in logs or config,
- token expiration creating poor UX if not handled properly.

### 12.2 Design Risks
- overengineering a simple release,
- introducing a user model larger than necessary,
- scattering auth checks across the application.

### 12.3 Mitigation
- keep auth layer small and centralized,
- standardize protected route handling,
- standardize API client auth behavior,
- test the complete protected surface,
- document config clearly.

---

## 13. Acceptance Checklist

The technical implementation is considered complete when:

- [ ] backend exposes login endpoint,
- [ ] backend validates password hash securely,
- [ ] backend issues signed access token,
- [ ] backend protects active v3 operational routes,
- [ ] frontend provides login page,
- [ ] frontend stores and applies auth token correctly,
- [ ] frontend blocks protected routes when unauthenticated,
- [ ] logout clears access correctly,
- [ ] auth config is externalized,
- [ ] no plaintext credentials are used in runtime logic,
- [ ] manual E2E validation succeeds.

---

## 14. Final Technical Summary

**Dinamic Inventory v3.2.1** should be implemented as a **minimal, centralized, token-based single-admin authentication layer**.

The design must remain intentionally narrow:
- one administrator,
- one login flow,
- protected v3 routes,
- frontend session handling,
- safe configuration.

This approach is the correct balance for the current stage of the product because it:
- solves the immediate security problem,
- is simple to deploy in dev,
- avoids premature complexity,
- creates a clean base for later hardening in future releases.
```
