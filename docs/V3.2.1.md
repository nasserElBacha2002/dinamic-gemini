# **Dinamic Inventory v3.2.1 — Release Documentation**

## **1\. Release Name**

**v3.2.1 — Minimal Administrative Authentication**

---

## **2\. Purpose of the Release**

The purpose of release **3.2.1** is to introduce a **minimal but robust authentication layer** into Dinamic Inventory so the platform is no longer openly accessible.

At this stage, the goal is **not** to build a full user management system. The goal is to establish a **single-administrator access model** that protects the active v3 application flows and creates a safe base for future releases.

This release is the first security-oriented step in the v3.2 roadmap and is intended to:

* protect access to the frontend application,  
* protect the backend v3 API surface,  
* ensure that only an authorized administrator can create inventories, upload evidence, run jobs, and review results,  
* prepare the system for remote dev deployment without leaving the platform exposed.

---

## **3\. Release Context**

Before v3.2.1, the system is functionally usable but does not yet enforce a controlled access boundary. This is acceptable in early local development, but becomes a risk once the application is:

* shared across environments,  
* exposed outside localhost,  
* used by collaborators,  
* connected to a persistent backend and remote storage,  
* prepared for a dev or staging deployment.

Because of that, **authentication must be introduced before the system is deployed to a shared environment**.

This release intentionally focuses on the **smallest secure model that solves the current product need**:

* one admin user,  
* one login flow,  
* protected routes,  
* stable frontend session handling.

---

## **4\. Primary Objective**

Implement a **single-admin authentication system** that prevents unauthenticated access to the Dinamic Inventory v3 application and API.

---

## **5\. Secondary Objectives**

* Establish a clean authentication boundary in backend v3.  
* Introduce session/token handling in the frontend.  
* Protect the main business actions of the system.  
* Keep the implementation intentionally small and maintainable.  
* Avoid overengineering roles, multi-user support, or identity administration before they are needed.

---

## **6\. Functional Scope**

This release includes the following functional capabilities:

### **6.1 Login**

* A login screen must exist in the frontend.  
* The user must provide administrator credentials.  
* Valid credentials must grant access to the application.  
* Invalid credentials must be rejected with a clear error message.

### **6.2 Authenticated Session**

* After a successful login, the frontend must keep the authenticated state.  
* The authentication state must be available across protected screens.  
* The frontend must be able to attach the access credential to backend requests.

### **6.3 Backend Route Protection**

* Active v3 backend routes must require authentication, except for explicitly public endpoints.  
* Unauthenticated requests to protected routes must fail with the appropriate response.

### **6.4 Logout**

* The frontend must support logout.  
* Logout must clear the local authenticated state and block further use of protected routes until login occurs again.

### **6.5 Single Administrator Model**

* The system must support one administrator account only.  
* Credentials must be defined through secure configuration, not through an open registration flow.

---

## **7\. Non-Goals**

The following items are explicitly out of scope for v3.2.1:

* multi-user support,  
* role-based access control,  
* organization/workspace separation,  
* signup or public registration,  
* password reset flow,  
* email verification,  
* OAuth / SSO integration,  
* invite-based user management,  
* advanced audit of login history,  
* enterprise-grade IAM design.

This release must remain minimal.

---

## **8\. Product Requirements**

### **8.1 Login UX**

* The application must present a dedicated login screen before the user can access the protected interface.  
* The login UI must be simple, clear, and aligned with the current frontend structure.  
* The user must not be able to bypass the login screen by navigating manually in the frontend.

### **8.2 Protected Application Access**

The following frontend flows must require authentication:

* inventory creation,  
* asset upload,  
* job execution,  
* result visualization,  
* review actions,  
* historical job/result access.

### **8.3 Protected API Access**

The following backend categories must require authentication unless specifically exempted:

* inventory endpoints,  
* job endpoints,  
* asset upload endpoints,  
* result endpoints,  
* review endpoints,  
* any future active v3 operational routes.

### **8.4 Public Endpoints**

A minimal set of routes may remain public if operationally necessary, for example:

* healthcheck,  
* login endpoint,  
* optionally a simple auth verification endpoint.

---

## **9\. Technical Requirements**

### **9.1 Authentication Strategy**

The recommended strategy for v3.2.1 is:

* backend-issued token-based authentication,  
* password stored as a hash,  
* frontend stores authenticated session state,  
* backend validates the credential on each protected request.

A lightweight JWT-based flow is acceptable if implemented carefully.

### **9.2 Credential Source**

The administrator credentials must not be hardcoded directly in business logic.

Recommended options:

* environment variables,  
* secure configuration file not committed with secrets,  
* bootstrap configuration loaded at runtime.

The password must be stored as a **hash**, not as plaintext.

### **9.3 Password Handling**

* Password comparison must be done against a hash.  
* Plaintext password persistence is not allowed.  
* The implementation must avoid exposing credential details in logs.

### **9.4 Auth Boundary in Backend**

The backend must provide:

* a login endpoint,  
* a reusable auth validation mechanism,  
* a reusable protection mechanism for all active v3 routes.

This boundary should be designed so it can be extended later to support more users or roles without rewriting the entire backend.

### **9.5 Frontend Session Handling**

The frontend must:

* keep authentication state in a predictable place,  
* redirect unauthenticated users to login,  
* attach auth credentials to protected API requests,  
* clear auth state on logout,  
* handle expired/invalid session responses gracefully.

---

## **10\. Proposed Architecture Decisions**

### **10.1 Minimal Auth Domain**

A small auth layer should be introduced with clear separation of concerns:

* credential validation,  
* token/session creation,  
* current-user resolution,  
* route protection.

### **10.2 Single Admin Identity**

For this release, the authenticated identity can be modeled as a fixed administrator principal, for example:

* `id = admin`  
* `role = administrator`

The system does not need full user records yet unless implementation convenience requires it.

### **10.3 Future Extensibility**

Even though only one user exists today, the implementation should avoid making future multi-user support unnecessarily difficult.

That means:

* avoid spreading auth checks across arbitrary business code,  
* centralize auth validation,  
* keep route protection reusable,  
* keep token claims structured and explicit.

---

## **11\. Suggested Backend Deliverables**

### **11.1 Auth Module / Package**

Introduce a dedicated auth area in backend code for responsibilities such as:

* login request/response handling,  
* password verification,  
* token generation and validation,  
* auth dependency/middleware,  
* current admin resolution.

### **11.2 Protected Route Integration**

Integrate auth into the active v3 route surface so that protected routes uniformly require a valid admin session.

### **11.3 Response Contract**

Define clear response shapes for:

* successful login,  
* invalid credentials,  
* unauthorized access,  
* invalid or expired token.

### **11.4 Security Defaults**

* do not log credentials,  
* do not expose internal validation details,  
* do not leak whether username or password was incorrect separately if that creates unnecessary exposure.

---

## **12\. Suggested Frontend Deliverables**

### **12.1 Login Screen**

A dedicated screen or route with:

* username field,  
* password field,  
* submit action,  
* loading state,  
* error state.

### **12.2 Auth State Management**

A small auth state layer capable of:

* storing current auth status,  
* storing token/session payload if needed,  
* restoring session where appropriate,  
* reacting to unauthorized API responses.

### **12.3 Route Protection**

Protected app routes must redirect to login if the user is not authenticated.

### **12.4 Logout Flow**

A clear logout action must exist and remove access to the protected interface immediately.

---

## **13\. API Surface Expectations**

The final exact route names may vary, but the release should provide an API surface conceptually equivalent to:

* `POST /auth/login`  
* `GET /auth/me` or equivalent optional validation endpoint  
* protected v3 application routes requiring a valid auth credential

Protected routes should return a consistent unauthorized response when access is missing or invalid.

---

## **14\. Data and Configuration Requirements**

### **14.1 Required Configuration**

The system must define secure runtime configuration for at least:

* admin username,  
* admin password hash,  
* token secret or signing configuration,  
* token/session lifetime if applicable.

### **14.2 Environment Separation**

The configuration must support later differentiation between:

* local,  
* dev/staging,  
* production.

Even if only local/dev are used immediately, the structure should not assume a single environment forever.

---

## **15\. Acceptance Criteria**

The release is considered complete when all of the following are true:

1. The frontend shows a login boundary before protected usage.  
2. A valid administrator login grants access.  
3. Invalid credentials are rejected correctly.  
4. Protected backend v3 endpoints reject unauthenticated requests.  
5. Protected frontend routes cannot be used without an authenticated state.  
6. Logout removes access correctly.  
7. Administrator credentials are not stored in plaintext.  
8. Healthcheck and explicitly public routes continue to work.  
9. The auth implementation does not break the existing v3 operational flows after login.  
10. Basic manual validation confirms end-to-end login → use app → logout behavior.

---

## **16\. Testing Requirements**

### **16.1 Backend Tests**

At minimum, backend tests should cover:

* valid login,  
* invalid login,  
* access to protected route without token/session,  
* access to protected route with valid token/session,  
* invalid/expired auth behavior if expiration is implemented.

### **16.2 Frontend Tests**

At minimum, frontend validation should cover:

* login form submission,  
* error handling on invalid credentials,  
* protected route redirect,  
* authenticated access after login,  
* logout behavior.

### **16.3 Manual E2E Validation**

Manual testing should confirm:

* user cannot operate the app before login,  
* user can log in and use the main v3 workflows,  
* logout blocks further use,  
* browser refresh/session restoration works as intended.

---

## **17\. Risks and Implementation Notes**

### **17.1 Main Risks**

* Incomplete route protection leaving sensitive endpoints exposed.  
* Storing credentials unsafely.  
* Introducing auth in a way that tightly couples frontend and backend to a fragile implementation.  
* Breaking current API flows by injecting auth inconsistently.  
* Forgetting to handle unauthorized states in frontend API calls.

### **17.2 Mitigation Approach**

* Protect routes centrally, not ad hoc.  
* Keep secrets in configuration, not source code.  
* Use a small dedicated auth layer.  
* Validate auth across the complete active v3 route surface.  
* Add explicit unauthorized handling in frontend.

---

## **18\. Dependencies**

This release has low technical dependency on the rest of the v3.2 roadmap and is intentionally designed to go first.

It is especially important as a prerequisite for:

* remote dev deployment,  
* shared testing environments,  
* future operational hardening.

---

## **19\. Relationship with Later Releases**

This release creates the foundation for later v3.2 stages:

* **3.2.2** will improve quantity inference logic,  
* **3.2.3** will normalize duplicate labels before final counting,  
* **3.2.4** will introduce reference label images,  
* **3.2.5+** will further harden the v3 pipeline and review flows.

Without 3.2.1, those later improvements would still run on an application that is not safely protected for shared environments.

---

## **20\. Final Summary**

**Dinamic Inventory v3.2.1** is a focused release that adds the first real access control boundary to the platform.

It does not attempt to solve full user management. Instead, it introduces a minimal administrative authentication model that is:

* secure enough for the current stage,  
* small enough to implement quickly,  
* clean enough to extend later,  
* necessary before deploying the system to a shared dev environment.

This makes v3.2.1 the correct first milestone of the v3.2 roadmap.
