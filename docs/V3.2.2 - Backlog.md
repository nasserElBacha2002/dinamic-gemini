# **Dinamic Inventory v3.2.2 — Technical Release Document**

## **1\. Release Overview**

### **1.1 Release ID**

**v3.2.2 — Minimum Count Rule and `qty` Hardening**

### **1.2 Release Type**

Backend/domain hardening release focused on quantity consistency across the inventory pipeline.

### **1.3 Release Goal**

The goal of this release is to ensure that the system returns a **consistent, non-null quantity** whenever there is already **valid product evidence**, even if the model or an intermediate stage did not provide an explicit quantity.

This release introduces a formal backend rule so that:

```
if valid evidence exists and explicit quantity is missing, qty_final = 1
```

In addition, the release formalizes quantity provenance so the system can distinguish between:

* **detected quantity**  
* **inferred quantity**

This avoids semantically invalid outputs such as:

* `qty = null` with valid evidence  
* `qty = 0` with confirmed product presence

### **1.4 Business Motivation**

From a product and operations perspective, reporting a product as detected while returning `null` or `0` as quantity creates an inconsistent contract for:

* frontend consumption,  
* operational review,  
* downstream reporting,  
* and future analytics.

This release improves reliability by aligning quantity semantics with evidence semantics.

---

## **2\. Problem Statement**

### **2.1 Current Behavior**

In the current pipeline, some results reach final consolidation with:

* valid evidence,  
* a probable or confirmed product association,  
* but no valid explicit quantity.

In those scenarios, the final output may still expose:

* `qty = null`, or  
* `qty = 0`

This can happen due to one or more of the following:

1. The model returns no explicit quantity.  
2. The parser accepts missing/invalid quantity without normalization.  
3. Intermediate consolidation preserves an unresolved `qty` state.  
4. DTO generation serializes unresolved quantity values directly.  
5. Persistence stores quantity without provenance or inference metadata.

### **2.2 Why This Is a Problem**

This behavior breaks an important invariant:

If the pipeline accepts that a product is present with valid evidence, the final quantity must not be semantically empty.

A valid result cannot simultaneously mean:

* “we saw the product”  
* and “we have zero or no quantity”

unless there is an explicit domain rule supporting that state, which is not the case here.

### **2.3 Desired Outcome**

If product presence is accepted by the backend, then the final quantity contract must be hardened so that:

* explicit valid count wins,  
* inferred minimum count is used when explicit count is absent,  
* and unresolved null states do not leak into valid frontend-visible entities.

---

## **3\. Scope**

This release includes technical review and implementation changes in the following areas.

### **3.1 Model Prompt Review**

Review and improve the current prompt so the model is less likely to emit inconsistent quantity outputs when evidence clearly indicates at least one visible product.

Expected prompt-level improvements:

* discourage `qty = 0` when presence is confirmed,  
* discourage `qty = null` when a valid single item is visible,  
* improve semantic alignment between detection and quantity.

### **3.2 Model Response Parsing**

Review the response parser to ensure it can distinguish between:

* field missing,  
* field explicitly null,  
* zero value,  
* invalid numeric format,  
* and valid positive quantity.

The parser must stop behaving as a passive transport layer and instead expose normalized quantity state to downstream logic.

### **3.3 Intermediate Consolidation**

Review the consolidation layer to ensure quantity decisions are not lost, duplicated, or left unresolved before final result generation.

This is the preferred layer for applying the minimum-count rule because it has enough context to determine whether evidence is strong enough.

### **3.4 Final DTOs and API Output**

Review output DTOs so the frontend receives:

* a stable `qty` value,  
* explicit provenance metadata,  
* and a contract that does not require client-side reconstruction.

### **3.5 Persistence**

Review database/entity persistence so quantity provenance is recorded and traceable.

This release requires persistence support for:

* final quantity,  
* quantity source,  
* inference reason,  
* optional raw/original quantity.

### **3.6 Rule Formalization**

Document and centralize the minimum-count business rule in backend logic.

## **4\. Out of Scope**

The following items are explicitly not part of v3.2.2:

* model retraining,  
* major OCR redesign,  
* advanced multi-item counting heuristics,  
* customer-specific counting rules,  
* shelf-depth or facing-based multiplicity estimation,  
* broad frontend redesign,  
* large structural refactors unrelated to `qty` consistency.

This release is a hardening release, not a counting-accuracy overhaul.

## **5\. Release Requirements**

### **5.1 Functional Requirement**

When the system accepts a result as having valid product evidence, and there is no valid explicit quantity, the backend must produce:

```
qty_final = 1
```

### **5.2 Contract Requirement**

Frontend-visible valid entities must not expose unjustified `null` quantity values.

### **5.3 Provenance Requirement**

The system must expose whether quantity was:

* detected,  
* consolidated from explicit sources,  
* or inferred.

### **5.4 Traceability Requirement**

The system must record the reason why quantity was inferred.

### **5.5 Centralization Requirement**

The minimum-count rule must live in backend/domain logic, not in frontend.

## **6\. Domain Definitions**

### **6.1 Valid Evidence**

A result has **valid evidence** when it satisfies the current domain thresholds and validation conditions required for the backend to accept that a product, label, or inventory entity is truly present.

Examples may include:

* accepted primary evidence linked to a position,  
* validated image/crop associated with a SKU,  
* readable label associated with a valid detection,  
* evidence attached to a consolidated entity marked as accepted.

This release does not redefine evidence thresholds; it relies on the existing pipeline acceptance logic.

### **6.2 Explicit Quantity**

A quantity value is **explicit** when it was directly provided by:

* model output,  
* OCR,  
* counting logic,  
* or another structured step that produces a concrete numeric value.

### **6.3 Inferred Quantity**

A quantity value is **inferred** when the backend assigns a quantity through business rules rather than direct measurement.

### **6.4 Minimum Inferred Quantity**

A specific inferred quantity value of:

```
1
```

used when product presence is valid but cardinality above 1 cannot be safely supported.

---

## **7\. Formal Business Rule**

### **7.1 Main Rule**

If all the following conditions are true:

1. the entity has valid evidence,  
2. the entity is accepted as product-present,  
3. there is no valid explicit quantity,

then:

```
qty_final = 1
qty_source = inferred
qty_inference_reason = valid_evidence_without_explicit_quantity
```

### **7.2 Quantity Resolution Priority**

Final quantity resolution must follow this priority order:

1. valid explicit detected quantity,  
2. valid consolidated quantity from explicit sources,  
3. inferred minimum quantity \= 1,  
4. reject or do not materialize the entity if evidence is not sufficient.

### **7.3 Non-Applicability Cases**

The minimum-count rule must not be applied when:

* evidence is below acceptance threshold,  
* product identification is too ambiguous,  
* the entity is only hypothetical,  
* the entity is not accepted by domain validation,  
* or the result should remain discarded.

### **7.4 Interpretation of `qty = 0`**

A zero quantity must not be accepted at face value when the same entity is also considered product-present with valid evidence, unless a separate explicit business rule defines that state.

For v3.2.2, the expected interpretation is:

* if `0` appears in a valid product-present entity without strong justification, treat it as an invalid/unresolved quantity state,  
* then apply standard quantity resolution,  
* which may result in inferred minimum quantity.

## **8\. Technical Design**

## **8.1 Design Principle**

The provider/model may help reduce bad outputs, but the **backend is the source of truth**.

The release must not rely exclusively on prompt behavior.

The definitive quantity decision must be made in backend logic after parsing and after evidence validation.

### **8.2 Recommended Decision Point**

The recommended place to resolve final quantity is the **domain consolidation layer** or a central normalization function executed immediately before persistence/DTO materialization.

This is the strongest point because it has access to:

* evidence validity,  
* product acceptance status,  
* raw model output,  
* parser-normalized quantity state,  
* and final entity context.

### **8.3 Central Resolver Function**

A dedicated resolver should be introduced, for example conceptually:

```
resolve_final_qty(entity_context, raw_qty_state, evidence_state) -> QtyResolution
```

The resolver should produce a structured result rather than only a number.

Recommended conceptual output:

```
QtyResolution {
  qty_final,
  qty_source,
  qty_inference_reason,
  raw_qty,
  normalization_notes
}
```

### **8.4 Parser Hardening**

The parser should normalize quantity into explicit internal states such as:

* missing,  
* null,  
* invalid,  
* zero,  
* positive\_integer,  
* positive\_float\_if\_supported.

This makes it possible to distinguish:

* provider ambiguity,  
* parse failure,  
* and true quantity absence.

### **8.5 DTO Hardening**

The final DTO must expose a stable contract.

Minimum recommended fields:

```ts
qty: number
qtySource: 'detected' | 'consolidated' | 'inferred'
qtyInferenceReason: string | null
```

If the codebase prefers a smaller enum, `consolidated` can be collapsed into `detected`, but the distinction should remain internally available.

### **8.6 Persistence Hardening**

Persistence should store both the final decision and enough raw context for auditability.

Recommended persisted fields:

* `qty_final`  
* `qty_source`  
* `qty_inference_reason`  
* `raw_qty`  
* `qty_parse_status` or equivalent optional field

### **8.7 Frontend Responsibility Boundary**

The frontend should consume quantity, not infer it.

Frontend may display badges or tooltips such as:

* “Detected quantity”  
* “Inferred minimum quantity”

But it must not reconstruct missing quantity logic itself.

## **9\. Data Contract Proposal**

### **9.1 Internal Domain Contract**

Conceptual backend contract:

```ts
interface QtyResolution {
  qtyFinal: number
  qtySource: 'detected' | 'consolidated' | 'inferred'
  qtyInferenceReason?: 'valid_evidence_without_explicit_quantity' | null
  rawQty?: number | null
  qtyParseStatus?: 'missing' | 'null' | 'invalid' | 'zero' | 'valid'
}
```

### **9.2 External API Contract**

Minimum stable API contract:

```ts
interface InventoryResultDto {
  qty: number
  qtySource: 'detected' | 'inferred'
  qtyInferenceReason?: string | null
}
```

### **9.3 Validation Invariants**

The following invariants should hold for final valid entities:

1. `qty >= 1`  
2. `qty !== null`  
3. `qtySource = inferred` implies `qty = 1` for this release  
4. `qtySource = inferred` implies `qtyInferenceReason != null`  
5. valid visible results must not expose unresolved quantity state

## **11\. Test Strategy**

## **11.1 Unit Tests**

At minimum, cover the following cases:

1. valid evidence \+ missing qty \-\> final qty \= 1 inferred  
2. valid evidence \+ null qty \-\> final qty \= 1 inferred  
3. valid evidence \+ unjustified zero qty \-\> final qty \= 1 inferred  
4. valid evidence \+ explicit qty \= 2 \-\> final qty \= 2 detected  
5. invalid evidence \+ null qty \-\> no inference  
6. inferred qty includes correct source and reason  
7. final DTO does not expose unjustified null qty for valid entities

### **11.2 Integration Tests**

Cover end-to-end or semi-integrated flows where:

* model returns valid product evidence without quantity,  
* parser normalizes correctly,  
* resolver infers minimum count,  
* persistence stores provenance,  
* API returns stable DTO.

### **11.3 Regression Tests**

Add specific regression coverage for historical failures observed between branches where:

* product/SKU was correct,  
* quantity became null,  
* or quantity regressed to zero after prompt or parser changes.

## **13\. Risks**

### **13.1 Overcounting Ambiguous Cases**

There is a risk of inferring `1` in borderline cases where evidence is not strong enough.

**Mitigation:**  
Only apply the rule after domain acceptance of valid evidence.

### **13.2 Conflating Weak Detection with Accepted Presence**

There is a risk of treating weak hypotheses as confirmed presence.

**Mitigation:**  
Keep evidence acceptance and quantity inference as separate steps, with quantity inference only after acceptance.

### **13.3 Logic Duplication**

There is a risk of implementing part of the rule in parser, part in consolidation, and part in frontend.

**Mitigation:**  
Centralize final resolution in a dedicated backend resolver.

### **13.4 Persistence/API Mismatch**

There is a risk that persistence stores one value and API returns another.

**Mitigation:**  
Reuse the same final resolved structure for both persistence and serialization.

## **14\. Dependencies**

This release is logically related to v3.2.1, but can be developed in parallel as long as:

* the consolidation contract is stable enough,  
* and the DTO layer is not simultaneously undergoing breaking changes.

Recommended sequencing:

* after v3.2.1 if possible,  
* in parallel only if ownership of quantity flow is well isolated.

## **16\. Acceptance Criteria**

The release is accepted only if all of the following are true:

1. when a valid entity has a single valid evidence and no explicit quantity, `qty_final = 1`  
2. valid entities do not expose unjustified `null` quantity  
3. valid product-present entities do not expose unjustified `0` quantity  
4. frontend receives a consistent quantity value for valid results  
5. frontend can know whether quantity was detected or inferred  
6. final quantity logic is centralized in backend  
7. known regression cases are covered by automated tests

---

## **17\. Recommended Engineering Notes**

* Do not treat prompt quality as the primary safeguard.  
* Do not let DTO serialization leak unresolved internal quantity states.  
* Do not push quantity semantics into frontend.  
* Prefer a small, explicit backend resolver over scattered conditionals.  
* Make provenance visible now to reduce future debugging cost.

---

## **18\. Conclusion**

v3.2.2 is a consistency-hardening release focused on a narrow but high-impact invariant:

accepted product presence must produce a coherent quantity contract.

This release does not try to solve advanced counting. It solves something more foundational:

* semantic consistency,  
* backend determinism,  
* API reliability,  
* and traceability of quantity decisions.

By formalizing minimum inferred quantity and exposing quantity provenance, the system becomes safer to consume, easier to debug, and better prepared for future counting improvements.
