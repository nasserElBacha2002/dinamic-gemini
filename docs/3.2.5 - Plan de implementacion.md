# **Plan de Implementación — Release 3.2.5**

## **Bloque C — Consolidación técnica v3**

**Versión:** 3.2.5  
**Objetivo general:** consolidar el sistema alrededor del modelo v3, cerrar inconsistencias entre capas, mejorar la review operativa y dejar trazabilidad suficiente para debugging antes de deploy en dev.

---

## **1\. Estrategia general**

La release 3.2.5 no debe ejecutarse como un conjunto de cambios mezclados.  
Conviene implementarla en **bloques secuenciales**, porque cada uno reduce incertidumbre para el siguiente.

Orden recomendado:

1. **Auditoría y cierre de superficie v3**  
2. **Consolidación backend, DTOs, jobs y persistencia**  
3. **Alineación de artifacts y lectura histórica**  
4. **Review operativa y trazabilidad manual**  
5. **Observabilidad y debugging**  
6. **Validación end-to-end y cierre**

La idea es que primero se defina **qué modelo existe realmente**, después se alinee backend, luego frontend, y recién entonces se agregue trazabilidad extra y validación final.

---

# **2\. Fases del plan**

# **Fase 1 — Auditoría y cierre de superficie activa v3**

## **Objetivo**

Identificar con precisión qué rutas, DTOs, estados, artifacts y flujos están realmente activos en v3, y detectar cualquier resto legacy que todavía afecte comportamiento.

## **Tareas**

* Inventariar todas las rutas v3 activas.  
* Identificar rutas legacy todavía expuestas o todavía usadas indirectamente.  
* Relevar request/response DTOs de:  
  * inventories,  
  * aisles,  
  * jobs,  
  * positions,  
  * review,  
  * evidences,  
  * artifacts.  
* Relevar estados reales del lifecycle de jobs.  
* Mapear qué consume efectivamente el frontend.  
* Detectar campos:  
  * ambiguos,  
  * redundantes,  
  * inconsistentes,  
  * viejos pero aún activos,  
  * persistidos pero no consumidos,  
  * consumidos pero no garantizados por backend.  
* Identificar dónde el frontend compensa inconsistencias del backend.

## **Entregables**

* Documento de route surface activa.  
* Matriz backend/frontend DTO alignment.  
* Mapa de lifecycle actual de jobs.  
* Lista de deuda legacy con severidad:  
  * bloqueante,  
  * importante,  
  * cleanup opcional.

## **Criterios de salida**

* Está claro qué superficie es la “verdad activa”.  
* Está claro qué restos legacy deben eliminarse en esta release.  
* Está claro qué contratos deben cambiar y cuáles deben mantenerse.

---

# **Fase 2 — Consolidación de contratos backend/frontend**

## **Objetivo**

Cerrar la semántica v3 a nivel de contratos de API para que frontend y backend hablen el mismo idioma sin compensaciones frágiles.

## **Tareas**

* Revisar y normalizar DTOs de lectura y detalle.  
* Revisar naming y obligatoriedad de campos.  
* Definir contratos finales para:  
  * job summary,  
  * job detail,  
  * result/position list,  
  * position detail,  
  * review action result,  
  * evidence payload,  
  * artifact references.  
* Eliminar o deprecatear campos que arrastren semántica vieja.  
* Alinear responses de listado vs detalle.  
* Verificar consistencia entre schemas, servicios y repositorios.  
* Corregir cualquier divergencia entre persistencia y payload expuesto.  
* Documentar contratos finales v3.

## **Entregables**

* DTOs v3 ajustados.  
* Contratos request/response cerrados.  
* Documento de contratos actualizado.  
* Lista de cambios frontend requeridos por contrato.

## **Criterios de salida**

* No existen campos cuya semántica cambie según endpoint.  
* List y detail son compatibles conceptualmente.  
* Frontend puede consumir el backend sin lógica compensatoria innecesaria.

---

# **Fase 3 — Revisión del lifecycle de jobs**

## **Objetivo**

Dejar el lifecycle de jobs coherente, documentado y compatible con cancelación, retries y lectura histórica.

## **Tareas**

* Relevar estados actuales de job.  
* Definir set final de estados válidos.  
* Revisar transiciones permitidas.  
* Validar qué ocurre en:  
  * enqueue,  
  * start,  
  * processing,  
  * completion,  
  * failure,  
  * cancellation,  
  * retry.  
* Revisar cómo se persisten:  
  * errores,  
  * metadata contextual,  
  * referencias a output/resultados.  
* Verificar que los estados sean interpretables por frontend.  
* Revisar si retry crea nuevo job, reusa job o referencia intento previo.  
* Definir semántica histórica para jobs viejos.  
* Actualizar documentación del lifecycle.

## **Entregables**

* Lifecycle v3 documentado.  
* Tabla de estados y transiciones.  
* Ajustes en repositorios / servicios / DTOs de jobs.  
* Criterios claros para cancel y retry.

## **Criterios de salida**

* Cada estado significa una sola cosa.  
* Cancelación y retry no dejan estados ambiguos.  
* Frontend puede renderizar correctamente cada job.

---

# **Fase 4 — Alineación de persistencia y artifacts**

## **Objetivo**

Asegurar que la persistencia y los artifacts reflejen la lógica real del pipeline y sean consistentes con lo que consume la UI y con lo que se necesita para debugging.

## **Tareas**

* Auditar qué se persiste hoy por job, position, evidence y review.  
* Verificar que cada artifact importante:  
  * exista,  
  * tenga referencia persistida válida,  
  * sea accesible desde la UI o debugging,  
  * tenga naming estable.  
* Detectar artifacts huérfanos o referencias rotas.  
* Revisar consistencia entre evidencia persistida y evidencia mostrada.  
* Revisar disponibilidad de:  
  * imagen original,  
  * previews,  
  * normalized versions,  
  * primary evidence,  
  * secondary evidence,  
  * raw output relevantes.  
* Alinear persistencia de artifacts con lectura frontend.  
* Revisar compatibilidad de lectura histórica.  
* Corregir diferencias entre resultado final y soporte visual/documental.

## **Entregables**

* Modelo de artifacts alineado.  
* Correcciones de persistencia.  
* Validación de referencias entre evidence/result/artifacts.  
* Documento de artifacts relevantes por etapa.

## **Criterios de salida**

* No hay artifacts relevantes inaccesibles.  
* No hay referencias persistidas inválidas o semánticamente engañosas.  
* La evidencia mostrada coincide con el dato final o explica su construcción.

---

# **Fase 5 — Consolidación de resultados y posiciones**

## **Objetivo**

Alinear el modelo de positions/resultados con la lógica real del pipeline, incluyendo origen del conteo, needs\_review y consistencia entre dato detectado, consolidado y corregido.

## **Tareas**

* Revisar shape final de `positions`.  
* Identificar qué campos son:  
  * detectados,  
  * inferidos,  
  * consolidados,  
  * corregidos manualmente.  
* Definir cómo se expresa el origen del conteo.  
* Definir cómo se expresa el motivo de `needs_review`.  
* Verificar consistencia entre:  
  * summary,  
  * detail,  
  * evidence,  
  * review state.  
* Corregir diferencias entre parseo interno y exposición final.  
* Asegurar que `final_quantity` y campos relacionados tengan semántica estable.  
* Evitar nulls ambiguos sin explicación.  
* Documentar el contrato final del resultado.

## **Entregables**

* Contrato final de `position/result`.  
* Campos de origen del conteo.  
* Motivo de review visible en modelo.  
* Ajustes de DTO y persistencia donde haga falta.

## **Criterios de salida**

* Un item final puede explicarse con claridad.  
* El origen del qty es visible y consistente.  
* `needs_review` no aparece como bandera opaca.

---

# **Fase 6 — Review operativa y acciones manuales**

## **Objetivo**

Mejorar la experiencia de revisión para que el operador entienda qué pasó y pueda corregir con seguridad.

## **Tareas**

* Mejorar jerarquía visual de evidencia principal y secundaria.  
* Exponer de forma clara:  
  * motivo de review,  
  * origen del conteo,  
  * estado final actual,  
  * impacto de correcciones manuales.  
* Revisar acciones manuales disponibles:  
  * approve,  
  * reject,  
  * edit quantity,  
  * override fields,  
  * reclassify si aplica.  
* Definir qué capa corrige cada acción.  
* Alinear acción UI ↔ persistencia ↔ DTO de respuesta.  
* Persistir auditoría básica:  
  * acción realizada,  
  * valor previo,  
  * valor nuevo,  
  * timestamp,  
  * actor si corresponde,  
  * contexto mínimo.  
* Validar que luego de corregir, el resultado pueda releerse de forma consistente.

## **Entregables**

* UI de review mejorada.  
* Modelo de auditoría básica de acciones.  
* Contrato de acciones manuales claro.  
* Persistencia de correcciones manuales.

## **Criterios de salida**

* El operador entiende por qué el caso está en review.  
* Puede corregir sin ambigüedad.  
* La corrección queda persistida y trazable.

---

# **Fase 7 — Observabilidad y debugging**

## **Objetivo**

Dejar trazabilidad suficiente para reconstruir una corrida y aislar regresiones entre raw output, parseo, consolidación y resultado final.

## **Tareas**

* Definir metadata mínima obligatoria por job:  
  * prompt version,  
  * provider/model,  
  * reglas relevantes,  
  * inferencias de qty,  
  * merges aplicados,  
  * artifacts clave.  
* Definir qué se persiste y qué sólo se loguea.  
* Estructurar logging por etapa del pipeline.  
* Agregar contexto útil a errores:  
  * etapa,  
  * entity/position afectada,  
  * artifact relacionado,  
  * provider/model involucrado,  
  * parsing/consolidation step.  
* Habilitar comparación mínima entre:  
  * raw response,  
  * parsed payload,  
  * consolidated result,  
  * final exposed DTO.  
* Documentar procedimiento de investigación de regresiones.  
* Estandarizar nombres/versiones para metadata persistida.

## **Entregables**

* Logging mejorado.  
* Metadata técnica persistida.  
* Vistas o accesos mínimos para debugging.  
* Documento operativo de investigación.

## **Criterios de salida**

* Se puede reconstruir una corrida relevante.  
* Se puede distinguir si el fallo fue de parseo, consolidación o exposición.  
* Los errores tienen contexto suficiente para actuar.

---

## **Entregables**

* Consumo consistente de artifacts y evidences.

## **Criterios de salida**

* Frontend representa la semántica real del backend.

* La experiencia operativa es más clara y estable.

# **5\. Riesgos de implementación**

## **Riesgo 1 — Mezclar cleanup con features**

Mitigación: todo cambio debe justificarse contra un criterio de consolidación.

## **Riesgo 2 — Romper frontend por ajustes de DTO**

Mitigación: cerrar contratos primero, adaptar frontend después, no en paralelo caótico.

## **Riesgo 3 — Persistir metadata inútil**

Mitigación: cada nuevo campo debe tener consumidor o caso claro de debugging.

## **Riesgo 4 — Ambigüedad en correcciones manuales**

Mitigación: definir explícitamente qué corrige cada acción y dónde impacta.

## **Riesgo 5 — Sobrecargar la UI de review**

Mitigación: priorizar claridad operacional y exponer metadata técnica sólo donde aporte.

---

# **6\. Criterios de done del release**

La release 3.2.5 estará terminada cuando:

* la superficie activa v3 esté auditada y cerrada,  
* no haya rutas activas mezclando semánticas viejas y nuevas,  
* los DTOs estén alineados entre backend, persistencia y frontend,  
* el lifecycle de jobs esté definido y validado,  
* cancelación, retries e histórico sean coherentes,  
* artifacts y evidences estén alineados con la UI,  
* la review manual sea más clara y trazable,  
* exista auditoría básica de correcciones,  
* exista metadata suficiente para debugging,  
* se pueda reconstruir una corrida conflictiva,  
* la documentación final del release esté actualizada.

---

# **7\. Resultado esperado**

Al finalizar 3.2.5, el sistema debe quedar en una condición en la que:

* v3 sea la única semántica operativa real,  
* backend y frontend estén alineados,  
* jobs, results, evidences y artifacts formen un modelo coherente,  
* el operador pueda revisar mejor,  
* el equipo pueda debuggear mejor,  
* el deploy a dev tenga una base mucho más confiable.
