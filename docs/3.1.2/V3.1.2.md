# **Plan definitivo — Versión 3.1.2**

## **1\. Objetivo general**

La versión **3.1.2** tendrá como objetivo principal realizar una **consolidación técnica del sistema**, eliminando deuda acumulada, ordenando la arquitectura real del proyecto y mejorando la mantenibilidad general del backend, frontend y base de datos.

No es una versión pensada para agregar grandes funcionalidades de negocio nuevas, sino para dejar una base más limpia, coherente y escalable.

---

## **2\. Objetivos específicos**

### **Backend**

* Eliminar métodos, endpoints, servicios y lógica que ya no correspondan a la arquitectura vigente.  
* Quitar todo lo que no esté alineado con **v3**.  
* Detectar duplicaciones de código y oportunidades de mejora.  
* Reordenar directorios y módulos para que reflejen correctamente la responsabilidad de cada capa.  
* Revisar responses enviadas al frontend para identificar:  
  * datos duplicados  
  * campos innecesarios  
  * payloads excesivos  
  * problemas de performance

### **Base de datos**

* Eliminar tablas legacy que ya no se usen.  
* Renombrar tablas con prefijo o referencia a versión (`v3`) para que tengan nombres de dominio reales.  
* Revisar consistencia entre tablas, relaciones, índices y acceso desde backend.

### **Frontend**

* Reorganizar directorios y estructura interna.  
* Eliminar clientes, tipos, vistas o componentes que hayan quedado legacy.  
* Reducir duplicación de lógica y componentes.  
* Adaptar el frontend al contrato final estabilizado del backend.

### **Operación del sistema**

* Incorporar la capacidad de **cancelar un job en ejecución** cuando supere tiempos razonables o cuando se decida interrumpirlo manualmente.

---

## **3\. Principios de la versión**

### **3.1. No mantener legado innecesario**

Todo lo que no tenga relación con la arquitectura vigente debe salir del sistema.

### **3.2. El dominio debe reflejarse en nombres reales**

La base de datos, las carpetas y los contratos deben expresar conceptos de negocio, no nombres transitorios como `v3_*`.

### **3.3. Backend primero, frontend después**

El frontend debe reorganizarse una vez que el backend y la base ya estén estabilizados.

### **3.4. Refactor con impacto controlado**

Cada etapa debe dejar al sistema operable, sin hacer un “big bang refactor”.

### **3.5. Performance observable**

Cada mejora de responses o queries debe tener una justificación concreta.

# **4\. Etapas del plan**

---

# **Etapa 1 — Auditoría técnica integral**

## **Objetivo**

Realizar un relevamiento completo del estado actual del sistema para identificar qué debe mantenerse, qué debe eliminarse y qué debe refactorizarse.

## **Alcance**

* Backend  
* Base de datos  
* Frontend  
* Contratos API  
* Estructura de carpetas  
* Pipeline de jobs

## **Tareas**

* Inventariar todos los endpoints actuales.  
* Identificar cuáles siguen vigentes y cuáles son legacy.  
* Revisar servicios, casos de uso, repositories, adapters y helpers.  
* Detectar módulos, métodos e imports sin uso.  
* Auditar tablas actuales de la base.  
* Identificar tablas que ya no estén en uso real.  
* Analizar estructura actual de carpetas en backend y frontend.  
* Mapear cada response enviada al frontend.  
* Detectar payloads innecesarios o duplicados.  
* Detectar endpoints con potencial problema de performance.

## **Entregables**

* Documento de auditoría técnica.  
* Matriz de elementos vigentes vs legacy.  
* Lista priorizada de refactors.  
* Mapa de contratos backend → frontend.

## **Criterio de salida**

Se conoce con precisión:

* qué se elimina  
* qué se renombra  
* qué se reorganiza  
* qué se optimiza  
* qué se mantiene

---

# **Etapa 2 — Limpieza de legado en backend**

## **Objetivo**

Eliminar del backend todo artefacto que ya no tenga relación con la arquitectura actual.

## **Alcance**

* Endpoints viejos  
* Métodos legacy  
* Services y use cases fuera de v3  
* Repositories/adapters sin uso  
* DTOs y modelos obsoletos  
* Código muerto

## **Tareas**

* Eliminar rutas antiguas no utilizadas.  
* Eliminar handlers o controllers legacy.  
* Eliminar services/use cases fuera de la arquitectura vigente.  
* Quitar lógica de compatibilidad ya innecesaria.  
* Eliminar mappers, DTOs y schemas no usados.  
* Revisar dependencias cruzadas que sigan trayendo módulos viejos.  
* Limpiar tests que dependan de funcionalidad obsoleta.

## **Entregables**

* Backend reducido al flujo vigente.  
* Listado de artefactos eliminados.  
* Validación de que no queden referencias activas a código legacy.

## **Criterio de salida**

El backend deja de cargar compatibilidad innecesaria con implementaciones anteriores.

---

# **Etapa 3 — Normalización y limpieza de base de datos**

## **Objetivo**

Dejar el esquema de base alineado con el dominio real del sistema y no con nombres atados a una versión.

## **Alcance**

* Tablas  
* Relaciones  
* Índices  
* Foreign keys  
* Nombres de constraints  
* Acceso desde repositories/queries

## **Tareas**

* Identificar tablas legacy que ya no se usan.  
* Eliminar tablas que hayan quedado obsoletas.  
* Renombrar tablas con prefijo o nomenclatura `v3`.  
* Renombrar constraints e índices si corresponde.  
* Revisar nombres de columnas ambiguas.  
* Validar integridad referencial.  
* Adaptar consultas SQL, ORM o repositories.  
* Generar scripts de migración seguros.  
* Actualizar documentación de esquema.

## **Entregables**

* Esquema limpio y consistente.  
* Migraciones/scripts de renombre y eliminación.  
* Acceso a datos alineado al nuevo naming.

## **Criterio de salida**

La base de datos expresa entidades de negocio, no etapas técnicas del proyecto.

---

# **Etapa 4 — Reorganización estructural del backend**

## **Objetivo**

Reordenar el backend para que su estructura sea más clara, mantenible y coherente con la arquitectura real.

## **Alcance**

* Módulos  
* Directorios  
* Imports  
* Separación por responsabilidades  
* Organización por dominio o feature

## **Tareas**

* Definir estructura objetivo del backend.  
* Separar claramente API, dominio, aplicación e infraestructura si corresponde.  
* Agrupar funcionalidades por contexto real.  
* Reubicar módulos de jobs, inventories, positions, evidences, reports, pipeline, etc.  
* Eliminar carpetas intermedias innecesarias.  
* Reducir acoplamientos y dependencias circulares.  
* Corregir imports y referencias tras el movimiento.

## **Entregables**

* Nueva estructura de carpetas backend.  
* Convención organizativa documentada.  
* Código reubicado sin romper comportamiento.

## **Criterio de salida**

La estructura del backend queda clara y consistente para evolución futura.

---

# **Etapa 5 — Consolidación técnica y optimización del backend**

## **Objetivo**

Mejorar la calidad interna del backend, eliminando duplicaciones y optimizando contratos y performance.

## **Subobjetivos**

* reducir duplicación  
* simplificar responses  
* mejorar queries  
* reducir payload innecesario  
* mejorar tiempos de respuesta

## **Tareas**

* Detectar lógica duplicada entre services, mappers y handlers.  
* Consolidar validaciones repetidas.  
* Unificar construcción de responses.  
* Revisar DTOs summary vs detail.  
* Eliminar campos duplicados o innecesarios en respuestas.  
* Analizar qué consume realmente el frontend.  
* Optimizar queries costosas o redundantes.  
* Detectar problemas N+1 o sobrecarga de joins.  
* Revisar serialización excesiva.  
* Separar endpoints de resumen y detalle donde haga falta.

## **Entregables**

* Responses más limpias.  
* Menos duplicación interna.  
* Backend más consistente y más eficiente.  
* Documento comparativo payload actual vs payload optimizado.

## **Criterio de salida**

El backend expone contratos más claros, más livianos y con mejor rendimiento.

---

# **Etapa 6 — Cancelación de jobs y control de procesos largos**

## **Objetivo**

Agregar una capacidad operativa real para interrumpir jobs que demoren demasiado o deban detenerse manualmente.

## **Alcance**

* Modelo de estado del job  
* Backend API  
* Worker/pipeline  
* Persistencia  
* Integración posterior con frontend

## **Tareas**

* Definir nuevos estados del job, por ejemplo:  
  * `cancel_requested`  
  * `canceled`  
  * `timed_out`  
* Incorporar endpoint para solicitar cancelación.  
* Registrar motivo de cancelación.  
* Agregar chequeos cooperativos de cancelación dentro del pipeline.  
* Interrumpir el job en puntos seguros.  
* Persistir estado final y timestamps.  
* Definir política de timeout.  
* Evaluar cómo quedan artifacts parciales.  
* Preparar respuesta del backend para que el frontend pueda reflejarlo.

## **Recomendación técnica**

Implementar primero **cancelación cooperativa**, no forzada:

* el sistema marca cancelación  
* el worker chequea  
* corta ordenadamente  
* persiste estado final

## **Entregables**

* Endpoint de cancelación.  
* Soporte en worker/pipeline.  
* Estado de job actualizado correctamente.  
* Tests de cancelación y timeout.

## **Criterio de salida**

El sistema deja de depender de jobs irreversibles o colgados.

---

* Hooks

* Menos duplicación.

## **Criterio de salida**

El frontend consume y representa datos con menos complejidad y menos código redundante.

# **5\. Orden definitivo de ejecución**

El orden recomendado para ejecutar la v3.1.2 es:

1. **Auditoría técnica integral**  
2. **Limpieza de legado en backend**  
3. **Normalización y limpieza de base de datos**  
4. **Reorganización estructural del backend**  
5. **Consolidación técnica y optimización del backend**  
6. **Cancelación de jobs y control de procesos largos**  
7. **Reorganización estructural del frontend**  
8. **Limpieza y optimización del frontend**  
9. **Testing, validación integral y cierre**

---

# **6\. División por bloques**

## **Bloque A — Núcleo técnico**

Incluye todo lo que define la base real del sistema:

* auditoría  
* backend legacy  
* base de datos  
* estructura backend  
* optimización backend  
* cancelación de jobs

## **Bloque B — Consumo y presentación**

Incluye todo lo que consume el contrato ya estabilizado:

* reorganización frontend  
* optimización frontend

## **Bloque C — Estabilización**

Incluye:

* testing  
* validación  
* documentación  
* cierre

---

# **7\. Riesgos principales**

## **Riesgo 1 — Eliminar código todavía referenciado**

Mitigación: auditoría previa \+ búsqueda de referencias \+ smoke tests.

## **Riesgo 2 — Romper acceso a datos al renombrar tablas**

Mitigación: migraciones controladas \+ adaptación coordinada de repositories.

## **Riesgo 3 — Reorganizar demasiado en una sola etapa**

Mitigación: dividir cambios por capas y mantener criterios de salida claros.

## **Riesgo 4 — Optimizar responses sin validar uso real del frontend**

Mitigación: mapear consumo real antes de remover campos.

## **Riesgo 5 — Cancelar jobs en puntos inseguros**

Mitigación: cancelación cooperativa con checkpoints definidos.

---

# **8\. Resultado esperado de la versión 3.1.2**

Al cerrar esta versión, el sistema debería quedar:

* sin legado innecesario  
* con backend más limpio y ordenado  
* con base de datos alineada al dominio  
* con menos duplicación  
* con mejores contratos API  
* con respuestas más eficientes  
* con frontend reorganizado sobre contratos estables  
* con control operativo para cancelar jobs largos  
* con una base mucho más sólida para futuras versiones

# **9\. Recomendación de implementación**

A nivel práctico, esta versión conviene ejecutarla como una **serie de épicas separadas**, una por etapa, para evitar mezclar:

* limpieza  
* renombre de DB  
* refactor de estructura  
* optimización de contratos  
* cambios funcionales de jobs  
* frontend

Eso te va a permitir revisar cada bloque con más control y menos riesgo.
