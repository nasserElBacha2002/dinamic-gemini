# **EPIC\_V3.1.2\_BACKEND\_CLEANUP.md**

## **1\. Summary**

This epic covers the removal of backend legacy code that no longer belongs to the active architecture of Dinamic Inventory.

The goal is to reduce technical debt, simplify the backend surface area, and ensure the system only contains code paths aligned with the current product model.

---

## **2\. Objective**

Remove obsolete backend routes, handlers, services, repositories, DTOs and compatibility logic that are no longer used by the current system.

---

## **3\. Scope**

Included:

* legacy routes  
* deprecated controllers/handlers  
* old services/use cases  
* unused repositories/adapters  
* obsolete DTOs/schemas/mappers  
* dead imports and references  
* stale tests tied to removed flows

Excluded:

* DB table renaming  
* frontend cleanup  
* performance optimization  
* job cancellation behavior

---

## **4\. User stories**

### **Story 1 — Remove legacy routes**

**As a** developer  
**I want** obsolete backend routes removed  
**So that** the API surface reflects only supported flows.

**Acceptance criteria**

* routes outside the active architecture are removed  
* no active frontend flow depends on them  
* route registry stays consistent  
* API boots successfully after cleanup

---

### **Story 2 — Remove obsolete business logic**

**As a** developer  
**I want** deprecated services and use cases removed  
**So that** backend behavior is easier to understand and maintain.

**Acceptance criteria**

* obsolete service/use-case classes are removed  
* internal references are cleaned up  
* no dead dependency chains remain  
* tests are updated accordingly

---

### **Story 3 — Remove obsolete DTOs and mappers**

**As a** developer  
**I want** outdated DTOs, schemas and mappers removed  
**So that** response and request contracts remain coherent.

**Acceptance criteria**

* unused DTOs and schemas are deleted  
* mapping utilities not used by active flows are removed  
* imports compile cleanly  
* API models align with active routes only

---

### **Story 4 — Eliminate dead compatibility code**

**As a** maintainer  
**I want** transitional compatibility layers removed  
**So that** the backend no longer carries historical baggage.

**Acceptance criteria**

* old fallbacks or compatibility branches are removed  
* no code path references unsupported versions  
* documentation is updated if behavior changed

---

## **5\. Technical tasks**

* inventory all backend routes  
* trace active usage from frontend and internal modules  
* delete legacy route registrations  
* remove deprecated handlers/controllers  
* remove unused services/use cases  
* remove stale repositories/adapters  
* delete obsolete DTOs/schemas/mappers  
* clean imports  
* fix test suite  
* run smoke validation

---

## **6\. Risks**

* removing code still referenced indirectly  
* deleting routes still used by frontend  
* hidden dependencies through shared utilities

**Mitigation**

* audit first  
* reference search before deletion  
* smoke tests after each cleanup block

---

## **7\. Exit criteria**

* backend contains only active flows  
* no dead legacy modules remain  
* tests pass or are intentionally updated  
* application starts and core flows still work

---

# **EPIC\_V3.1.2\_JOB\_CANCELLATION.md**

## **1\. Summary**

This epic adds controlled cancellation and timeout handling for long-running jobs.

---

## **2\. Objective**

Allow the system to stop jobs safely, persist their final state and expose cancellation clearly to API consumers.

---

## **3\. Scope**

Included:

* job states  
* cancel endpoint  
* worker/pipeline checkpoints  
* timeout handling  
* cancellation persistence  
* API exposure of canceled/timed-out states

Excluded:

* forced process killing as primary strategy  
* unrelated scheduler redesign

---

## **4\. User stories**

### **Story 1 — Manually request cancellation**

**As a** user/operator  
**I want** to request cancellation of a running job  
**So that** I can stop work that is no longer needed.

**Acceptance criteria**

* API supports cancellation request  
* running job can transition to `cancel_requested`  
* final state becomes `canceled` when interruption completes safely

---

### **Story 2 — Timeout long-running jobs**

**As a** system operator  
**I want** excessively long jobs to be timed out  
**So that** stuck jobs do not continue indefinitely.

**Acceptance criteria**

* timeout policy is defined  
* timed-out jobs transition to `timed_out`  
* reason and timestamps are persisted

---

### **Story 3 — Gracefully stop the pipeline**

**As a** maintainer  
**I want** cancellation checks at safe checkpoints  
**So that** job interruption does not corrupt state.

**Acceptance criteria**

* cancellation is cooperative  
* checkpoints exist in long-running stages  
* partial state handling is defined

---

## **5\. Technical tasks**

* define new states  
* add cancel endpoint  
* persist reason/timestamps  
* add checkpoint checks in worker/pipeline  
* define timeout policy  
* expose updated state in responses  
* add unit/integration tests  
* define handling of partial artifacts

---

## **6\. Exit criteria**

* manual cancellation works  
* timeout path works  
* states are persisted consistently  
* frontend can consume new status values

---
