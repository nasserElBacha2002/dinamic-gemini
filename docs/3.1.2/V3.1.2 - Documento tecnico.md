# **PLAN\_V3.1.2\_IMPLEMENTATION.md**

## **1\. Summary**

This document defines the implementation plan for **Dinamic Inventory v3.1.2**.

The purpose of this version is not to add major new business capabilities, but to perform a **technical consolidation** of the platform so the system becomes cleaner, easier to maintain, less coupled to legacy structures, and better prepared for future evolution.

The main focus areas are:

* removal of legacy code and database structures not aligned with the current architecture  
* backend cleanup and structural refactor  
* database normalization and renaming of version-based tables  
* response contract review and backend performance improvements  
* frontend reorganization after backend stabilization  
* addition of a controlled job cancellation mechanism for long-running processes

This version should be treated as a **hardening and simplification release**.

---

## **2\. Goals**

### **2.1 Primary goals**

* Remove all backend logic, endpoints, methods, modules and database tables that are no longer relevant to the current v3 architecture.  
* Eliminate naming tied to technical versioning, especially in database table names such as `v3_*`.  
* Reorganize backend and frontend directories so they reflect the real domain and architectural responsibilities of the system.  
* Detect and reduce duplication across backend and frontend.  
* Audit all responses sent to the frontend and improve them where necessary by:  
  * removing duplicated data  
  * removing unused fields  
  * reducing payload size  
  * improving endpoint response performance  
* Add a mechanism to stop or cancel jobs that are taking too long or should no longer continue running.

### **2.2 Secondary goals**

* Improve developer experience and project readability.  
* Reduce technical debt before future feature expansion.  
* Improve consistency across API contracts, naming and folder structure.  
* Increase operational control over long-running jobs.

---

## **3\. Non-goals**

The following are explicitly out of scope for v3.1.2 unless they become strictly necessary to complete the refactor:

* introducing major new business workflows unrelated to cleanup and technical consolidation  
* redesigning the product UX from scratch  
* changing the core business meaning of inventories, positions, evidences or reports  
* replacing the main technical stack  
* large-scale infrastructure migration unrelated to this version  
* aggressive runtime behavior changes not required by cleanup, performance or cancellation support

---

## **4\. Guiding principles**

### **4.1 Remove unused legacy aggressively, but safely**

Any code, route, service, table or component that no longer belongs to the real architecture should be removed. However, removal must be based on prior audit and reference validation.

### **4.2 The system should express domain, not version labels**

Database tables, modules and contracts should represent business concepts rather than transitional version names like `v3`.

### **4.3 Backend and database first, frontend second**

The frontend should only be reorganized after backend contracts and data structures are stabilized.

### **4.4 Refactor incrementally, not as a big bang**

Each stage should preserve a working system and avoid unnecessary multi-layer instability.

### **4.5 Optimize only where there is a concrete reason**

Response and query optimizations should be backed by real frontend usage, payload inspection or observed latency.

### **4.6 Operational control is part of product quality**

Long-running or stuck jobs should not remain uncontrollable. Cancellation and timeout handling are part of a production-ready system.

## **6\. High-level execution order**

The implementation order for v3.1.2 will be:

1. Technical audit  
2. Backend legacy cleanup  
3. Database normalization and cleanup  
4. Backend structural reorganization  
5. Backend consolidation and optimization  
6. Job cancellation and timeout control  
7. Frontend structural reorganization  
8. Frontend cleanup and optimization  
9. Testing, validation and technical closure

This order is intentional and follows the dependency chain:

**database → backend → frontend**

## **7\. Stage-by-stage plan**

---

## **Stage 1 — Technical audit**

### **Objective**

Produce a full technical inventory of the current system and identify what should be kept, removed, renamed, reorganized or optimized.

### **Scope**

* backend  
* frontend  
* database  
* API contracts  
* job execution flow  
* folder structure

### **Tasks**

* inventory all current backend endpoints  
* classify routes as active, deprecated or legacy  
* inspect services, use cases, repositories, adapters and helpers  
* detect dead code, orphan modules and unused imports  
* inventory existing database tables and identify real usage  
* detect legacy tables still present in the schema  
* inspect frontend structure and identify outdated areas  
* map backend responses to actual frontend consumption  
* identify duplicated or unnecessarily large payloads  
* flag endpoints with suspicious latency or overly complex queries

### **Deliverables**

* technical audit report  
* active vs legacy matrix  
* backend/frontend/database cleanup candidate list  
* API contract map  
* prioritized refactor list

### **Exit criteria**

There is a clear and validated understanding of:

* what will be removed  
* what will be renamed  
* what will be moved  
* what will be optimized  
* what must remain untouched

---

## **Stage 2 — Backend legacy cleanup**

### **Objective**

Remove everything in the backend that is no longer part of the real v3 architecture.

### **Scope**

* old routes  
* deprecated handlers/controllers  
* legacy services/use cases  
* obsolete repositories and adapters  
* dead DTOs and schemas  
* stale compatibility code

### **Tasks**

* remove routes outside the active architecture  
* remove handlers/controllers that are no longer referenced  
* remove deprecated service/use-case layers  
* remove compatibility logic that no longer has consumers  
* delete obsolete DTOs, schemas and mapping utilities  
* clean internal references to removed modules  
* remove or update tests that depend on obsolete flows  
* verify that no frontend code still expects removed endpoints

### **Deliverables**

* cleaned backend codebase  
* list of removed artifacts  
* reduced backend surface area

### **Exit criteria**

The backend only contains logic aligned with the current architecture and no longer carries dead compatibility layers.

---

## **Stage 3 — Database normalization and cleanup**

### **Objective**

Make the database schema reflect the actual business domain instead of version-based naming or legacy leftovers.

### **Scope**

* table removal  
* table renaming  
* constraint/index naming  
* schema consistency  
* data access adaptation

### **Tasks**

* identify unused legacy tables  
* remove obsolete tables that are no longer required  
* rename version-based tables such as `v3_*`  
* rename related constraints, indexes and foreign keys where necessary  
* review column names for ambiguity  
* validate referential integrity after rename/removal  
* adapt repositories, queries and ORM bindings  
* create safe migration scripts  
* update schema documentation

### **Deliverables**

* normalized schema  
* migration scripts  
* updated repositories/queries  
* revised schema documentation

### **Exit criteria**

The database expresses stable domain concepts and no longer uses version labels as table names.

---

## **Stage 4 — Backend structural reorganization**

### **Objective**

Reorganize the backend directory layout so it clearly reflects actual responsibilities and architectural boundaries.

### **Scope**

* folders  
* modules  
* imports  
* feature/domain grouping  
* separation of concerns

### **Tasks**

* define the target backend structure  
* group modules by domain or feature where appropriate  
* clearly separate API, application, domain and infrastructure layers if applicable  
* move jobs, inventories, positions, evidences, reports and pipeline-related code into coherent modules  
* remove unnecessary intermediate folders  
* reduce circular imports and accidental coupling  
* update imports and internal references after moves  
* document the resulting structure

### **Deliverables**

* reorganized backend tree  
* documented directory conventions  
* stabilized imports after refactor

### **Exit criteria**

The backend becomes easier to navigate, reason about and extend.

---

## **Stage 5 — Backend consolidation and optimization**

### **Objective**

Improve internal backend quality by reducing duplication, simplifying responses and optimizing query/serialization behavior.

### **Scope**

* duplicated backend logic  
* response DTOs  
* payload design  
* query efficiency  
* endpoint latency

### **Tasks**

* identify duplicated logic in services, handlers, repositories and mappers  
* consolidate repeated validation and response-construction patterns  
* distinguish summary vs detail responses where needed  
* remove duplicated or unused fields in responses  
* inspect actual frontend field usage before removing data  
* review endpoints for N+1 issues or overly expensive joins  
* reduce serialization overhead where possible  
* split endpoints if they currently mix summary and detail responsibilities  
* benchmark key endpoints before and after optimization where practical

### **Deliverables**

* lighter and clearer response contracts  
* reduced duplication in backend logic  
* improved backend consistency  
* payload comparison document  
* endpoint optimization notes

### **Exit criteria**

The backend sends cleaner and more efficient responses and contains less repeated logic.

---

## **Stage 6 — Job cancellation and timeout control**

### **Objective**

Add the ability to stop long-running or no-longer-needed jobs in a controlled and auditable way.

### **Scope**

* job state model  
* cancellation endpoint  
* worker/pipeline checks  
* persistence of cancellation reason and timing  
* timeout policy

### **Tasks**

* define new job states such as:  
  * `cancel_requested`  
  * `canceled`  
  * `timed_out`  
* add an endpoint to request job cancellation  
* persist cancellation reason and relevant timestamps  
* introduce cooperative cancellation checks in long-running pipeline stages  
* stop jobs only at safe interruption points  
* define timeout policy for jobs that exceed expected duration  
* determine how partial artifacts/results should be handled  
* ensure the backend exposes enough state for the frontend to reflect cancellation accurately  
* add tests covering manual cancellation and timeout flows

### **Recommended approach**

Implement **cooperative cancellation first**:

* the job is marked as cancellation requested  
* the worker detects this at safe checkpoints  
* the pipeline stops gracefully  
* final job status and reason are persisted

Forced termination should only be considered later if truly necessary.

### **Deliverables**

* cancellation endpoint  
* updated job state lifecycle  
* worker/pipeline cancellation support  
* timeout handling policy  
* test coverage for cancellation flow

### **Exit criteria**

The system can safely stop jobs that should no longer continue.

## **9\. Risks**

### **Risk 1 — Removing code that is still indirectly used**

**Mitigation:** perform reference checks, run audit first, and validate consumers before deletion.

### **Risk 2 — Breaking data access during table renaming**

**Mitigation:** use explicit migrations, coordinated repository updates and staged validation.

### **Risk 3 — Mixing cleanup, renaming and reorganization in the same change set**

**Mitigation:** keep stages separated and avoid overly broad refactor PRs.

### **Risk 4 — Optimizing responses without checking frontend usage**

**Mitigation:** map real field consumption before removing or reshaping payloads.

### **Risk 5 — Interrupting jobs at unsafe points**

**Mitigation:** implement cooperative cancellation with known safe checkpoints.

### **Risk 6 — Reorganizing frontend too early**

**Mitigation:** keep frontend reorganization strictly after backend contract stabilization.

## **10\. Acceptance criteria**

The v3.1.2 implementation will be considered complete when all of the following are true:

* legacy backend code not aligned with the current architecture has been removed  
* obsolete database tables have been removed  
* version-based database naming has been replaced by domain-based naming  
* backend directories have been reorganized according to the new structure  
* major backend duplications have been reduced or eliminated  
* API responses have been reviewed and improved where appropriate  
* long-running jobs can be canceled safely and their state is persisted correctly  
* frontend directories have been reorganized after backend stabilization  
* duplicated frontend logic has been reduced  
* tests and validations confirm that the system still works end to end  
* documentation reflects the new structure and decisions

---

## **11\. Rollout strategy**

### **Recommended rollout model**

This version should be executed as a sequence of **independent but coordinated epics**, ideally one major implementation block per stage or per tightly related pair of stages.

### **Suggested implementation style**

* keep database renames and removals isolated from frontend work  
* avoid mixing structural refactors with functional behavior changes in the same PR when possible  
* complete and stabilize backend contracts before frontend cleanup begins  
* add cancellation support after backend restructuring and response cleanup, not before  
* reserve final validation only after all major structural changes are done

### **Suggested PR grouping**

* PR 1: audit artifacts and cleanup plan  
* PR 2: backend legacy cleanup  
* PR 3: database normalization  
* PR 4: backend structural reorganization  
* PR 5: backend optimization  
* PR 6: job cancellation  
* PR 7: frontend reorganization  
* PR 8: frontend optimization  
* PR 9: validation and closure

---
