# **Plan de implementación técnico — v3.2.4**

## **Referencias visuales opcionales por inventario**

---

## **1\. Objetivo de implementación**

Implementar una feature que permita:

* cargar imágenes de referencia de forma opcional al crear un inventario desde el frontend,  
* persistir esas referencias asociadas al inventario,  
* incorporarlas al `AnalysisContext` común del sistema,  
* permitir que las strategies/provider compatibles las consuman,  
* mantener compatibilidad con providers que no soporten contexto visual adicional.

---

## **2\. Resultado esperado al finalizar**

Al terminar esta versión, el sistema debería permitir este flujo:

1. el usuario crea un inventario;  
2. en un segundo paso opcional sube referencias visuales;  
3. el backend las almacena y asocia al inventario;  
4. al analizar un pasillo de ese inventario, la aplicación resuelve esas referencias;  
5. las referencias se agregan al contexto común de análisis;  
6. la strategy elegida decide cómo consumirlas;  
7. el job deja trazabilidad de referencias resueltas y consumidas.

---

## **3\. Enfoque arquitectónico**

La implementación debe respetar esta separación:

### **Frontend**

Responsable de capturar la intención del usuario y subir referencias.

### **API / Application**

Responsable de persistir referencias y construir el contexto común.

### **Domain / Contracts**

Responsable de modelar `InventoryVisualReference` y extender el contrato de análisis.

### **Provider strategies**

Responsables de mapear `visualReferences` al formato del proveedor.

### **Job metadata**

Responsable de auditar qué referencias se resolvieron y si fueron consumidas.

### **Recomendación SQL conceptual**

```sql
CREATE TABLE inventory_visual_references (
    id TEXT PRIMARY KEY,
    inventory_id TEXT NOT NULL,
    filename TEXT NOT NULL,
    storage_path TEXT NOT NULL,
    mime_type TEXT NOT NULL,
    file_size INTEGER NOT NULL,
    created_at TEXT NOT NULL,
    FOREIGN KEY (inventory_id) REFERENCES inventories(id)
);

CREATE INDEX idx_inventory_visual_references_inventory_id
ON inventory_visual_references(inventory_id);
```

### **Decisión importante**

No hace falta soft delete en esta versión si querés mantener scope corto.

# **Fase 3 — Contratos de análisis multi-provider**

## **4.7 Extender contrato común**

Acá está el cambio arquitectónico más importante.

El contrato común de análisis debe incorporar referencias visuales.  
No debe quedar encapsulado en Gemini.

### **Tipo conceptual**

```ts
type AnalysisImage = {
  source: string
  mimeType: string
  role: 'primary_evidence' | 'inventory_reference'
}
```

### **Nuevo bloque**

```ts
type VisualReferenceContext = {
  referenceId: string
  source: string
  mimeType: string
  role: 'inventory_reference'
}
```

### **Extensión de AnalysisContext**

```ts
type AnalysisContext = {
  primaryEvidence: AnalysisImage[]
  visualReferences: VisualReferenceContext[]
  instructions: string[]
  metadata?: Record<string, unknown>
}
```

### **Reglas**

* `visualReferences` puede ser lista vacía,  
* el caso de uso siempre puede construir el contexto,  
* el provider decide si lo consume.

---

## **4.8 Capabilities del provider**

Si todavía no existe algo similar, conviene formalizarlo.

```ts
type ProviderCapabilities = {
  supportsImageInputs: boolean
  supportsVisualReferenceContext: boolean
  supportsStructuredOutput: boolean
}
```

Cada strategy debe exponer estas capacidades.

## **4.10 Nuevo componente: AnalysisContextBuilder**

Responsabilidades:

* recibir evidencia principal,  
* recibir inventory id,  
* resolver referencias del inventario,  
* construir `AnalysisContext`,  
* agregar instrucciones comunes sobre referencias.

### **Interfaz conceptual**

```ts

  buildForAisleAnalysis(params: {

```

---

## **4.11 Instrucción común provider-agnostic**

El builder debería agregar una instrucción común del negocio, por ejemplo:

Se incluyen imágenes de referencia del inventario que representan ejemplos válidos del estándar visual esperado. Utilizalas como contexto comparativo para reconocer mejor etiquetas, pallets o formatos relevantes. No reemplazan la evidencia principal del pasillo analizado.

Eso después cada provider lo adapta a su formato interno.

# **Fase 5 — Integración con use case de análisis**

## **4.12 Cambios en el caso de uso que ejecuta análisis de pasillo**

Hoy seguramente el flujo hace algo parecido a:

1. resolver imágenes principales;  
2. construir prompt;  
3. elegir provider;  
4. analizar.

Debe pasar a hacer:

1. resolver imágenes principales;  
2. resolver inventario del pasillo;  
3. construir `AnalysisContext` con `AnalysisContextBuilder`;  
4. elegir strategy;  
5. pasar `AnalysisContext` al provider;  
6. registrar metadata de referencias.

---

## **4.13 Punto exacto de integración**

La construcción de contexto debe ocurrir **antes** de invocar a la strategy.

Nunca dentro del controller.  
Nunca hardcodeado dentro de Gemini.  
Nunca resuelto duplicado en cada provider.

---

# **Fase 6 — Provider strategies**

## **4.14 Cambio de contrato de entrada a strategy**

Cada provider debería recibir algo equivalente a:

```ts
type ProviderAnalysisRequest = {
  analysisContext: AnalysisContext
  promptPayload: PromptPayload
}
```

---

## **4.15 Qué hace cada provider**

### **Providers con soporte visual**

* consumen `primaryEvidence`,  
* consumen `visualReferences`,  
* agregan instrucciones correspondientes,  
* reportan `visualReferencesConsumed = true`.

### **Providers sin soporte visual**

* consumen solo `primaryEvidence`,  
* ignoran `visualReferences`,  
* reportan `visualReferencesConsumed = false`.

---

## **4.16 Ejemplo de mapping por strategy**

### **Gemini / provider multimodal**

* imágenes principales → inputs visuales principales,  
* referencias → imágenes adicionales con rol contextual,  
* instrucciones → bloque system/user según diseño actual.

### **Provider textual**

* referencias no se envían como imágenes,  
* opcionalmente se agrega una nota:  
  * “Hay referencias visuales disponibles pero este provider no las consume”  
* o simplemente no se usan.

---

## **4.17 Resultado del provider**

Conviene extender el resultado para trazabilidad.

```ts
type ProviderAnalysisResult = {
  rawOutput: unknown
  parsedOutput: unknown
  providerMetadata?: {
    visualReferencesAvailable?: boolean
    visualReferencesConsumed?: boolean
    visualReferenceCount?: number
  }
}
```

# **8\. Criterios de Done técnicos**

La feature está técnicamente terminada cuando:

* existe persistencia real de referencias por inventario,  
* el frontend permite cargarlas opcionalmente en create inventory,  
* el sistema construye un `AnalysisContext` común con `visualReferences`,  
* al menos una strategy compatible las consume,  
* las strategies no compatibles no rompen el flujo,  
* el job registra metadata de resolución/consumo,  
* todo el flujo sigue funcionando cuando no hay referencias.

# **9\. Recomendaciones finales de implementación**

## **Recomendación 1**

No diseñarlo como “feature Gemini”.  
Diseñarlo como “feature del contexto de análisis”.

## **Recomendación 2**

No mezclar upload de referencias con create inventory en una sola request.  
Primero crear inventario, después subir referencias.

## **Recomendación 3**

No meter lógica de referencias en controllers/routes.  
Usar `InventoryReferenceService` \+ `AnalysisContextBuilder`.

## **Recomendación 4**

No asumir que todos los providers pueden consumir imágenes.  
Formalizar capabilities.

## **Recomendación 5**

Persistir tanto “referencias resueltas” como “referencias consumidas”.  
Eso te va a ahorrar muchísimo debugging.
