# **Etapa 2.3.A — Refactor estructural del core**

## **Objetivo general**

La etapa **2.3.A** tiene como objetivo transformar la base actual de v2.2 desde una arquitectura de tipo:

* pipeline centralizado,  
* lógica distribuida de forma irregular,  
* responsabilidades mezcladas,

hacia una arquitectura donde el sistema quede dividido en **capas y componentes con responsabilidades claras**, siguiendo sobre todo:

* **SRP**  
* **DIP**  
* **OCP**

En términos concretos, esta etapa busca:

* reducir el tamaño y responsabilidad del `HybridInventoryPipeline`  
* sacar lógica de aplicación de los endpoints  
* dejar explícito el flujo de ejecución por etapas  
* introducir contratos internos más claros entre componentes  
* preparar el terreno para testing, reemplazo de implementaciones y escalabilidad posterior

---

# **Alcance funcional de 2.3.A**

En esta etapa **no se cambia el objetivo funcional del sistema**.  
El sistema debe seguir pudiendo:

* crear un job  
* recibir video o fotos  
* preparar frames  
* ejecutar el análisis global  
* parsear entidades  
* aplicar reglas de negocio  
* generar evidencia  
* generar reporte  
* marcar el job como completado

Lo que cambia es **cómo está organizado internamente el código**.

---

# **Problema actual que resuelve 2.3.A**

Hoy el principal problema es que el flujo está funcionalmente correcto, pero estructuralmente muy mezclado.

Los síntomas más importantes son:

* `HybridInventoryPipeline` concentra demasiadas decisiones  
* `api/routes/jobs.py` hace trabajo de aplicación y de infraestructura  
* `job_store.py` mezcla persistencia, resolución de estrategia y fallback  
* el sistema no tiene una capa clara de **application services**  
* los pasos del pipeline existen conceptualmente, pero no están modelados como unidades explícitas del sistema

Eso vuelve difícil:

* extender el flujo  
* testear en aislamiento  
* reemplazar implementaciones  
* reusar lógica  
* agregar nuevas variantes del pipeline

---

# **Resultado esperado al terminar 2.3.A**

Cuando terminemos esta etapa, el sistema debería quedar así, a nivel conceptual:

## **Antes**

* endpoint crea carpetas  
* endpoint valida input  
* endpoint decide tipo de entrada  
* worker llama un pipeline “grande”  
* pipeline hace casi todo

## **Después**

* endpoint delega a un servicio de aplicación  
* servicio de aplicación crea el job y persiste input  
* worker invoca un caso de uso de ejecución  
* el caso de uso usa un pipeline orquestador liviano  
* el pipeline llama etapas explícitas, cada una con responsabilidad concreta

---

# **Objetivos técnicos específicos de 2.3.A**

## **1\. Convertir `HybridInventoryPipeline` en un orquestador liviano**

### **Estado actual**

Hoy el pipeline hace demasiadas cosas dentro de una sola unidad lógica.

### **Objetivo**

Transformarlo en un componente que solo haga:

* recibir contexto de ejecución  
* coordinar etapas  
* pasar outputs de una etapa a la siguiente  
* consolidar el resultado final

### **No debería hacer directamente**

* lógica detallada de selección de input  
* normalización concreta  
* carga manual de imágenes  
* parseo detallado de provider  
* generación concreta de evidencia  
* construcción detallada del reporte

---

## **2\. Introducir etapas explícitas del pipeline**

La idea es que el pipeline se exprese como una secuencia formal de componentes.

### **Propuesta de etapas para 2.3.A**

#### **A. `InputPreparationStage`**

Responsable de:

* validar qué tipo de input tiene el job  
* resolver si el origen es video o fotos  
* preparar directorios y referencias necesarias  
* devolver un objeto tipado con metadata de input

#### **B. `FrameAcquisitionStage`**

Responsable de:

* construir el `FrameSource`  
* obtener bundle de frames  
* devolver metadata de adquisición  
* no tomar decisiones de negocio

#### **C. `AnalysisStage`**

Responsable de:

* invocar al provider  
* recibir la respuesta cruda  
* devolver resultado raw del análisis

En 2.3.A esta etapa todavía puede seguir usando tu provider actual. La mejora fina de abstracción provider vendrá más en 2.3.B.

#### **D. `EntityResolutionStage`**

Responsable de:

* validar schema  
* parsear entidades  
* aplicar orden  
* asignar `pallet_id`  
* calcular score y `count_status`

#### **E. `EvidenceStage`**

Responsable de:

* construir evidence pack  
* mapear artifacts visuales y metadata asociada

#### **F. `ReportingStage`**

Responsable de:

* ensamblar el `hybrid_report`  
* persistir artifacts JSON finales del run

---

## **3\. Introducir un `RunContext`**

Esto es muy importante.

Hoy seguramente el estado del run está distribuido entre argumentos, paths, config, job data y variables locales.  
Eso dificulta muchísimo mantener el sistema.

### **Propuesta**

Crear un objeto central, por ejemplo `RunContext`, que viva durante toda la ejecución.

## **Qué debería contener**

* `job_id`  
* `run_id`  
* `input_type`  
* `workspace_path`  
* `artifacts_path`  
* `effective_config`  
* `provider_name`  
* `schema_version`  
* `created_at`  
* `metadata`  
* `counters`  
* `stage_outputs` opcionalmente  
* `warnings`  
* `timings`

## **Beneficios**

* evita pasar demasiados parámetros  
* reduce acoplamiento entre etapas  
* permite logging más limpio  
* facilita observabilidad  
* hace más claro qué datos pertenecen al run

# **Diseño de contratos internos**

## **Contrato base de etapa**

Cada etapa debería cumplir una interfaz simple y homogénea.

Ejemplo conceptual:

```py
class PipelineStage(Protocol):
    def run(self, context: RunContext, input_data: Any) -> Any:
        ...
```

Mejor todavía, si querés más tipado:

```py
TIn = TypeVar("TIn")
TOut = TypeVar("TOut")

class PipelineStage(Protocol, Generic[TIn, TOut]):
    def run(self, context: RunContext, data: TIn) -> TOut:
        ...
```

Esto después te ayuda muchísimo para tests y para reemplazar implementaciones.

---

# **Refactor de responsabilidades por componente**

## **1\. Endpoint de jobs**

### **Estado actual**

El endpoint probablemente:

* recibe archivos  
* persiste contenido  
* decide si es video o fotos  
* crea estructura de job  
* encola  
* define metadata

### **Problema**

Tiene demasiada lógica de aplicación y de infraestructura.

### **Objetivo en 2.3.A**

El endpoint debe quedar casi como adaptador HTTP:

* recibir request  
* convertir request a comando  
* llamar servicio  
* devolver respuesta

### **Ejemplo conceptual**

```py
@router.post("/jobs")
async def create_job(...):
    command = CreateJobCommand(...)
    result = await create_job_service.execute(command)
    return CreateJobResponse.from_result(result)
```

---

## **2\. `CreateJobService`**

Este servicio debería absorber lógica que hoy está en el endpoint.

## **Responsabilidades**

* validar comando de creación  
* resolver tipo de input  
* persistir input recibido  
* crear registro de job  
* inicializar workspace  
* encolar job

## **No debería hacer**

* ejecución del pipeline  
* lógica de dominio de entidades  
* reporting

---

## **3\. `ExecuteJobService`**

Este servicio debería encapsular el caso de uso completo de ejecución.

## **Responsabilidades**

* cargar job  
* actualizar estado a `RUNNING`  
* construir `RunContext`  
* invocar pipeline  
* persistir resultados finales  
* actualizar estado final del job  
* registrar errores del run

## **Beneficio**

Sacás del worker la lógica de aplicación real.  
El worker pasa a ser un ejecutor técnico, no el dueño del caso de uso.

---

## **4\. Worker**

### **Estado deseado**

El worker debería hacer algo parecido a:

```py
while True:
    job_id = queue.pop()
    execute_job_service.execute(job_id)
```

Nada más.

---

# **Aplicación de SOLID en 2.3.A**

## **S — Single Responsibility Principle**

### **Aplicación concreta**

Cada componente nuevo debe tener una única razón de cambio.

#### **Ejemplos**

* `CreateJobService`: cambia si cambia el caso de uso de creación  
* `FrameAcquisitionStage`: cambia si cambia cómo se obtienen frames  
* `ReportingStage`: cambia si cambia el formato del reporte  
* `EvidenceStage`: cambia si cambia la lógica de artifacts/evidencia

Hoy varias de estas razones de cambio están mezcladas.

---

## **O — Open/Closed Principle**

En 2.3.A todavía no vamos a cerrar todas las extensiones, pero sí vamos a dejar la estructura lista.

### **Ejemplo**

Si mañana agregás:

* nuevo tipo de input  
* nueva estrategia de frame source  
* nueva variante de reporting

deberías poder agregar un nuevo componente sin modificar todo el pipeline central.

---

## **D — Dependency Inversion Principle**

En esta fase lo importante es empezar a depender más de **interfaces lógicas** que de implementaciones concretas.

### **Ejemplo**

`ExecuteJobService` no debería conocer internamente:

* filesystem concreto  
* Gemini concreto  
* SQL Server concreto

Debería depender de abstracciones o servicios intermedios.

No hace falta resolverlo todo en 2.3.A, pero sí dejar la forma del sistema preparada para eso.

# **Secuencia técnica recomendada de implementación**

## **Paso 1 — Introducir `RunContext`**

Primero definí el objeto central de ejecución.

Eso te va a ordenar todo el resto.

## **Paso 2 — Extraer etapas puras desde `HybridInventoryPipeline`**

No reescribas todo de cero.  
Primero extraé lógica actual a nuevas clases manteniendo comportamiento.

Orden sugerido:

* `FrameAcquisitionStage`  
* `EntityResolutionStage`  
* `ReportingStage`

Después:

* `InputPreparationStage`  
* `AnalysisStage`  
* `EvidenceStage`

## **Paso 3 — Simplificar `HybridInventoryPipeline`**

Una vez extraídas etapas, el pipeline debe quedar como coordinador.

## **Paso 4 — Crear `ExecuteJobService`**

Mover del worker y parte del pipeline la lógica de caso de uso.

## **Paso 5 — Crear `CreateJobService`**

Mover del endpoint toda la lógica de creación.

## **Paso 6 — Dejar endpoints y worker como adaptadores finos**

Éste es uno de los criterios más fuertes de éxito de 2.3.A.

---

# **Riesgos técnicos de la etapa**

## **1\. Refactor demasiado grande de una vez**

Conviene hacerlo incremental.  
Si intentás cambiar pipeline, services, jobs y store todo junto, vas a abrir demasiados frentes.

## **2\. Cambiar estructura sin tests**

Antes de mover lógica sensible, necesitás al menos tests de humo sobre:

* creación de job  
* ejecución completa  
* generación de reporte  
* flujo con fotos  
* flujo con video

## **3\. Crear demasiadas abstracciones vacías**

No conviene sobrediseñar.  
La regla debería ser:

* abstraer donde ya hay más de una responsabilidad  
* no inventar capas si todavía no tienen uso claro

---

# **Criterios de aceptación técnicos de 2.3.A**

La etapa debería considerarse terminada si se cumplen estos puntos:

## **Arquitectura**

* `HybridInventoryPipeline` ya no contiene la mayor parte de la lógica concreta del flujo  
* existen etapas explícitas del pipeline  
* existe `RunContext`

## **API**

* `api/routes/jobs.py` delega creación de jobs a un servicio  
* el endpoint ya no concentra lógica de persistencia y preparación

## **Ejecución**

* `jobs/worker.py` delega ejecución a `ExecuteJobService`  
* el worker queda reducido a rol de adaptador técnico

## **Código**

* menor acoplamiento entre módulos  
* menos parámetros dispersos  
* menos `dict` internos implícitos entre pasos del pipeline

## **Comportamiento**

* no cambia el resultado funcional esperado del sistema  
* los reportes siguen generándose correctamente  
* el flujo con video y fotos sigue funcionando

---

# **Qué no resolvería todavía esta etapa**

Para mantener el scope sano, 2.3.A no debería intentar resolver por completo:

* provider abstraction avanzada  
* migración completa a cola distribuida  
* unificación definitiva DB vs FS  
* métricas avanzadas  
* caching  
* pipeline distribuido  
* versionado de datasets

Eso corresponde más a 2.3.B y 2.3.C.

---

# **Mi recomendación de enfoque**

Yo encararía 2.3.A con esta filosofía:

**“Extraer primero, mejorar después.”**

O sea:

1. mover lógica actual a componentes correctos  
2. mantener comportamiento estable  
3. recién después refinar contratos y abstracciones

Porque si intentás rediseñar e innovar al mismo tiempo, el riesgo de regresión sube mucho.

## **Nombre**

**v2.3.A — Core Structural Refactor**

## **Objetivo**

Reorganizar la arquitectura interna del sistema para separar responsabilidades, introducir etapas explícitas del pipeline y desacoplar API/worker del core de negocio, alineando la base con principios SOLID.

## **Entregables**

* `RunContext`  
* pipeline dividido en etapas  
* `HybridInventoryPipeline` como orquestador  
* `CreateJobService`  
* `ExecuteJobService`  
* endpoint de jobs simplificado

## **Beneficio principal**

# **Etapa 2.3.B — Abstracciones, contratos y desacople del core**

## **Objetivo general**

La etapa **2.3.B** tiene como objetivo consolidar la arquitectura resultante de 2.3.A mediante la definición explícita de:

* contratos entre componentes  
* interfaces de servicios clave  
* modelos tipados intermedios  
* límites entre dominio, aplicación e infraestructura  
* reglas claras de sustitución y extensión

En términos concretos, esta etapa busca que el sistema deje de depender de:

* implementaciones concretas ocultas  
* `dict` ambiguos entre módulos  
* comportamientos implícitos  
* acoplamientos “de hecho”

y pase a depender de:

* **interfaces pequeñas y claras**  
* **DTOs/Result objects explícitos**  
* **contratos estables entre etapas**  
* **inyección de dependencias**  
* **componentes reemplazables sin modificar el core**

---

# **Qué problema resuelve 2.3.B**

Aunque en 2.3.A ya hayamos separado componentes, todavía puede quedar este problema:

* el pipeline ya está dividido, pero las etapas siguen pasándose datos ambiguos  
* los services existen, pero dependen de clases concretas  
* hay interfaces declaradas, pero no contratos completos  
* el sistema “parece” desacoplado, pero no lo está realmente

Eso genera una falsa sensación de arquitectura limpia.

La 2.3.B resuelve exactamente eso:  
**hace explícitas las dependencias y obliga a que el sistema se comporte como una arquitectura extensible de verdad**.

---

# **Relación con SOLID**

Acá entran muy fuerte:

## **O — Open/Closed Principle**

Los componentes deben poder extenderse agregando nuevas implementaciones, no modificando el core.

## **L — Liskov Substitution Principle**

Toda implementación concreta de una interfaz debe poder reemplazar a otra sin romper el comportamiento esperado.

## **I — Interface Segregation Principle**

Nada de interfaces “gordas” que obliguen a una clase a implementar métodos que no necesita.

## **D — Dependency Inversion Principle**

Las capas altas no deben depender de Gemini, SQL Server, filesystem o queue concreta, sino de contratos.

---

# **Resultado esperado al terminar 2.3.B**

Al finalizar esta etapa, el sistema debería cumplir estas propiedades:

* el pipeline depende de contratos, no de implementaciones concretas  
* los providers LLM son realmente intercambiables  
* las fuentes de frames son realmente sustituibles  
* la persistencia relevante se consume vía interfaces  
* los services de aplicación usan puertos explícitos  
* los resultados entre etapas están tipados y validados  
* el sistema puede crecer sin meter `if provider == ...` o `if input_type == ...` por todos lados

---

# **Enfoque arquitectónico de 2.3.B**

La forma más útil de encararlo es pensar el sistema en tres capas claras:

## **1\. Dominio**

Define:

* entidades  
* value objects  
* reglas de negocio puras  
* servicios de dominio

No debería conocer:

* FastAPI  
* SQL Server  
* Gemini  
* OpenAI  
* filesystem  
* threads  
* queue

## **2\. Aplicación**

Orquesta casos de uso:

* crear job  
* ejecutar job  
* persistir resultados  
* resolver revisión manual

Conoce contratos como:

* `JobRepository`  
* `AnalysisProvider`  
* `ArtifactStorage`  
* `JobQueue`

Pero no conoce implementaciones concretas.

## **3\. Infraestructura**

Implementa los contratos:

* `GeminiAnalysisProvider`  
* `SqlJobRepository`  
* `FileSystemArtifactStorage`  
* `InMemoryJobQueue`

# **Objetivos técnicos específicos de 2.3.B**

## **1\. Cerrar el contrato de provider de análisis**

Este es uno de los puntos más importantes.

### **Problema actual**

Hoy existe una idea de provider abstraction, pero en la práctica:

* Gemini tiene comportamiento privilegiado  
* no todos los campos del request se respetan  
* el provider devuelve respuestas que todavía arrastran detalles de implementación  
* el pipeline sigue sabiendo demasiado del proveedor

### **Objetivo**

Definir una interfaz real, estable y pequeña para el análisis.

## **Propuesta de contrato**

```py
from typing import Protocol

class AnalysisProvider(Protocol):
    def analyze(self, request: AnalysisRequest) -> AnalysisResponse:
        ...
```

## **`AnalysisRequest`**

Debe contener solo lo que el proveedor necesita de forma abstracta:

```py
@dataclass(frozen=True)
class AnalysisRequest:
    run_id: str
    prompt: str
    schema_version: str
    images: list[AnalysisImage]
    metadata: dict[str, Any]
```

## **`AnalysisImage`**

```py
@dataclass(frozen=True)
class AnalysisImage:
    image_id: str
    path: Path | None
    image_bytes: bytes | None
    mime_type: str | None
    frame_index: int | None
```

## **`AnalysisResponse`**

```py
@dataclass(frozen=True)
class AnalysisResponse:
    provider_name: str
    model_name: str
    raw_text: str
    parsed_payload: dict[str, Any] | None
    usage: ProviderUsage | None
    latency_ms: int | None
    warnings: list[str]
```

## **Beneficio**

Con esto, `AnalysisStage` deja de depender de Gemini o de cómo cada SDK devuelve datos.

---

## **2\. Separar construcción de prompt del provider**

### **Problema actual**

Muchas veces la lógica de prompt queda pegada al cliente del provider o al pipeline.

### **Objetivo**

El provider no debe decidir la semántica de negocio del análisis.  
Eso debe vivir en un componente aparte.

## **Propuesta**

```py
class PromptBuilder(Protocol):
    def build(self, context: RunContext, frames: FrameBundleResult) -> PromptPayload:
        ...
```

## **`PromptPayload`**

```py
@dataclass(frozen=True)
class PromptPayload:
    system_prompt: str | None
    user_prompt: str
    schema_version: str
```

Así:

* el pipeline pide un prompt  
* el prompt builder lo arma  
* el provider solo ejecuta análisis

Esto evita que Gemini “se coma” reglas de negocio.

---

## **3\. Cerrar la abstracción de `FrameSource`**

### **Problema**

La idea de `FrameSource` es buena, pero para ser sólida debe tener un contrato completo.

### **Propuesta**

```py
class FrameSource(Protocol):
    def collect(self, request: FrameCollectionRequest) -> FrameCollectionResult:
        ...
```

## **`FrameCollectionRequest`**

```py
@dataclass(frozen=True)
class FrameCollectionRequest:
    input_type: Literal["video", "photos"]
    video_path: Path | None
    photo_paths: list[Path]
    max_frames: int
    sampling_strategy: str
    metadata: dict[str, Any]
```

## **`FrameCollectionResult`**

```py
@dataclass(frozen=True)
class FrameCollectionResult:
    frames: list[CollectedFrame]
    total_source_items: int
    selected_count: int
    truncated: bool
    warnings: list[str]
```

## **`CollectedFrame`**

```py
@dataclass(frozen=True)
class CollectedFrame:
    frame_id: str
    frame_index: int | None
    source_ref: str
    path: Path
    width: int | None
    height: int | None
```

## **Regla importante**

Toda implementación:

* video  
* photos  
* futura fuente remota

debe poder reemplazarse sin cambiar el pipeline.

---

## **4\. Separar storage de artifacts del sistema de archivos**

Este punto es importante para DIP.

### **Problema actual**

Muchos componentes probablemente escriben directamente a `Path`, `mkdir`, `open`, etc.

### **Objetivo**

El core no debería depender de filesystem como detalle concreto.

## **Contrato sugerido**

```py
class ArtifactStorage(Protocol):
    def write_json(self, relative_path: str, payload: dict[str, Any]) -> str:
        ...
```

# **Aplicación de LSP en tus abstracciones**

Esto es importante porque muchas arquitecturas “con interfaz” igual rompen Liskov.

### **Incorrecto**

* otro provider devuelve texto pero no payload

## **Caso 2: `FrameSource`**

Toda implementación debe:

* devolver `FrameCollectionResult`  
* respetar límites de selección

# **Aplicación de ISP**

No conviene interfaces tipo:

```py

    def write_bytes(...)
    def delete(...)
    def list(...)
    def sign_url(...)
    def move(...)
    def copy(...)
```

Porque no todos los consumidores necesitan todo eso.

Mejor separar en puertos chicos:

```py

        ...
```

```py
class BinaryArtifactWriter(Protocol):

        ...
```

o, si no querés exagerar tanto, al menos mantener interfaces contenidas y coherentes.

# **Etapa 2.3.C — Ejecución, persistencia y trazabilidad operativa**

## **Objetivo general**

La etapa **2.3.C** tiene como objetivo transformar la arquitectura refactorizada de 2.3.A/2.3.B en un sistema que pueda ejecutarse de forma más confiable, observable y auditable.

En términos concretos, esta etapa busca resolver cuatro problemas grandes:

1. **la ejecución actual no es suficientemente robusta**  
2. **la persistencia todavía no está completamente consolidada**  
3. **la trazabilidad del run no está modelada como entidad de primer nivel**  
4. **el sistema todavía depende demasiado del proceso local y del filesystem como soporte implícito**

El resultado esperado es una base donde ya podamos decir:

* el job tiene ciclo de vida claro  
* el run tiene identidad propia  
* los artifacts están organizados de forma consistente  
* los errores quedan registrados correctamente  
* la ejecución puede recuperarse y auditarse  
* la API y el worker ya están pensados como componentes operativos separados

---

# **Qué problema resuelve 2.3.C**

Aunque en 2.3.A y 2.3.B hayas mejorado mucho el diseño, todavía pueden seguir existiendo estos riesgos:

* jobs en memoria que se pierden  
* worker mezclado con el proceso API  
* resultados repartidos entre FS y DB sin criterio fuerte  
* falta de “run state” claro  
* logging débil  
* poca reproducibilidad de una ejecución  
* errores difíciles de investigar  
* poca base para escalar a ejecución paralela o distribuida

La **2.3.C** resuelve esa brecha:  
**lleva la arquitectura desde “bien diseñada” a “operativamente confiable”**.

---

# **Relación con SOLID**

En esta etapa siguen aplicando SOLID, pero con foco más operativo.

## **S — Single Responsibility Principle**

Separar:

* ejecución de jobs  
* persistencia de jobs  
* persistencia de resultados  
* manejo de artifacts  
* tracking del run  
* logging / audit trail

## **D — Dependency Inversion Principle**

La aplicación no debería depender de:

* queue en memoria concreta  
* filesystem concreto como única estrategia  
* thread embebido del servidor  
* SQL Server concreto en todos lados

Debe depender de puertos.

## **O — Open/Closed Principle**

El sistema debería poder incorporar:

* nueva cola  
* nuevo artifact storage  
* nueva estrategia de ejecución  
* nueva persistencia de resultados

sin rediseñar el core.

---

# **Resultado esperado al terminar 2.3.C**

Al finalizar esta etapa, el sistema debería tener estas propiedades:

* API y worker separados a nivel de arquitectura  
* ciclo de vida de job explícito  
* entidad `Run` o equivalente con metadata, tiempos y estado  
* DB definida como fuente de verdad para estado y resultados  
* filesystem o artifact storage definido como soporte de binarios/artifacts  
* logging estructurado por job y run  
* errores persistidos con contexto útil  
* artifacts reproducibles y organizados  
* capacidad de evolucionar después a queue distribuida real sin rehacer el core

# **Objetivos técnicos específicos de 2.3.C**

## **1\. Formalizar el ciclo de vida de ejecución**

Este es uno de los puntos más importantes.

### **Problema actual**

Muchas veces el sistema modela bien el `job`, pero no modela explícitamente el concepto de **run**.

* errores  
* artifacts generados

# **3\. Separar definitivamente API y worker**

## **Problema actual**

Si el worker sigue demasiado acoplado al server process, la arquitectura queda frágil.

### **Riesgos**

* duplicación accidental de workers  
* competencia por CPU/RAM  
* caída compartida API \+ ejecución  
* difícil escalado horizontal  
* difícil observabilidad real

### **Objetivo**

Aunque quizás en 2.3.C todavía no metas Redis/Celery full, sí deberías dejar claro que:

* la API crea jobs  
* la cola entrega jobs  
* el worker ejecuta jobs  
* el worker es un proceso separado del servidor web

### **Proceso API**

Responsable de:

* recibir requests  
* crear jobs  
* persistir input  
* encolar ejecución

### **Proceso Worker**

Responsable de:

* tomar jobs pendientes  
* crear `Run`

* cerrar ejecución

# **Refactor de `ExecuteJobService` en 2.3.C**

En esta etapa este servicio pasa a ser el centro operativo real.

## **Responsabilidades**

* cargar job pendiente  
* crear `RunRecord`  
* persistir transición a `RUNNING`  
* instanciar `RunContext`  
* ejecutar pipeline  
* persistir estado por etapa  
* persistir artifacts/resultados  
* cerrar run  
* actualizar estado final del job  
* persistir errores si falla

## **Flujo conceptual**

```py
def execute(job_id: str) -> ExecuteJobResult:
    job = job_repository.get(job_id)
    run = run_repository.create_for_job(job_id)

    try:
        run_repository.mark_running(run.run_id)

        context = run_context_factory.create(job=job, run=run)

        pipeline_result = pipeline.execute(context)

        result_repository.save_run_result(run.run_id, pipeline_result)
        run_repository.mark_succeeded(run.run_id, summary=pipeline_result.summary)
        job_repository.mark_succeeded(job_id)

        return ExecuteJobResult.success(run.run_id)

    except DomainExecutionError as exc:
        error_repository.save(...)
        run_repository.mark_failed(run.run_id, ...)
        job_repository.mark_failed(job_id, ...)
        raise
```

# **Estrategia de persistencia recomendada**

## **Regla fuerte**

No volver a parsear `hybrid_report.json` como mecanismo principal de persistencia.

### **Mejor enfoque**

Las etapas del pipeline producen objetos tipados.

El JSON final queda como **artifact derivado**, no como fuente principal.

Eso es un cambio importante y correcto.

# **Criterios operativos nuevos en 2.3.C**

Esta etapa debería introducir además criterios claros de consistencia:

## **1\. Un run no puede quedar sin estado final**

Todo run debe terminar en:

* `SUCCEEDED`  
* `FAILED`  
* `PARTIAL`  
* `CANCELED`

## **2\. Un job no debe depender del proceso que lo creó**

Aunque el server reinicie, el job debe seguir existiendo.

## **3\. Todo artifact relevante debe estar referenciado**

Nada importante debería quedar “solo en carpeta” sin metadato persistido.

## **4\. Todo error relevante debe quedar consultable**

No alcanza con loguearlo.

# **Riesgos técnicos de 2.3.C**

## **1\. Intentar meter infraestructura demasiado pesada**

No hace falta que esta etapa ya tenga Kubernetes, Celery, S3 y tracing distribuido.

Necesitás dar el paso correcto, no todos los pasos juntos.

## **2\. Duplicar persistencia temporalmente**

Durante la transición puede pasar que persistas en JSON y en DB. Está bien como transición, pero el diseño final debe dejar claro cuál es la fuente de verdad.

## **3\. Sobreinstrumentar demasiado pronto**

No hace falta persistir absolutamente todo.  
Persistí lo útil para:

* reconstruir el run  
* entender fallos  
* auditar resultados

## **4\. Hacer logs estructurados pero sin convención**

Si no definís nombres de eventos y campos, termina siendo ruido.

# **Criterios de aceptación técnicos de 2.3.C**

La etapa debería considerarse terminada si se cumple esto:

## **Ejecución**

* existe concepto explícito de `Run`  
* cada ejecución crea un run persistido

* DB es la source of truth para estado, runs y resultados estructurados

## **Trazabilidad**

* cada run guarda config efectiva, provider, timestamps y summary

## **Operación**

* logging con `job_id` y `run_id`  
* el sistema puede reconstruir qué pasó en una ejecución  
* los jobs sobreviven al reinicio del proceso
